Most Docker Compose health checks fail in one of two ways: they are so shallow that they mark broken apps healthy, or they are so noisy that nobody trusts them anymore.

This page is about the middle ground that actually helps in production: probes that fail for real readiness problems instead of cosmetic ones.

> **TL;DR**
>
> - Pick one readiness signal that predicts user pain, not one that only proves the process exists.
> - Keep the probe narrow and dependency-light so it fails for the right reason.
> - Tune `start_period`, `interval`, and `retries` to your real boot behavior, not copied defaults.
> - After rollout, confirm the probe flips healthy and also fails the way you expect when a dependency is broken.

## Good Probe vs Bad Probe

| Service type | Good health check | Bad health check |
| --- | --- | --- |
| Next.js or API app | lightweight local HTTP readiness endpoint | homepage that depends on every downstream service |
| PostgreSQL | `pg_isready` against the correct DB/user | "container is running" only |
| Redis | `redis-cli ping` | no check at all |
| Worker | narrow command that proves the process and critical dependency are ready | full queue execution as the health probe |

## When to use health checks

Use them when:

- a container can start before it is actually ready for traffic
- one service depends on another and startup order matters
- you want bad rollouts to fail faster and more visibly
- you care about operator trust in the green/red signal

Do not add health checks just to satisfy a template. If the probe does not change an operational decision, it is probably not helping.

## Prerequisites

- a service that exposes a meaningful readiness signal
- a small command inside the image that can test that signal
- enough understanding of boot time to set sane timing values

One easy miss: the image must actually contain the command your probe calls. If you use `curl` or `wget` in the health check, confirm the image includes it.

## Step 1: Decide what "healthy" really means for this service

The right question is not "Is the process alive?"

The right question is "Would I send real traffic to this service right now?"

For a web app, that usually means the server is listening and a lightweight endpoint returns success.

For a database, it means the server accepts connections.

For a worker, it means the process booted and can reach the one dependency it absolutely needs.

## Step 2: Start with a narrow probe

For an HTTP app:

```yaml
services:
  app:
    build: .
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://127.0.0.1:3000/api/health || exit 1"]
      interval: 15s
      timeout: 3s
      retries: 5
      start_period: 20s
```

For PostgreSQL:

```yaml
services:
  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d appdb"]
      interval: 10s
      timeout: 3s
      retries: 5
      start_period: 15s
```

These are boring on purpose. That is what makes them maintainable.

## Step 3: Use `depends_on` carefully

If startup order matters, Compose can wait for a service to become healthy before starting the dependent app:

```yaml
services:
  app:
    depends_on:
      db:
        condition: service_healthy
```

That is useful for reducing noisy startup races. It is not a substitute for proper retry logic inside the app when dependencies become temporarily unavailable later.

## Step 4: Tune timing values to match reality

Three timing mistakes show up constantly:

- `start_period` is too short for the actual boot time
- `interval` is so aggressive that slow boots look broken
- `retries` are too low for services with migrations or warm-up time

Your values should match the app you actually run, not whatever a copied blog post used.

## Step 5: Verify the probe in production-like conditions

Do not stop after writing the YAML. After deployment, check:

```bash
docker compose ps
docker inspect --format '{{json .State.Health}}' your_container_name
```

Then read the logs long enough to confirm the service stays healthy after the first real traffic.

## What health checks are not for

Health checks are not full integration tests.

They should not:

- render your heaviest page
- depend on every downstream system if only one dependency matters for readiness
- hide application retry problems behind endless waiting

Their job is smaller: catch obviously bad rollouts and expose broken startup assumptions early.

## Common mistakes

### Probing the homepage instead of a lightweight readiness path

That often creates false negatives and burns time.

### Using a command that is not available in the image

The YAML looks correct while the probe fails for the wrong reason.

### Treating a health check as a replacement for logging and monitoring

It is one signal, not the whole operating model.

### Copying timing values from another stack

Slow boot plus short retry windows creates fake instability.

## Verification checklist

Before you trust the new probe:

- deploy the stack once with the health check enabled
- confirm the container reaches `healthy`
- confirm an intentionally broken dependency causes the right failure
- confirm logs stay readable after the probe starts running
- confirm the probe reflects user-facing readiness, not just process survival

## FAQ

### What is the best first health check for a web app?

A small local HTTP endpoint that proves the app is ready to serve traffic is usually the best first choice.

### Should every container have a health check?

No. Add them where the signal changes an operational decision.

### Do health checks replace retry logic in the app?

No. They help with rollout and readiness, but the app still needs to handle transient dependency issues sensibly.

### Why does a copied health check often fail in production?

Because the image, boot time, route behavior, or dependencies differ from the environment it was copied from.

### Should I use the homepage or `/` as a health check?

Usually no. The homepage is often heavier, more cache-sensitive, and more likely to depend on things that are not required for basic readiness. A smaller local endpoint tells you more with less noise.
