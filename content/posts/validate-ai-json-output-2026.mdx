Getting JSON back from a model is not the hard part anymore. The hard part is trusting that output when it feeds a queue, an internal tool, or the next automation step. Most workflow failures are not dramatic hallucinations. They are quieter than that: one missing key, one invalid enum, one partial object that still looks close enough to pass until something downstream breaks.

I trust AI JSON only after three checks: parse it, validate it against a strict schema, and decide whether the failure should retry or fail closed. The fallback path matters as much as the prompt.
</Callout>

## The JSON failure modes I see most often

The common failures are usually boring:

- missing required fields
- invalid enum values
- strings in the wrong format
- partial objects
- extra keys that quietly bypass logic
- schema drift after prompt changes

That is why "the model returned JSON" is not a useful success definition. The real question is whether the output still matches the contract your workflow expects.

## Parse errors are the easiest problem

If the response is not valid JSON, the failure is obvious. You can reject it immediately and log the raw output.

The harder failures are the ones that parse cleanly but are still wrong:

- `"priority": "urgent"` when the workflow only accepts `low`, `medium`, or `high`
- a `date` field that is present but malformed
- a `customer_id` key that suddenly becomes `customerId`
- an array where the downstream step expected one object

Those cases are why parsing is only the first layer.

## Validation should happen outside the model

I do not ask the model to be its own validator. The model generates. My code validates.

That means I check:

- required keys
- allowed enums
- type shape
- string formats
- array bounds when needed

Prompting reduces drift. Validation is what makes the workflow safe.

## Schema drift is a real ops problem

Schema drift usually shows up after one of these:

- you add one field to the prompt
- you relax the instructions to get better wording
- you switch models
- the workflow starts handling broader inputs than it did on day 1

The response still looks "mostly right," but one assumption is gone.

This is why I like strict schema validation and explicit versioning of the output contract. If the workflow changes, I want the validation failure to happen immediately instead of silently contaminating state.

## The validation shape I trust

My default pattern is:

1. ask the model for one narrow object
2. parse the response
3. validate against a schema
4. route success or failure explicitly

That means the workflow always lands in one of three states:

- valid and usable
- invalid but retryable
- invalid and should be rejected or escalated

I do not let "almost valid" become its own category.

## A minimal example in JavaScript

```js
const result = JSON.parse(raw)
const priorities = ['low', 'medium', 'high']
if (!result.summary || typeof result.summary !== 'string') {
  throw new Error('Missing summary')
}
if (!priorities.includes(result.priority)) {
  throw new Error('Invalid priority enum')
}
if (typeof result.needs_human !== 'boolean') {
  throw new Error('needs_human must be boolean')
}
```

That is not glamorous, but it catches the failures that matter most.

## Where validation should happen

I validate at the first point where the model output crosses into deterministic logic.

That usually means:

- right after the API response in a server action or worker
- before the output is written to a database
- before it enters an automation branch
- before another service treats it as trusted input

I do not wait until a later node or UI component discovers the problem indirectly. The earlier the contract is enforced, the cheaper the failure is.

## Retry vs reject is the real decision

Not every invalid response deserves a retry.

I retry when:

- the model output was truncated
- the provider had a temporary issue
- the JSON wrapper broke in a recoverable way
- the prompt contract is still valid and the failure looks incidental

I reject or escalate when:

- required fields are consistently missing
- the schema changed and the prompt is now wrong
- the task itself is too ambiguous for the output shape
- the workflow would be risky if bad data slipped through

That distinction matters because repeated retries can turn one design flaw into noisy token waste.

## Bad enums and invalid formats are not small errors

A bad enum often tells you something important:

- the prompt is too open-ended
- the schema is underspecified
- the task should be broken into smaller calls

The same is true for bad formats. If the model keeps returning loose timestamps, phone numbers, or IDs, that is usually a signal that the output contract is too optimistic for the task.

## My safe fallback path

I want one boring fallback path for invalid output:

- log the raw response
- log the validation error
- store the input that triggered it
- route to manual review or a simpler deterministic fallback

Examples:

- default to `needs_human: true`
- skip enrichment and keep the core workflow moving
- write the item into a review queue
- return a narrow safe response instead of pretending success

That is what turns a model failure into an operationally tolerable event instead of a broken chain reaction.

## My recommendation

If AI JSON output touches anything important, validate it like you would validate user input. Prompting helps, but the contract belongs in code. Parse the response, enforce the schema, decide clearly between retry and reject, and keep one safe fallback path for the cases that still drift.

That is the difference between a model-powered workflow that feels clever in demos and one that survives real traffic without quietly poisoning downstream logic.

## FAQ

### Where should I validate AI JSON output?

Validate it at the first deterministic boundary, usually immediately after the model response and before any database write, workflow branch, or trusted downstream action.

### Should I retry invalid AI JSON automatically?

Only when the failure looks incidental, like truncation or a transient formatting issue. Repeated missing fields or bad enums usually indicate a prompt or task-design problem, not a retry problem.

### What is the biggest structured-output risk?

The biggest risk is not malformed JSON. It is valid-looking JSON that quietly violates the workflow contract with missing fields, bad enums, or shifted keys.

### What is the safest fallback for bad model output?

The safest fallback is usually to fail closed, log the raw response and validation error, and route the item to manual review or a simpler deterministic path.
