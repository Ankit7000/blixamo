## What the Stack Actually Looks Like

Three components, one VPS, no vendor lock-in:

**WAHA (WhatsApp HTTP API)** runs in Docker on port 3010. When a message arrives, WAHA fires a webhook to n8n. When n8n sends a reply, WAHA delivers it back to WhatsApp.

**n8n** runs on port 5678 — already deployed on my Hetzner CPX22 as part of my [multi-project VPS setup](/blog/multiple-projects-single-vps). It receives the WAHA webhook, sends the message to Claude API, and fires the response back.

**Claude API** (Anthropic) is the only external dependency. You call `https://api.anthropic.com/v1/messages` with your message, get a response, done. No SDK needed — just an HTTP POST.

The full flow:
```bash
WhatsApp message → WAHA webhook → n8n workflow → Claude API → n8n → WAHA → WhatsApp reply
```
Total RAM added to my VPS: ~180MB for the WAHA container. My CPX22 runs 5 projects at ~715MB total — adding WAHA brings it to ~895MB out of 4GB.

## Step 1 of 4 — Deploy WAHA on Your Hetzner VPS
#
WAHA is an open-source Docker image that exposes WhatsApp as a REST API. It handles authentication, message receiving, and sending — you get a clean HTTP interface without any client-side code.
#
SSH into your Hetzner VPS:
#
```bash
docker run -d \
  --name waha \
  --restart unless-stopped \
  -p 3010:3000 \
  -v /var/waha-data:/app/.sessions \
  devlikeapro/waha
```
The `-v` flag persists your WhatsApp session across container restarts — skip it and you'll have to re-scan the QR code after every restart.
#
Verify it started:
#
```bash
docker ps | grep waha
docker logs waha --tail 20
```
<Callout type="tip" title="Why this matters">
The `-v /var/waha-data:/app/.sessions` flag persists your WhatsApp session. Without it, you'd need to re-scan the QR code after every Docker restart or server reboot — which would break your bot every time you update.
</Callout>
#
Now open the WAHA API dashboard at `http://your-vps-ip:3010/dashboard`. Call `POST /api/sessions/start` with this body:
#
```bash
{
  "name": "default",
  "config": {
    "webhooks": [
      {
        "url": "https://your-n8n-domain.com/webhook/whatsapp",
        "events": ["message"]
      }
    ]
  }
}
```
Then call `GET /api/sessions/default/auth/qr` to get the QR code. Scan it with WhatsApp on your phone. Once scanned, WAHA status changes to `WORKING` — you're connected.
#
With WAHA running and your WhatsApp account linked, set up the n8n workflow that connects incoming messages to Claude API.

## Step 2 of 4 — Set Up the n8n Webhook Workflow
#
The n8n workflow is 4 nodes. Open your n8n instance and create a new workflow.
#
**Node 1: Webhook trigger**
#
Add a Webhook node. Set method to POST, path to `whatsapp`, and response mode to "Respond to Webhook". This gives you the URL `https://your-n8n-domain.com/webhook/whatsapp` — the same URL you put in the WAHA config.
#
**Node 2: Extract message text**
#
Add a Set node after the webhook. Extract the message content:
#
```bash
body: {{ $json.body.payload.body }}
from: {{ $json.body.payload.from }}
```
The WAHA payload has the message text at `payload.body` and the sender's number at `payload.from`.
#
**Node 3: Claude API HTTP Request**
#
Add an HTTP Request node. Configure it:
#
```bash
Method: POST
URL: https://api.anthropic.com/v1/messages
Authentication: Header Auth
  Header: x-api-key = [your Claude API key]
  Header: anthropic-version = 2023-06-01
```
Body (JSON):
#
```bash
{
  "model": "claude-sonnet-4-5",
  "max_tokens": 1024,
  "messages": [
    {
      "role": "user",
      "content": "={{ $json.body }}"
    }
  ]
}
```
**Node 4: Send reply via WAHA**
#
Add another HTTP Request node to send Claude's response back to WhatsApp:
#
```bash
Method: POST
URL: http://localhost:3010/api/sendText
```
Body (JSON):
#
```bash
{
  "chatId": "={{ $('Extract Message').item.json.from }}",
  "text": "={{ $json.content[0].text }}"
}
```
Activate the workflow. Send yourself a WhatsApp message. Watch it reply.
#
The basic bot works but responds generically. A system prompt gives it personality and keeps responses concise for a chat interface.

## Step 3 of 4 — Add a System Prompt for Personality
#
Out of the box, Claude responds as a general-purpose assistant. For a personal WhatsApp bot, give it context about who you are and what kinds of questions it'll handle. Update the HTTP Request body in Node 3:
#
```bash
{
  "model": "claude-sonnet-4-5",
  "max_tokens": 1024,
  "system": "You are a helpful personal assistant for an indie developer. Answer concisely — 2 to 4 sentences max unless a code question. Use plain text, no markdown.",
  "messages": [
    {
      "role": "user",
      "content": "={{ $json.body }}"
    }
  ]
}
```
<Callout type="tip" title="Keep the System Prompt Short">
Claude defaults to long, detailed responses. The most important constraint for WhatsApp is "keep replies concise" — nobody wants to scroll through a 500-word answer in a chat app. Keep your system prompt under 200 words.
</Callout>

## Step 4 of 4 — Add Nginx Proxy for WAHA (Recommended)
#
Running WAHA on a raw port (`your-ip:3010`) exposes it publicly without SSL. Proxy it through Nginx:
#
```bash
server {
    listen 443 ssl;
    server_name waha.yourdomain.com;
#
    location / {
        proxy_pass http://localhost:3010;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```
Get an SSL certificate:
#
```bash
certbot --nginx -d waha.yourdomain.com
```
Update your WAHA webhook URL to use the HTTPS domain. Reload Nginx:
#
```bash
nginx -t && systemctl reload nginx
```
This is the setup I run — WAHA at `waha.blixamo.com`, proxied through Nginx with SSL. Now let's cover the errors you'll likely hit.

## Common Errors and How to Fix Them
#
**Webhook not receiving messages from WAHA**
#
Most likely cause: your WAHA webhook URL uses `localhost` instead of the public n8n URL. WAHA runs inside Docker — it can't reach `localhost` of the host machine.
#
```bash
# Check WAHA session config
curl http://localhost:3010/api/sessions/default
```
Look at the `webhooks` array — the URL must be publicly reachable. Update via `PATCH /api/sessions/default`.
#
**Claude API returning 401 Unauthorized**
#
Claude requires two headers: `x-api-key` AND `anthropic-version`. Missing either returns 401. Test from your VPS:
#
```bash
curl https://api.anthropic.com/v1/messages \
  -H "x-api-key: YOUR_KEY" \
  -H "anthropic-version: 2023-06-01" \
  -H "content-type: application/json" \
  -d '{"model":"claude-sonnet-4-5","max_tokens":100,"messages":[{"role":"user","content":"ping"}]}'
```
If this returns a response, the API key is fine — the issue is in your n8n credential configuration.
#
**WAHA QR code session disconnects after restart**
#
You started WAHA without the volume mount. Fix: stop the container, remove it, re-run with the `-v` flag. You'll need to scan the QR code once more.
#
```bash
docker stop waha && docker rm waha
#
docker run -d \
  --name waha \
  --restart unless-stopped \
  -p 3010:3000 \
  -v /var/waha-data:/app/.sessions \
  devlikeapro/waha
```
**Bot replies to its own messages (infinite loop)**
#
WAHA sends a webhook for every message including outgoing ones. Fix: add an IF node after the Webhook node:
#
```bash
Condition: {{ $json.body.payload.fromMe }} equals false
```
Only process messages where `fromMe` is false — messages from other people, not from your bot.

## FAQ
#
### Is this WhatsApp AI assistant free to run?
#
The infrastructure cost is $0 extra if you already run n8n on a Hetzner VPS. WAHA is open-source — no subscription. Claude API is the only cost: under $2/month for 50–100 messages per day at personal use volume.
#
### Does this work with a regular WhatsApp account or only WhatsApp Business?
#
Both work. WAHA connects via the same mechanism as WhatsApp Web — any account that can scan a QR code works. You can use WhatsApp normally on your phone while the bot runs in the background.
#
### Is this setup against WhatsApp's Terms of Service?
#
WAHA uses an unofficial connection. For a personal bot responding to your own messages, the practical risk is very low — WhatsApp primarily targets mass marketing bots and spam. For customer-facing services or mass messaging, use the official Meta WhatsApp Business API.
#
### How do I make Claude remember previous messages?
#
The basic setup is stateless — Claude treats each message as new. To add memory, use n8n's built-in Window Buffer Memory node or store conversation history in SQLite and include previous messages in the `messages` array sent to Claude. For most personal use cases, stateless works fine.
#
### Why use Claude API instead of OpenAI?
#
Claude Sonnet 4.5 is fast enough for real-time WhatsApp replies (under 2 seconds in my testing) and handles code questions and complex queries well. You can swap Claude for any other API that accepts HTTP POST requests by updating Node 3 — the n8n workflow structure stays identical.
#
### What happens to the bot when n8n restarts?
#
Nothing breaks permanently. The WAHA session stays connected because it's a separate Docker container. Messages sent during the n8n restart window won't be processed. For high-reliability setups, add a Redis message queue between WAHA and n8n. For personal use, the basic setup is fine.

## Extending the Assistant
#
Once the base workflow is running, three extensions that are worth adding:
#
**1. Message history (context window)**
Store the last 10 messages per sender in n8n's static data or a Redis key. Pass them as conversation history to Claude. The assistant remembers what was discussed earlier in the session.
#
**2. Command routing**
Add an IF node before Claude: if the message starts with `/image`, `/summary`, `/task` — route to a different workflow branch. This turns the assistant into a multi-function bot without complicating the main flow.
#
**3. Rate limiting**
Add a counter per phone number in Redis. If a sender sends more than 30 messages in an hour, skip the Claude API call and reply with a cooldown message. Prevents runaway API costs from a single sender.
#
All three are n8n node additions — no code required. The [n8n self-hosting guide](/blog/self-hosting-n8n-hetzner-vps) covers the Redis setup you'll need for options 1 and 3.
