## Optimization 1 — ISR + Cloudflare CDN (Biggest Win: ~1,200ms)

Incremental Static Regeneration (ISR) is Next.js's most powerful performance feature. Pages are generated once, cached at the CDN edge, and served statically — while still updating in the background at your configured interval.

Without ISR, every request hits your Node.js server and renders the page on demand. With ISR + Cloudflare, requests are served from Cloudflare's edge nodes (under 30ms) without touching your server at all.

**Setting ISR in Next.js 15 App Router:**

```bash
// app/blog/[slug]/page.tsx
export const revalidate = 3600 // Revalidate every 1 hour
// ---
export async function generateStaticParams() {
  const posts = await getAllPosts()
  return posts.map(post => ({ slug: post.slug }))
}
```
**The critical nginx config for ISR + Cloudflare:**
//
This is the config most guides get wrong. You need specific `Cache-Control` headers that tell Cloudflare to cache the page while Next.js handles background revalidation.
//
```bash
# /etc/nginx/sites-available/blixamo
server {
    server_name blixamo.com www.blixamo.com;
# ---
    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        # ISR cache headers — critical for Cloudflare
        proxy_hide_header Cache-Control;
        add_header Cache-Control "public, max-age=0, s-maxage=3600, stale-while-revalidate=86400";
    }
# ---
    location /_next/static/ {
        proxy_pass http://localhost:3000;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }
# ---
    listen 443 ssl;
    ssl_certificate /etc/letsencrypt/live/blixamo.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/blixamo.com/privkey.pem;
}
```
<Callout type="warning" title="Cloudflare proxy must be ON for CDN caching">
Set DNS to proxied (orange cloud) in Cloudflare. Without this, traffic hits your origin directly and ISR cache is useless. Also set Cloudflare Cache Level to "Standard" in the Caching rules.
</Callout>
//
**Why `s-maxage` matters:** `max-age=0` tells browsers not to cache (so users always get fresh content). `s-maxage=3600` tells Cloudflare to cache for 1 hour. `stale-while-revalidate=86400` tells Cloudflare to serve stale content while fetching a fresh version — eliminating cache-miss latency spikes.
//
After applying this config, blog pages went from ~800ms origin response to ~28ms Cloudflare edge response. That's the biggest single win.

## Optimization 2 — next/image With WebP Conversion (~300ms saved)
//
Every image on blixamo.com was originally served as JPEG. `next/image` converts them to WebP automatically — 25-35% smaller file size with the same visual quality. On a mobile 4G connection, this saves 200-400ms per image-heavy page.
//
```bash
// Before — plain img tag, no optimization
<img src="/images/posts/featured.jpg" alt="Article title" />
// ---
// After — next/image with automatic WebP + lazy loading
import Image from 'next/image'
// ---
<Image
  src="/images/posts/featured.jpg"
  alt="Article title — blixamo.com"
  width={1200}
  height={630}
  priority={isAboveFold}
  quality={82}
/>
```
**Key settings:**
//
- `priority={true}` for above-the-fold images (LCP image) — disables lazy loading for the hero image
- `priority={false}` (default) for all other images — lazy loads as user scrolls
- `quality={82}` — sweet spot between quality and file size, tested visually
//
**Configure next/image for self-hosted:**
//
```bash
// next.config.js
module.exports = {
  images: {
    formats: ['image/webp'],
    deviceSizes: [640, 750, 828, 1080, 1200],
    imageSizes: [16, 32, 48, 64, 96],
    minimumCacheTTL: 31536000,
  }
}
```
<Callout type="tip" title="Expected output">
After deploying: check network tab in Chrome DevTools. Image requests should show Content-Type: image/webp instead of image/jpeg.
</Callout>
//
**Impact on blixamo.com:** Featured image went from 440KB JPEG to 68KB WebP. Article listing page (12 thumbnails) went from 3.2MB total images to 890KB. On 4G, that's the difference between 320ms and 89ms for images.

## Optimization 3 — Font Optimization (~150ms saved)
//
Google Fonts adds a render-blocking request on every page load. Even with `display=swap`, the FOUT (flash of unstyled text) and layout shift hurts CLS score. Next.js 15 has built-in font optimization that downloads and self-hosts fonts at build time.
//
```bash
// app/layout.tsx — before (slow)
// <link href="https://fonts.googleapis.com/css2?family=Inter" rel="stylesheet" />
// ---
// After — next/font (self-hosted, zero network request)
import { Inter } from 'next/font/google'
// ---
const inter = Inter({
  subsets: ['latin'],
  display: 'swap',
  variable: '--font-inter',
  preload: true,
})
// ---
export default function RootLayout({ children }) {
  return (
    <html className={inter.variable}>
      <body>{children}</body>
    </html>
  )
}
```
`next/font` downloads the font at build time, hosts it on your server, and inlines the CSS. Zero render-blocking network request. Zero FOUT because the font CSS is inlined in `<head>`. CLS from font swap eliminated.
//
The `subsets: ['latin']` is critical — it downloads only the characters you use instead of the full Unicode font file. Inter Latin subset is 14KB. Full Inter is 180KB.
//

## Optimization 4 — Bundle Analysis and Tree Shaking (~200ms saved)
//
Next.js ships every package you import, including parts you don't use. Bundle analysis reveals which packages are bloating your JS.
//
```bash
# Install bundle analyzer
npm install @next/bundle-analyzer --save-dev
```
```bash
// next.config.js
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
})
// ---
module.exports = withBundleAnalyzer({
  // your existing config
})
```
```bash
# Generate bundle report
ANALYZE=true npm run build
# Opens browser with interactive treemap
```
<Callout type="tip" title="Expected output">
Two browser tabs open: client.html and server.html showing all JS chunks with sizes.
</Callout>
//
**What I found on blixamo.com:**
//
- `date-fns` was imported entirely — only used `format()` and `parseISO()`. Fix: `import { format, parseISO } from 'date-fns'` instead of `import * as dateFns`
- `gray-matter` (MDX frontmatter parser) was in the client bundle — should be server-only. Fix: moved all MDX parsing to server components
- `rehype-highlight` loaded all 180 languages — only needed bash, typescript, json. Fix: configure selective language import
//
```bash
// Before — full rehype-highlight (380KB)
import rehypeHighlight from 'rehype-highlight'
// ---
// After — only the languages you use (28KB)
import rehypeHighlight from 'rehype-highlight'
import bash from 'highlight.js/lib/languages/bash'
import typescript from 'highlight.js/lib/languages/typescript'
import json from 'highlight.js/lib/languages/json'
// ---
const options = {
  rehypePlugins: [[rehypeHighlight, { languages: { bash, typescript, json } }]]
}
```
Total JS reduction: 340KB → 118KB compressed. Parse time on mobile: ~400ms → ~120ms.

## Optimization 5 — Preconnect and DNS Prefetch
//
Every external resource (fonts, analytics, CDN assets) requires a DNS lookup + TCP handshake before the first byte. Preconnect eliminates this latency for critical resources.
//
```bash
// app/layout.tsx
export default function RootLayout({ children }) {
  return (
    <html>
      <head>
        {/* Preconnect to analytics (Plausible self-hosted) */}
        <link rel="preconnect" href="https://analytics.blixamo.com" />
        {/* DNS prefetch for less critical third parties */}
        <link rel="dns-prefetch" href="https://www.googletagmanager.com" />
      </head>
      <body>{children}</body>
    </html>
  )
}
```
<Callout type="tip" title="Expected output">
In Chrome DevTools Network tab: analytics.blixamo.com shows 0ms DNS time on subsequent requests — preconnect already established the connection.
</Callout>
//
Only preconnect to resources used in the first 500ms of page load. Too many preconnects compete for connections and hurt performance.

## Optimization 6 — Metadata and OG Image Optimization
//
Next.js 15's `generateMetadata` is evaluated server-side per request unless you use ISR. Combined with static OG images, metadata generation adds zero runtime overhead.
//
```bash
// app/blog/[slug]/page.tsx
export async function generateMetadata({ params }) {
  const post = await getPost(params.slug)
  return {
    title: post.title,
    description: post.description,
    openGraph: {
      title: post.title,
      description: post.description,
      images: [{
        url: `/images/posts/${params.slug}/featured.jpg`,
        width: 1200,
        height: 630,
      }],
    },
    alternates: {
      canonical: `https://blixamo.com/blog/${params.slug}`,
    }
  }
}
```
OG images are static JPEGs served from your VPS with `immutable` cache headers — zero compute on social share. Generating them dynamically with `@vercel/og` on every Twitter share adds 200-500ms latency unnecessarily for a blog.

## Optimization 7 — PM2 Cluster Mode
//
By default, Node.js is single-threaded. On a 3-vCPU Hetzner CPX22, you're using 33% of available CPU in fork mode. Cluster mode runs one worker per CPU core.
//
```bash
// ecosystem.config.js
module.exports = {
  apps: [{
    name: 'blixamo',
    script: 'node_modules/.bin/next',
    args: 'start',
    instances: 'max', // Uses all CPU cores
    exec_mode: 'cluster',
    max_memory_restart: '1G',
    env: {
      PORT: 3000,
      NODE_ENV: 'production'
    }
  }]
}
```
```bash
pm2 start ecosystem.config.js
pm2 save
```

│ id  │ name       │ mode    │ pid   │ status   │ cpu  │
├─────┼────────────┼─────────┼───────┼──────────┼──────┤
│ 0   │ blixamo    │ cluster │ 12341 │ online   │ 0%   │
│ 1   │ blixamo    │ cluster │ 12342 │ online   │ 0%   │
│ 2   │ blixamo    │ cluster │ 12343 │ online   │ 0%   │

//
<Callout type="warning" title="Cluster mode with ISR">
ISR revalidation state is not shared between cluster workers by default. On a 3-worker cluster, a revalidation request goes to one worker — the other 2 may still serve stale content until their next revalidation cycle. For a blog with 1-hour ISR, this is acceptable. For real-time data, use Redis for shared ISR state or stick to fork mode.
</Callout>
//
Impact: p95 response time dropped from 106ms to 71ms under concurrent load. For a blog with occasional traffic spikes (Reddit/HN frontpage), cluster mode prevents single-request bottlenecks.

## Troubleshooting Performance Issues
//
**Problem: Pages not being cached by Cloudflare despite correct headers**
//
```bash
# Check what headers Cloudflare is actually sending
curl -I https://yourdomain.com/blog/your-post
# Look for: CF-Cache-Status: HIT (cached) or MISS (not cached)
```
If you see `CF-Cache-Status: BYPASS`, Cloudflare is bypassing cache. Common causes: cookie present in response, `Cache-Control: private` being set somewhere upstream.
//
**Problem: ISR not revalidating — old content showing after revalidate interval**
//
```bash
# Force revalidation of a specific path
curl -X POST https://yourdomain.com/api/revalidate \
  -H "Content-Type: application/json" \
  -d '{"path": "/blog/your-slug", "secret": "YOUR_SECRET"}'
```
**Problem: next/image returning 400 on external images**
//
Add external domains to `next.config.js`:
//
```bash
module.exports = {
  images: {
    remotePatterns: [{
      protocol: 'https',
      hostname: 'your-image-host.com',
    }],
  }
}
```
**Problem: Lighthouse score high but real users report slow load**
//
Lighthouse measures lab conditions (simulated throttling). Check Cloudflare Analytics → Cache Hit Rate. If below 80%, most users are hitting your origin instead of the CDN edge. Fix: ensure `s-maxage` is set correctly in nginx Cache-Control headers.

## FAQ
//
### What is a good load time for a Next.js blog in 2026?
//
Under 1 second LCP is the target for passing Google's Core Web Vitals threshold. Under 500ms is excellent. blixamo.com averages 306ms globally with Cloudflare CDN — achievable on any self-hosted VPS with ISR configured correctly.
//
### Does ISR work on self-hosted Next.js?
//
Yes — ISR is a Next.js feature, not a Vercel-only feature. The revalidation happens on your server. You need a CDN (Cloudflare free plan works) to cache and serve the ISR output at the edge. Without a CDN, ISR still works but every request hits your origin.
//
### How do I know if Cloudflare is actually caching my pages?
//
Check the `CF-Cache-Status` response header. `HIT` = served from Cloudflare cache (fast). `MISS` = fetched from your origin. `BYPASS` = caching skipped (usually a cookie or auth header). Target 80%+ cache hit rate for a blog.
//
### Should I use Vercel instead of self-hosting for performance?
//
Vercel's edge network is excellent and their ISR implementation is well-tested. But blixamo.com achieves 306ms on a €5.19/month Hetzner VPS with Cloudflare free. Vercel Pro costs €20/month. For a blog or small SaaS, self-hosting with these optimizations matches or exceeds Vercel performance at 1/4 the cost.
//
### Does PM2 cluster mode work with Next.js App Router?
//
Yes. The App Router is stateless — each request is independent. Cluster mode distributes requests across workers correctly. The only caveat is ISR state not being shared between workers, covered in Optimization 7 above.
//
### How often should I run bundle analysis?
//
After every major dependency update and whenever you add a new library. npm packages frequently include unexpectedly large transitive dependencies. Running `ANALYZE=true npm run build` takes 2 extra minutes and catches bloat before it ships.
