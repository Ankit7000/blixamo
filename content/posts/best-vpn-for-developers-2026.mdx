## Why Most VPNs Fail Developer Workflows

Most VPN reviews test Netflix and torrenting. Developer workflows are different — and harder on VPNs.

When you SSH into a remote server through a VPN, any latency spike above 80ms causes visible lag in your terminal. Drop below 60ms and it feels like a local connection. Go above 150ms and you're fighting the connection on every keystroke. Most consumer VPNs sit around 90-120ms on their "optimized" servers — fine for streaming, frustrating for SSH.

The second problem is simultaneous region access. As a developer, you might need to access your Indian bank dashboard (India IP), test your SaaS from a US IP, and SSH into a Hetzner server in Germany — all at the same time. Standard VPNs route all traffic through one server. You can't do that.

Third: kill switch behavior. A kill switch that blocks all traffic when the VPN drops is great for privacy. But if it also blocks your `npm install` mid-way and corrupts your `node_modules`, you've lost 20 minutes. Some VPNs let you configure which apps the kill switch applies to. Others don't.

## What Makes a VPN Good for Developers

Before comparing options, here's the framework I use to evaluate any VPN for developer use. Consumer VPN reviews measure download speed. That metric is almost irrelevant for our use case.

The 4 developer-specific metrics that actually matter:

Split tunneling / App Mode — can you route specific apps through the VPN while others use your real IP? Without this, you're constantly enabling and disabling the VPN every time you need to switch context.

SSH latency and stability — SSH is brutally sensitive to latency. A VPN adds 20-60ms depending on server distance. What kills SSH is *variance* — if latency jumps from 40ms to 200ms randomly, your terminal freezes. Test: run `ping -c 100 your-server-ip` through the VPN. Standard deviation should be under 15ms.

Kill switch granularity — does the kill switch apply to all apps, or can you exclude your terminal? A global kill switch that blocks `npm install` mid-run is a productivity killer. Per-app kill switches are the right design.

DNS leak protection — if your VPN leaks DNS queries, API endpoints can still identify your real location. Run `dnsleaktest.com` after connecting. All queries should resolve through the VPN's DNS, not your ISP's.

I tested every VPN below against these 4 metrics on the same Hetzner VPS infrastructure I use for [self-hosting Next.js on a VPS](/blog/how-to-self-host-nextjs-on-vps).

## Troubleshooting: Common VPN Problems in Dev Workflows

**SSH session drops every 30-60 minutes**
This is the VPN's idle timeout, not a connection issue. Fix: set `ServerAliveInterval 60` and `ServerAliveCountMax 3` in your `~/.ssh/config`. This sends keepalive packets and prevents the session from timing out through the VPN tunnel.

**`npm install` fails or hangs through VPN**
Check if the VPN kill switch is interfering. Temporarily disable it and retry. If that fixes it, configure the kill switch to exclude your terminal app. Alternatively, whitelist `registry.npmjs.org` in your split tunnel rules.

**DNS leaks showing real IP on `dnsleaktest.com`**
Your VPN's DNS override isn't working. Go to VPN settings → DNS → enable Custom DNS. Set it to the VPN provider's DNS servers, not `8.8.8.8`. Most modern VPNs handle this automatically, but older configs can break after system updates.

**High latency spikes during `git push` to remote**
Large git pushes send sustained traffic that can trigger throttling on some VPN servers. Switch to a less-loaded server in the same region. Run `ping -c 20 server-ip` before pushing — if you see latency spikes, change servers first.

**VPN disconnects during `npm run build`**
Build processes can briefly saturate CPU and cause the VPN client to miss heartbeat packets. Fix: increase the VPN reconnect timeout in advanced settings. SafeShell and NordVPN both expose this setting.
