# CapRover to Coolify Migration Checklist (2026)

If you already know you are moving from CapRover to Coolify, this page should be your checklist, not your theory article.

The full migration walkthrough lives in [Migrate from CapRover to Coolify](/blog/migrate-from-caprover-to-coolify-2026). This page is the shorter pre-cutover and cutover checklist I would keep open while doing the move. Keep [Self-Hosting](/category/self-hosting), the [Self-Hosting Complete Guide](/guides/self-hosting-complete-guide), and the [Resources Hub](/tag/deployment) nearby if you want the longer background reads while you work.

<Callout type="info" title="TL;DR">

- Inventory every app, env var, volume, worker, cron job, and domain before touching Coolify.
- Run CapRover and Coolify in parallel until the new stack passes smoke tests under a test domain.
- Move data before DNS, switch DNS last, and keep CapRover alive until the new setup survives real traffic.

</Callout>

## Quick Table

| Stage | Done when | Biggest mistake |
|---|---|---|
| Inventory | Every app and dependency is listed | Forgetting workers, cron, or storage |
| Coolify setup | Dashboard works and server is healthy | Installing under pressure on cutover day |
| App recreation | Builds pass and env vars are correct | Mixing build-time and runtime config |
| Data move | Databases and volumes are restored and checked | Trusting a copy without opening the app |
| DNS cutover | Requests land on Coolify and SSL is valid | Switching traffic before smoke tests |
| Observation | Logs are quiet and scheduled tasks run | Tearing down CapRover too early |

## Pre-Migration Checklist

Check every item below before you create the first production app in Coolify.

- List every CapRover app, worker, scheduled job, and one-off service.
- Export or copy every environment variable for every app.
- List every persistent volume and which app depends on it.
- Write down every domain and subdomain currently routed through CapRover.
- Note any CapRover-specific behavior you rely on:
  custom nginx config, captain definitions, internal hostnames, build assumptions.
- Decide where Coolify will run:
  separate VPS is safer than same-host migration.
- Write one rollback sentence for each critical app:
  "If verification fails, traffic goes back to CapRover and this app stays there."
- Keep these support pages open:
  [Coolify vs CapRover in 2026](/blog/coolify-vs-caprover-2026),
  [Coolify Complete Guide 2026](/blog/coolify-complete-guide-2026),
  [Coolify Backup and Restore Runbook](/blog/coolify-backup-restore-runbook-2026),
  [Docker Volume Backup Strategy on VPS](/blog/docker-volume-backup-strategy-vps-2026).

If you cannot complete the list above in one sitting, you are not ready for cutover yet. That does not mean the migration is hard. It means the current state is still too fuzzy.

## Cutover Preparation Checklist

These are the items I would verify before touching DNS.

### 1. Coolify server and dashboard are stable

- Coolify dashboard loads cleanly.
- The server shows healthy status.
- Disk space, memory, and CPU are not already under pressure.
- Firewall and ports are correct for the dashboard and app traffic.

### 2. Apps build and boot in Coolify

- Every app has the correct repo or Docker image.
- Build-time variables are separated from runtime variables where needed.
- Health checks are configured for apps that support them.
- App boot logs are clean enough to trust.

### 3. Data and stateful services are restored

- Database dumps were restored into the new environment.
- Upload directories or other persistent volumes were copied.
- Restore checks were done in the app, not just in the shell.
- Secrets and credentials were rechecked after restore.

### 4. Test domains work before production domains move

- Each app has a temporary domain or internal test path.
- Login, main request flow, and one critical write action were tested.
- Background jobs run in Coolify, not only web requests.

## Minimal Command Checklist

The exact commands depend on your stack, but the boring checks below are the right baseline.

```bash
# snapshot current state before migration
docker ps
docker volume ls
git rev-parse HEAD
```

```bash
# database dump example
docker exec caprover-postgres pg_dump -U myuser mydb > mydb.sql
```

```bash
# volume copy example
rsync -avz /var/lib/docker/volumes/source-volume/ user@coolify-vps:/var/lib/docker/volumes/target-volume/
```

Those commands are not the migration. They are the proof that you are moving in a controlled way instead of relying on memory.

## DNS Cutover Checklist

Do these in order.

1. Lower TTL before cutover day.
2. Confirm the app works on Coolify under a test domain.
3. Confirm the database on Coolify has the current data.
4. Confirm background workers or scheduled tasks are running on Coolify.
5. Move DNS.
6. Watch logs on both sides during the propagation window.
7. Recheck SSL issuance and the main user path after traffic lands.

If step 2, 3, or 4 is still uncertain, do not move DNS yet. That is where self-hosting migrations become avoidable incidents.

## Post-Cutover Checklist

The migration is not done when the homepage loads once.

- Check the main logged-out path.
- Check the main logged-in path.
- Check one write action:
  form submit, DB write, upload, or webhook.
- Check app logs after real traffic hits.
- Check job runners, cron, or queues.
- Keep CapRover online during observation.
- Wait at least a few days, and a full week if the app has weekly jobs or quieter edge paths.

If the move exposed backup, restore, or health-check gaps, go straight to [Docker Compose Health Checks That Actually Catch Bad Deploys](/blog/docker-compose-health-checks-production-2026), [Docker Volume Backup Strategy on VPS](/blog/docker-volume-backup-strategy-vps-2026), and the [Self-Hosting Complete Guide](/guides/self-hosting-complete-guide).

## Common Checklist Failures

### Missing env vars

The most common migration bug is not "Coolify is broken." It is a missing or mis-scoped environment variable. Recheck build-time versus runtime values for Next.js and similar stacks.

### Worker or cron drift

Teams remember the web app and forget the worker, scheduler, or one-off job container. That is why the inventory step matters so much.

### Data copied but not verified

A successful file copy is not proof. Open the app and verify the data is actually where the app expects it.

### DNS moved too early

The clean rule is simple:
test domain first, production domain last.

## FAQ

### Can I keep both CapRover and Coolify live during migration?

Yes, and for most teams that is the safer path. Parallel operation makes the rollback story much cleaner because you are not betting everything on one cutover moment.

### Should I migrate on the same VPS?

Only if you are very clear on port conflicts, reverse proxy behavior, and resource headroom. A separate VPS is usually the calmer migration path.

### What should I back up first?

Back up the stateful parts first:
databases, uploaded files, and any volume-backed app data. Platform setup matters, but data loss is usually the expensive failure.

### When is it safe to shut CapRover down?

After the app has survived real traffic, logs look normal, and scheduled or delayed tasks have run cleanly. For most real setups, that means not on the same day as the DNS cutover.
