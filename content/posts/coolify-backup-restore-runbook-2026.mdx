The mistake with Coolify backups is thinking the platform backup is the whole job. It is not. You need the platform state, the app data, the secrets, and a restore drill that proves the backups are actually usable.

This page is the runbook version of that job: what to save, what to test, and what order to restore when the platform is actually needed again.

![Backup and restore runbook cover for a Coolify-operated stack](/images/posts/coolify-backup-restore-runbook-2026/desktop-cover.jpg)
*This visual introduces the backup workflow before the article moves into the real restore drill screenshots.*

> **TL;DR**
>
> - Back up the Coolify control plane, app data, secrets, and databases as separate recovery layers.
> - Dump active databases with database-aware tools before you archive their underlying volumes.
> - Keep one local copy for speed and one off-box copy for real disaster recovery.
> - Restore at least one app on a scratch target before you trust the policy.

## Backup Scope Table

| Layer | What to capture | Why it matters | Restore test |
| --- | --- | --- | --- |
| Coolify platform state | project/app definitions, deploy settings, and service notes | gets the control plane back faster | restore one app definition on a scratch server |
| Environment and secrets | runtime env vars, callback URLs, SMTP settings, tokens | apps fail even when data is present if secrets are missing | compare restored env set against the source of truth |
| Databases | logical dumps before or alongside volume backup | live database files are a bad surprise under load | restore into a disposable container and run one real query |
| Named Docker volumes | uploads, persistent files, and app state | this is what users notice missing first | mount the restored volume and inspect the expected files |

Capture the backup controls before you rely on them. This is the fastest way to prove the platform layer is actually configured, not just assumed.

## When this runbook is the right one

Use it when:

- Coolify is already running production or near-production workloads
- at least one app has stateful data you cannot afford to recreate
- you need a restore sequence, not just a list of storage commands

## Prerequisites

- SSH access to the VPS
- enough disk to stage backups temporarily
- an off-box destination for the final backup copy
- a list of which apps own databases, uploads, or other named volumes
- a restore target: scratch VPS, disposable container, or test environment

## Step 1: Define recovery targets before you back anything up

Do not start with commands. Start with recovery questions:

- what do you need back first: the Coolify UI, one app, or the full stack?
- how much data loss is acceptable?
- how long can restore take before it becomes a production incident?
- who owns the restore if you are not available?

These answers determine backup cadence and what "good enough" really means.

## Step 2: Separate Coolify state from app data

This is the most important distinction in the whole runbook.

Backing up Coolify state helps you rebuild the deployment layer faster.

Backing up app data helps you bring the actual workload back.

If you only restore the control plane, you may get a reassuring dashboard with missing uploads, stale databases, or broken secrets. That is not recovery. That is partial theater.

## Step 3: Use app-aware backups for databases

For databases, prefer a logical dump or app-aware backup first. A raw volume copy can still be useful, but it should not be your only plan for active databases.

Example for a PostgreSQL-backed app:

```bash
docker exec postgres pg_dump -U appuser appdb > appdb-$(date +%F).sql
```

Then back up the related volume if that is also part of your recovery model.

The point is consistency. A volume archive taken while a database is actively writing may not be the restore you think it is.

## Step 4: Back up named volumes explicitly

For uploads or file-based state, a volume archive is usually appropriate:

```bash
docker run --rm \
  -v app_uploads:/volume \
  -v $(pwd):/backup \
  alpine tar czf /backup/app_uploads-$(date +%F).tgz -C /volume .
```

That pattern is fine for persistent file data. It is not enough by itself for active databases.

This is where you prove the job produced a real artifact with a timestamp and a destination you can inspect later.

## Step 5: Store at least one copy off the VPS

If the only copy lives on the same server, you do not really have a backup strategy. You have an inconvenience strategy.

At minimum:

- keep one recent local copy for fast restores
- copy one backup off the VPS
- record where the off-box copy lives and who can access it

## Step 6: Run a restore drill before you need one

This is where most backup confidence falls apart.

Your restore drill should prove:

- the archive can be read
- the volume or dump restores cleanly
- the app boots with the restored data
- one real user path works afterward

Do not restore straight over production first. Use a scratch environment, a disposable container, or a separate server if you have one.

For a volume restore, the pattern is usually:

```bash
docker run --rm \
  -v app_uploads:/restore \
  -v $(pwd):/backup \
  alpine sh -c "cd /restore && tar xzf /backup/app_uploads-2026-04-09.tgz"
```

Then start the app against the restored volume and test something real.

This is the screenshot that matters most in the whole runbook. If the scratch restore is visibly working, the backup is finally moving from theory into proof.

## Verification checklist

Do not call the runbook healthy until you can confirm:

- Coolify settings needed for the app are documented
- required env vars can be recreated safely
- at least one database dump restores cleanly
- at least one named volume restore has been tested
- one off-box copy exists
- the first post-restore request behaves the way you expect

## Common failure modes

### Only the platform was backed up

That gets you the UI, not the actual application state.

### Database volumes were copied live without a consistency plan

For active databases, that is not the same as a trustworthy restore.

### Secrets were missing during restore

Backups of data without the runtime configuration often produce an app that starts but does not function.

### The first restore attempt happened during an incident

That is exactly what this runbook is supposed to prevent.

## Rollback and recovery notes

If a restore fails:

1. stop widening the blast radius
2. keep the original backup artifacts untouched
3. capture exactly which layer failed: platform, secrets, database, or volume
4. move to the last known-good artifact, not a half-tested new one

## FAQ

### What should a Coolify backup definitely include?

It should include deployment state, env settings, and the actual application data layer. Backing up only one of those is not enough.

### Is a raw Docker volume backup enough for a database?

Not by itself. Prefer a logical dump or app-aware backup for active databases, then use volume archives as an additional layer where useful.

### How often should I test restore?

Often enough that the first restore is not happening during the worst possible day. Even a lightweight monthly drill is better than blind confidence.

### Should I restore directly onto production to test the backup?

No. Restore to a scratch target first whenever possible.

### What is the minimum restore drill worth doing?

Restore one real backup into a scratch target, start the app against it, and verify one user-facing path plus one admin check such as login, upload access, or a database query. If you only confirm that the archive extracts, you still have not proved the app is recoverable.
