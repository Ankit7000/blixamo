The dangerous Coolify backup plan is the one that can rebuild the dashboard but not the app that actually matters. In practice, recovery fails when platform state, secrets, databases, and volume data are treated like one blob instead of separate restore layers.

This page is the operator runbook for the real job: define what must come back first, capture each layer with the right method, and run a restore drill that proves the backup is usable before an outage makes the test mandatory.

> **TL;DR**
>
> - Back up the Coolify control plane, secrets, databases, and file volumes as separate recovery layers.
> - Use database-aware dumps before you archive any active database volume.
> - Keep one fast local copy and one off-box copy, then verify one scratch restore end to end.
> - Do not trust a policy that has never restored one real app successfully.

## What this runbook is for

Use it when:

- Coolify is already running production or near-production workloads
- at least one app has user data, uploads, or state you cannot recreate quickly
- you need a restore order you can follow under pressure, not just backup commands

It is not for the "I am still learning Coolify" stage. If you are there, start with [Coolify Guide 2026 - Self-Host Everything for EUR5/Month](/blog/coolify-complete-guide-2026) first and come back when the recovery path matters.

## Backup Scope Table

| Layer | What to capture | Why it matters | Restore proof |
| --- | --- | --- | --- |
| Coolify platform state | project/app definitions, deploy settings, service notes | helps rebuild the control plane faster | recreate one app definition on a scratch server |
| Environment and secrets | runtime env vars, callback URLs, SMTP settings, tokens | apps fail even when data is present if secrets are missing | compare restored env set against source of truth |
| Databases | logical dumps before or alongside volume backup | live database files are a bad surprise under load | restore into a disposable container and run one real query |
| Named Docker volumes | uploads, persistent files, and app state | this is what users notice missing first | mount the restored volume and inspect expected files |

One useful standard: do not call the backup layer healthy unless you can point to one artifact outside the running app and one test that proves it can be restored.

Capture the backup controls before you rely on them. This is the fastest way to prove the platform layer is actually configured, not just assumed.

## Prerequisites

- SSH access to the VPS
- enough disk to stage backups temporarily
- an off-box destination for the final backup copy
- a list of which apps own databases, uploads, or other named volumes
- a scratch restore target: disposable VPS, test container, or separate environment

## Step 1: Define recovery targets before you back anything up

Do not start with commands. Start with recovery questions:

- what must come back first: the Coolify UI, one app, or the full stack?
- how much data loss is acceptable?
- how long can restore take before it becomes an incident?
- who owns the restore if you are unavailable?

These answers decide cadence, retention, and the restore order that actually matters under pressure.

## Step 2: Separate Coolify state from app data

This is the line that prevents a lot of fake confidence.

Backing up Coolify state helps you rebuild the deployment layer faster.

Backing up app data helps you restore the workload users care about.

If you only restore the platform layer, you can end up with a healthy-looking dashboard and missing uploads, stale databases, or broken secrets. That is not recovery. It is a partial reconstruction.

## Step 3: Use app-aware backups for databases

For databases, prefer a logical dump or app-aware backup first. A raw volume copy can still help as a secondary layer, but it should not be your only plan for active databases.

Example for PostgreSQL:

```bash
docker exec postgres pg_dump -U appuser appdb > appdb-$(date +%F).sql
```

Then archive the related volume only if it helps your recovery model. The goal is consistency, not just motion.

## Step 4: Back up named volumes explicitly

For uploads or file-based state, a volume archive is usually appropriate:

```bash
docker run --rm \
  -v app_uploads:/volume \
  -v $(pwd):/backup \
  alpine tar czf /backup/app_uploads-$(date +%F).tgz -C /volume .
```

That pattern is useful for persistent file data. It is not enough by itself for active databases.

This is also the step where the backup must leave behind a real artifact with a timestamp and a location you can inspect later.

## Step 5: Store at least one copy off the VPS

If the only copy lives on the same server, you do not really have a recovery strategy.

At minimum:

- keep one recent local copy for fast restores
- copy one backup off the VPS
- record where the off-box copy lives and who can access it

## Step 6: Run a restore drill before you need one

This is where most backup confidence falls apart.

Your restore drill should prove:

- the archive can be read
- the volume or dump restores cleanly
- the app boots with the restored data
- one real user path works afterward

Do not restore straight over production first. Use a scratch environment, disposable container, or separate server.

For a volume restore, the pattern is usually:

```bash
docker run --rm \
  -v app_uploads:/restore \
  -v $(pwd):/backup \
  alpine sh -c "cd /restore && tar xzf /backup/app_uploads-2026-04-09.tgz"
```

Then start the app against the restored volume and test something real.

This is the screenshot that matters most in the runbook. If the scratch restore is visibly working, the backup is finally moving from policy into proof.

## Verification checklist

Do not call the runbook healthy until you can confirm:

- Coolify settings needed for the app are documented
- required env vars can be recreated safely
- at least one database dump restores cleanly
- at least one named volume restore has been tested
- one off-box copy exists
- the first post-restore request behaves the way you expect

## Common failure modes

### Only the platform was backed up

That gets you the UI, not the application state.

### Database volumes were copied live without a consistency plan

For active databases, that is not the same thing as a trustworthy restore.

### Secrets were missing during restore

Data without runtime configuration often produces an app that starts but does not actually function.

### The first restore attempt happened during an incident

That is exactly what this runbook is supposed to prevent.

## Rollback and recovery notes

If a restore fails:

1. stop widening the blast radius
2. keep the original backup artifacts untouched
3. capture exactly which layer failed: platform, secrets, database, or volume
4. move to the last known-good artifact, not a half-tested new one

## FAQ

### What should a Coolify backup definitely include?

It should include deployment state, env settings, and the actual application data layer. Backing up only one of those is not enough.

### Is a raw Docker volume backup enough for a database?

Not by itself. Prefer a logical dump or app-aware backup for active databases, then use volume archives as a secondary layer where useful.

### How often should I test restore?

Often enough that the first restore is not happening during the worst possible day. Even a lightweight monthly drill is better than blind confidence.

### Should I restore directly onto production to test the backup?

No. Restore to a scratch target first whenever possible.

### What is the minimum restore drill worth doing?

Restore one recent backup into a scratch target, start the app against it, and verify one user-facing path plus one admin check such as login, upload access, or a database query.
