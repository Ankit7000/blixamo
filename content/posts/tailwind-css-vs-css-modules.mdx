## The Context That Changes Everything in 2026

Two things shifted the Tailwind vs CSS Modules debate in 2026:

First, **Next.js App Router** changed how styles load. With React Server Components, Tailwind integrates better — utility classes generate zero runtime JS, which matters for Core Web Vitals. CSS-in-JS solutions broke badly with RSC; CSS Modules and Tailwind both survived.

Second, **AI-assisted development** changed the speed equation. GitHub Copilot, Cursor, and other tools suggest Tailwind classes directly in JSX. Writing CSS Modules with AI assistance is slower because the model has to context-switch between your `.tsx` and `.module.css` files.

If you're building on Next.js 15 with the App Router in 2026, both approaches work. But they work differently.

The [Next.js MDX blog setup](/blog/nextjs-mdx-blog-2026) I documented uses Tailwind exclusively — worth checking if you're building a content site.

## What Is Tailwind CSS?

Tailwind is a utility-first CSS framework. Instead of writing this:

```css filename="button.module.css"
.button {
  padding: 8px 16px;
  background-color: #3b82f6;
  color: white;
  border-radius: 6px;
  font-weight: 500;
}
```
You write this directly in JSX:
#
```tsx filename="Button.tsx"
export function Button({ children }) {
  return (
    <button className="px-4 py-2 bg-blue-500 \
      text-white rounded-md font-medium">
      {children}
    </button>
  );
}
```
Tailwind's JIT (Just-In-Time) compiler scans your files and generates only the CSS you actually use. A production Tailwind bundle is typically 5-15KB — smaller than most hand-written CSS files.

## What Are CSS Modules?
#
CSS Modules are built into Next.js. Any file ending in `.module.css` gets auto-processed so class names become locally scoped. No naming conflicts between components.
#
```tsx filename="Button.tsx"
import styles from './Button.module.css';
//
export function Button({ children }) {
  return (
    <button className={styles.button}>
      {children}
    </button>
  );
}
```
```css filename="Button.module.css"
.button {
  padding: 8px 16px;
  background-color: #3b82f6;
  color: white;
  border-radius: 6px;
  font-weight: 500;
}
```
The compiled output becomes `.Button_button__abc123` — unique, scoped, no conflicts.

## Bundle Size — Real Numbers
#
This is where most comparisons lie. "Tailwind is smaller" is only true if you compare it to badly-written CSS.
#
On blixamo.com (20+ pages, 40+ components), the Tailwind production bundle is **11KB gzipped**. The same project with CSS Modules would likely be 18-25KB — I have a smaller client project at 22KB CSS Modules gzipped with fewer components.

Tailwind wins on bundle size for component-heavy apps because atomic utility classes are shared across components. `.flex` is generated once, not once per component.

<Callout type="info" title="The CSS Modules catch">
CSS Modules include all styles from every imported module. Even if two components use identical styles, they generate separate class names and separate CSS rules. Tailwind avoids this by design.

## The Hybrid Approach — What Production Apps Actually Use
#
Here is what blixamo.com's styling stack looks like:
#
```tsx filename="components/BlogPost.tsx"

//

  return (
    <article className="max-w-3xl mx-auto px-4 py-8">
```

This is the actual Next.js recommendation: Tailwind for layout, spacing, and common patterns. CSS Modules for complex, component-specific styles that would produce unreadable Tailwind class strings.

## Which One Should You Use?
#
**Choose Tailwind CSS if:**
- You are a solo developer or a team of 1-3
- You are building an MVP and speed matters more than structure
- You are using AI coding assistants (Copilot, Cursor)
- Your project uses shadcn/ui or other Tailwind-based component libraries
#
**Choose CSS Modules if:**
- Your team has 5+ frontend developers
- You have a designer who writes CSS but not React
- You are migrating a large existing project with established CSS
- You need complex animations that require `@keyframes` and multi-step transitions
#
**Use both if:**
- You are building a production Next.js app with complex content areas (like a blog or docs site)
- You want Tailwind for speed and CSS Modules for the parts that need precise control

## Troubleshooting Common Issues
#
### Tailwind classes not applying
#
Check your `tailwind.config.ts` content array. It must include every file path that uses Tailwind classes:
#
```ts filename="tailwind.config.ts"
import type { Config } from 'tailwindcss'
//
const config: Config = {
  content: [
    './src/pages/**/*.{js,ts,jsx,tsx,mdx}',
    './src/components/**/*.{js,ts,jsx,tsx,mdx}',
    './src/app/**/*.{js,ts,jsx,tsx,mdx}',
    './content/**/*.{mdx,md}', // Don't forget MDX files!
  ],
  theme: {
    extend: {},
  },
  plugins: [],
}
export default config
```
<Callout type="warning" title="Common mistake">
Forgetting `./content/**/*.mdx` in the content array means any Tailwind classes inside MDX files are purged in production. This is a hard-to-debug issue because it only shows up after `npm run build`.
</Callout>
#
### CSS Module styles not scoping correctly
#
Check you are importing the module correctly. The class must come from the `styles` object:
#
```tsx filename="Component.tsx"
// WRONG — class not scoped, will match globally
<div className="button">
//
// CORRECT — scoped to this component
import styles from './Component.module.css';
<div className={styles.button}>
```
### Tailwind and CSS Modules conflict
#
If you use both in the same component, specificity can cause unexpected results. Add `!important` or increase specificity in your Module for overrides:
#
```css filename="Component.module.css"
/* Override Tailwind's default */
.customCard {
  background-color: #custom !important;
}
```
## Frequently Asked Questions
#
### Is Tailwind CSS better than CSS Modules for SEO?
#
Neither affects SEO directly. Both produce static CSS with no runtime overhead. What matters for SEO is CSS bundle size (affects LCP) and render-blocking styles. Tailwind produces smaller bundles via JIT, which can improve Core Web Vitals scores by 2-5 points on Lighthouse in component-heavy apps.
#
### Can you use Tailwind and CSS Modules together in Next.js?
#
Yes, and Next.js officially recommends this hybrid approach. Use Tailwind for most styling via utility classes and CSS Modules for complex component-specific styles that would produce unreadable class strings. Both approaches compile to static CSS and are fully compatible with the App Router.
#
### Does Tailwind CSS work with React Server Components?
#
Yes. Tailwind generates pure static CSS — no JavaScript, no runtime. This makes it ideal for React Server Components in Next.js 15. CSS-in-JS solutions like styled-components and Emotion require additional configuration with RSC and some features don't work at all.
#
### Is Tailwind CSS harder to maintain than CSS Modules?
#
It depends on team size and discipline. Tailwind on large teams requires component extraction discipline — if the same class string repeats more than 3 times, it should become a component. Without this discipline, Tailwind becomes a maintenance problem. CSS Modules are easier to audit and refactor at scale because each component has an isolated `.module.css` file.
#
### What is the Tailwind CSS bundle size in 2026?
#
With Tailwind's JIT compiler, production bundles are typically 5-20KB gzipped. The JIT compiler scans your files and generates only the CSS classes you actually use. Compare this to Bootstrap at ~140KB or Foundation at ~120KB. CSS Modules bundle size depends entirely on how much CSS you write per component.
#
### Should I use Tailwind with shadcn/ui?
#
Yes. shadcn/ui is built on Tailwind and the two are designed to work together. If you are using shadcn/ui for components, Tailwind is the natural choice for the rest of your styling. Adding CSS Modules on top of a shadcn/ui + Tailwind project is possible but creates inconsistency — use Tailwind exclusively in this case.
#
### Does CSS Modules support dark mode?
#
Yes, via CSS custom properties and the `prefers-color-scheme` media query. However, Tailwind's `dark:` prefix is faster to write for simple dark mode implementations. For complex dark mode with many custom values, CSS Modules with CSS variables is more maintainable.

## Team Dynamics: Who's Using the Codebase?
#
This is underrated in most comparisons. The right choice depends heavily on who's maintaining the code.
#
**Solo developer:** Tailwind is faster. No context switching between files. Utility class muscle memory builds quickly.
#
**Team with designers:** CSS Modules (or CSS-in-JS) wins. Designers who write CSS are productive with `.module.css` files. Tailwind's class soup in JSX is hard to review in design handoff.
#
**Team with strict design system:** Tailwind + design tokens config wins. Define your colors, spacing, and typography scale in `tailwind.config.js` once. Every developer uses the same named values. No hardcoded colors scattered through module files.
#
**Team that onboards junior developers frequently:** CSS Modules. The mental model is simpler — one component, one file, traditional CSS syntax. Juniors don't need to learn Tailwind's class naming system before being productive.
#
**Open source projects:** CSS Modules — easier for contributors who don't know Tailwind.

## Maintainability Over 12 Months
#
I've maintained three Next.js apps for over a year — one with Tailwind, two with CSS Modules. Honest assessment:
#
**Tailwind over time:** The JSX stays readable if you enforce component extraction rigorously. Without that discipline, you end up with 20-class utility strings on every element. Tools like `prettier-plugin-tailwindcss` (sorts classes consistently) and `eslint-plugin-tailwindcss` (flags anti-patterns) help significantly.
#
**CSS Modules over time:** Files accumulate. A component with 15 variants has a long `.module.css` file with classes that are hard to find without text search. Renaming a class requires updating both the module file and every place it's used in JSX. TypeScript helps here — typed CSS Modules with `typescript-plugin-css-modules` gives you autocomplete and rename-across-files.
#
The maintenance advantage shifts based on team discipline. With good conventions, both are maintainable at 12 months. Without conventions, both become messy in different ways.

## Migration Guide: CSS Modules → Tailwind
#
If you're switching an existing codebase, this is the lowest-risk approach:
#
**Phase 1 — New components only** (weeks 1–4): Write all new components with Tailwind. Don't touch existing CSS Modules.
#
**Phase 2 — Leaf components** (weeks 5–8): Migrate small, isolated components (buttons, badges, cards). These have simple styles and are easy to convert without breaking layout.
#
**Phase 3 — Layout components** (weeks 9–12): Migrate headers, footers, navigation — components that define page structure. These are more complex but affect the entire app.
#
**Phase 4 — Delete module files**: Once a component is fully migrated, delete its `.module.css` file. Watch bundle size decrease.
#
Don't do a big-bang rewrite. Component-by-component migration keeps the app working throughout. A realistic timeline for a 30-component app: 6–10 weeks part-time.
