If you already know you want **n8n + FastAPI on one Hetzner VPS**, this is the setup I would actually ship in 2026: PostgreSQL for persistence, Redis for queue-ready execution, Nginx for SSL and routing, and one box that stays simple enough to debug at 2 AM.

If you are still deciding whether to self-host n8n at all, start with the [homepage](/), the [blog](/blog), the [community hub](/community), the [deployment resources hub](/tag/deployment), the [Automation category](/category/automation), the [Automation Guide for Developers](/guides/automation-guide-for-developers), the deeper [n8n Complete Guide](/blog/n8n-complete-guide-2026), and the full [Self-Host n8n on Hetzner VPS](/blog/self-hosting-n8n-hetzner-vps) walkthrough. This article is the next step: putting your API beside n8n cleanly, not just getting the editor online.

## TL;DR

- Use **one Hetzner VPS** for n8n + FastAPI only if you want low fixed cost and simple operations.
- Start with **PostgreSQL + Redis + Docker Compose + Nginx**. Do not run production n8n on SQLite if you already know this stack matters.
- Keep **Postgres and Redis private**, expose only 80/443, and route `/webhook/*` and `/api/*` deliberately.
- Add **health checks, backups, log review, and workflow alerts** before you trust the stack.
- When workflow concurrency gets real, **move n8n to queue mode and split workers before you split everything else**.

## Who this stack is for

This setup is a good fit if you are building:

- internal tools with scheduled jobs and webhook automation
- a SaaS backend where FastAPI owns business logic and n8n handles orchestration
- lead routing, notifications, sync jobs, and background actions on one box
- side projects where predictable monthly cost matters more than multi-region architecture

This setup is a bad fit if you need:

- strict high-availability from day one
- heavy media pipelines or CPU-heavy AI inference on the same box
- separate ops boundaries between API, workflows, and data teams
- compliance rules that push you toward managed infrastructure immediately

## Why n8n + FastAPI works well on one VPS

n8n and FastAPI solve different problems. n8n is great at orchestration, webhook handling, retries, scheduled automation, and glue code. FastAPI is better for typed application logic, validation, auth, and stable API contracts.

That split matters. When teams force all logic into n8n, workflows get hard to test and harder to reason about. When teams force every integration into the app codebase, they lose the speed that makes n8n useful in the first place.

The clean boundary is simple:

- **FastAPI owns core business logic**
- **n8n owns event flow and orchestration**
- **PostgreSQL stores durable application data**
- **Redis supports queue-ready execution and background reliability**
- **Nginx owns public exposure**

That gives you one predictable runtime story. Your API stays code-first. Your automation stays visible. Your box stays understandable.

## Server size and cost reality

A single Hetzner box is the right starting point because it keeps your operational surface small. If the stack is light to moderate, one VPS is easier to secure, back up, and recover than a scattered “microservices” setup you do not actually need.

My default starting point is:

| Layer | Starting choice | Notes |
|---|---|---|
| Compute | CPX22 class VPS | Good starting point for small production loads |
| OS | Ubuntu 24.04 | Stable and easy to support |
| Data | PostgreSQL volume | Keep it persistent and backed up |
| Queue | Redis | Small footprint, big operational payoff |
| App | FastAPI in Docker | Keep one image per app |
| Automation | n8n in Docker | Use Postgres, not SQLite |
| Edge | Nginx + Certbot | Only public layer |

If you are already running other apps on the same machine, read [Run Multiple Projects on One VPS in 2026](/blog/multiple-projects-single-vps) before you pile more services onto it.

## Folder layout I would use

Keep the stack boring:

```text
/opt/automation-stack
  docker-compose.yml
  .env
  /fastapi
    Dockerfile
    requirements.txt
    app/
  /nginx
    app.conf
  /backups
```

One folder. One compose file. One `.env`. One place to back up. One place to inspect.

## Docker Compose: infrastructure services

Start with the data layer first.

```yaml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: app
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes
    volumes:
      - redis_data:/data
    restart: unless-stopped
```

That is enough to give both n8n and FastAPI a stable persistence layer. The big rule: **do not publish Postgres or Redis ports to the public internet**.

## Docker Compose: app services

Now add the API and automation layer.

```yaml
services:
  fastapi:
    build: ./fastapi
    env_file: .env
    depends_on: [postgres, redis]
    restart: unless-stopped

  n8n:
    image: n8nio/n8n:latest
    env_file: .env
    depends_on: [postgres, redis]
    volumes:
      - n8n_data:/home/node/.n8n
    restart: unless-stopped
```

For a small deployment, this is enough to get moving. For a busier deployment, switch n8n to queue mode and add a worker before you start talking about splitting everything into separate servers.

## Docker Compose: edge layer

Only Nginx should be public.

```yaml
services:
  nginx:
    image: nginx:1.27-alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx/app.conf:/etc/nginx/conf.d/default.conf:ro
      - /etc/letsencrypt:/etc/letsencrypt:ro
    depends_on: [fastapi, n8n]
    restart: unless-stopped

volumes:
  postgres_data:
  redis_data:
  n8n_data:
```

That is the whole runtime shape: data, apps, edge. Nothing fancy, and that is exactly why it works.

## Environment variables that matter

Use one `.env` file for Docker Compose and keep secrets out of Git.

```bash
POSTGRES_PASSWORD=replace-me
N8N_HOST=n8n.example.com
N8N_PROTOCOL=https
N8N_PORT=5678
WEBHOOK_URL=https://n8n.example.com/
DB_TYPE=postgresdb
DB_POSTGRESDB_HOST=postgres
DB_POSTGRESDB_PORT=5432
DB_POSTGRESDB_DATABASE=app
DB_POSTGRESDB_USER=app
DB_POSTGRESDB_PASSWORD=${POSTGRES_PASSWORD}
QUEUE_BULL_REDIS_HOST=redis
QUEUE_BULL_REDIS_PORT=6379
FASTAPI_DATABASE_URL=postgresql+psycopg://app:${POSTGRES_PASSWORD}@postgres:5432/app
```

If you know the stack will carry real asynchronous load, add `EXECUTIONS_MODE=queue` for n8n and move to a worker pattern early. If you do not know that yet, keep the first deployment simple and upgrade once the workload proves the need.

## A minimal FastAPI Dockerfile

Keep the API image tiny and predictable.

```dockerfile
FROM python:3.12-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY app ./app
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

I would also add a `/health` route in FastAPI on day one. That gives you an easy probe target for Nginx checks, uptime tools, and deployment validation.

## Nginx routing: keep public paths deliberate

I prefer separate subdomains:

- `n8n.example.com` for the editor and webhook surface
- `api.example.com` for FastAPI

That is cleaner than trying to share everything on one domain path map.

### Nginx config for the API

```nginx
server {
  server_name api.example.com;

  location / {
    proxy_pass http://fastapi:8000;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
```

### Nginx config for n8n

```nginx
server {
  server_name n8n.example.com;

  location / {
    proxy_pass http://n8n:5678;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
```

After that, issue certificates with Certbot and force HTTPS. If your editor is private, put basic auth or IP restrictions in front of the n8n editor route instead of assuming the login screen is enough.

## The production defaults I would not skip

This is where thin tutorials usually fail. The stack is not “done” because the containers started.

### 1. Lock down the server

Use UFW. Keep only:

- 22 for SSH
- 80 for HTTP
- 443 for HTTPS

Do **not** expose 5432, 6379, 5678, or 8000 publicly. If you need a deeper checklist, follow [Ubuntu VPS Hardening Checklist](/blog/ubuntu-vps-hardening-checklist) and [VPS Security Hardening in 2026](/blog/vps-security-harden-ubuntu-2026).

### 2. Put backups on a schedule

At minimum, back up:

- PostgreSQL
- the n8n data directory
- the `.env` file
- your Nginx config

If you do not test restores, you do not have backups. Read [Verify a VPS Backup Before You Trust It in 2026](/blog/verify-vps-backup-before-trusting-it-2026) before you assume snapshots are enough.

### 3. Add health checks before the first real workflow

You want checks for:

- public HTTP 200 on the API
- public HTTP 200 on the n8n editor
- a real webhook round-trip
- disk usage
- memory pressure
- container restart loops

That is the difference between “it worked yesterday” and an operational setup you can trust.

### 4. Pin your image versions

Do not stay on `latest` forever. Test a version. Pin it. Upgrade intentionally.

### 5. Prune execution history

n8n can grow noisy over time. Old executions, logs, and binary data can turn a good server into a slow one if you never prune.

## When to move n8n into queue mode

Do not start with queue mode just to feel advanced. Start with it when one or more of these becomes true:

- multiple workflows can run at the same time
- webhook bursts start blocking other work
- long-running jobs make the editor feel sluggish
- retries and failures spike during traffic bursts
- you want to control worker concurrency separately

Once that happens, Redis stops being “maybe useful later” and becomes part of the core runtime. Queue mode is the right next step, not a full architectural rewrite.

## Monitoring and alerts: the part most people postpone

Monitoring is where this stack becomes real.

The minimum alert set I would add:

| Signal | Why it matters |
|---|---|
| API health endpoint failing | FastAPI is down or stuck |
| n8n editor unreachable | Reverse proxy, container, or TLS issue |
| webhook latency rising | Your workflows are about to become user-visible pain |
| disk > 80% | Postgres, logs, or execution data is filling the box |
| restart count increasing | You have a real stability issue, not a one-off |
| backup job missing | Quiet failure that only hurts later |

For the operational side, read [How I Monitor n8n Workflows in 2026](/blog/monitor-n8n-workflows-2026) and [Monitor a Small VPS Before Downtime in 2026](/blog/monitor-small-vps-before-downtime-2026).

## Common architecture mistakes

### Putting business logic in the wrong place

If the logic needs typed validation, auth, or stable API behavior, it belongs in FastAPI. If the logic is orchestration, scheduling, retries, notifications, or integration glue, it belongs in n8n.

### Publishing internal services directly

Do not expose Postgres or Redis. Do not expose raw FastAPI or raw n8n without the proxy layer unless you understand exactly why.

### Treating one VPS as infinite

A single server is a great starting point. It is not magic. Watch memory, disk, and execution patterns before you add more apps.

### Ignoring restore paths

A snapshot without a restore test is false confidence.

## Troubleshooting

### n8n loads, but webhooks fail

Most of the time this is a bad `WEBHOOK_URL`, a proxy header issue, or HTTPS misconfiguration. Fix the public URL first. Then trigger a real webhook from outside the VPS.

### FastAPI works locally but not through Nginx

Check the upstream name, the container network, and the forwarded headers. Also make sure your app is bound to `0.0.0.0`, not `127.0.0.1`.

### n8n gets slow after a few weeks

Look at execution history size, restart counts, memory pressure, and database growth before blaming Hetzner. A slow n8n instance is often an operations problem, not a provider problem.

### Redis is running, but queue mode does nothing

You probably forgot the n8n execution mode setting or you never added a worker. Redis by itself does not make n8n scalable.

### The box keeps running out of memory

Stop stacking unrelated services on the same VPS. Then check Postgres growth, container logs, and whether your FastAPI app is doing heavy work that should live in a queue.

### SSL renewals break the stack later

Automate renewal checks and verify the Nginx reload path. A certificate setup that worked once is not a certificate process.

## FAQ

### Should I put n8n and FastAPI on the same VPS?

Yes, if your workload is still small to moderate and you want one predictable operations surface. It is a strong setup for internal tools, automation-heavy SaaS backends, and side projects. Split them later when real load, team boundaries, or uptime requirements justify it.

### Is Hetzner good for n8n + FastAPI?

Yes. Hetzner is a strong fit when you want low fixed cost and direct server control. The catch is simple: you own updates, backups, and recovery. That trade-off is worth it for developer-led stacks. It is not worth it if you want managed everything.

### Should n8n use PostgreSQL or SQLite here?

Use PostgreSQL. SQLite is fine for quick testing, but once the stack matters, PostgreSQL is the better production choice. It gives you a cleaner path for reliability, backups, and queue-ready growth.

### Do I need Redis from day one?

Not always, but I would include it early if you already know the stack will grow. Redis makes the move to queue mode cleaner and helps you avoid redesigning the deployment after your first reliability problems.

### Should FastAPI call n8n or should n8n call FastAPI?

Usually n8n should call FastAPI for business actions that need typed validation or app state. FastAPI can also trigger n8n webhooks for background orchestration. The right answer is not “one replaces the other.” The right answer is keeping the boundary clean.

### What is the first sign I should split this stack into multiple servers?

When one box starts creating operational trade-offs you can feel: noisy webhook bursts, memory pressure, blocked deployments, worker contention, or a need to isolate risk between the API and workflow runtime. Split because the load forced it, not because a diagram on the internet looked cleaner.

## Final verdict

I would absolutely run **n8n + FastAPI on one Hetzner VPS** for an internal tool, automation backend, or early SaaS operations stack. It is cheaper, easier to understand, and faster to ship than the overbuilt alternatives most teams copy too early.

The exception is straightforward: if your API uptime matters more than your workflow runtime, or your traffic spikes are already unpredictable, separate FastAPI sooner and keep n8n behind it.

For most developer-run projects, though, the one-box version wins first. Make it boring. Make it observable. Make it recoverable. Then scale the exact layer that hurts.
