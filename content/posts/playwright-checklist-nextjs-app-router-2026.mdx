## Reliable Tests Come from Better Contracts

Flaky end-to-end tests are often blamed on the tool, but the real issue is a weak contract with the app. When selectors are unstable, auth is ad hoc, and environment state is messy, any test runner feels unreliable. The checklist should therefore begin with contracts: stable locators, predictable seed data, isolated browser contexts, and a clear line between smoke tests and deeper scenario tests.

Once those contracts exist, Playwright becomes much more pleasant.

## How to Use the Checklist Without Turning It into Ritual

A checklist works only when it changes behavior. That means the items must be concrete enough to stop a bad release, a bad restore, or a bad test run. â€œCheck everythingâ€ is not a checklist. â€œConfirm the database is reachable from the app containerâ€ is.

The second rule is ordering. Put the highest-risk items early. If one failed check should cancel the launch, it belongs near the top, not hidden in the middle. The third rule is evidence. ## Common Mistakes and How to Avoid Them

The first mistake is treating playwright checklist nextjs app router like a one-layer problem. Most failures span at least two layers: the config and the verification, the migration and the rollback, the tool choice and the collaboration model. If you only optimize the visible layer, the issue returns later in a different form.

The second mistake is hiding assumptions. Write them down. Which environment variables must exist? Which domain must resolve first? Which account permissions are required? Which health endpoint represents â€œgoodâ€? Short articles skip those details because they feel boring. In real work, those details are the difference between a clean handoff and an hours-long debugging session.

The third mistake is weak naming. Name collections, services, environments, and checkpoints clearly. Ambiguous labels create ambiguity in the review process too. Good names cut down on re-explanation.

The fourth mistake is skipping post-change observation. People celebrate once the main path works once. That is too early. Watch the system long enough to catch retry storms, delayed errors, silent bad defaults, or permissions issues that appear only on the second or third action.

## Troubleshooting and Verification

When something feels off, verify in this order.

1. **Inputs**: confirm the right variables, credentials, paths, domains, or source files are being used.
2. **Execution**: confirm the actual command, workflow step, or request you expect is the one being run.
3. **Dependencies**: confirm the database, queue, proxy, or upstream service is healthy before blaming the app.
4. **Output**: confirm the success signal is real and not a partial success hiding deeper issues.
5. **Rollback path**: decide early whether continuing is cheaper than restoring the last known good state.

This sequence matters because it keeps you from making a noisy situation noisier. For many developers, the hardest part of troubleshooting is not technical depth. It is emotional pacing. A failing migration or broken deploy creates urgency, and urgency pushes people toward random action. The verification order above slows that down just enough to make better choices.
