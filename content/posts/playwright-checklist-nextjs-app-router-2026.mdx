# Playwright Checklist for Next.js App Router in 2026

Playwright only pays off when the tests cover the parts that actually break in App Router apps: navigation, forms, auth, loading states, and server-client edges. A vague smoke suite will miss the expensive bugs.

This checklist helps you decide what to automate first and what should block release confidence.

> **TL;DR**
>
> - Test the flows App Router apps break most often: routing, auth, forms, loading states, errors, and cached data.
> - Use Playwright as a release gate for critical paths, not as a screenshot tour of every page.
> - Keep the suite small enough to run before deploy and specific enough to catch production-shaped bugs.

## Fast Pre-Launch Checklist

| Stage | What to confirm |
|---|---|
| Routes | Public, protected, dynamic, and not-found routes render the right state. |
| Auth | Sign-in, sign-out, redirects, expired sessions, and role checks work. |
| Forms | Server actions, validation errors, success states, and double submits behave correctly. |
| Loading and errors | `loading.tsx`, `error.tsx`, Suspense boundaries, and retry paths are visible when triggered. |
| Data freshness | Cached, revalidated, and user-specific data do not leak across sessions. |

## What You Need Before You Start

- Playwright installed with the browsers your app supports.
- A stable `baseURL` that points to the same build shape you plan to release.
- Test users for signed-out, signed-in, and restricted-role flows.
- Seed data for one realistic create, edit, and delete path.
- A rule for which failures block release and which become follow-up work.

## Why Playwright Checklist Next.js App Router Matters

App Router bugs often hide in route transitions, loading states, redirects, auth edges, server actions, cache-dependent UI, and forms that only fail once real data hits them. A useful Playwright checklist is a release checklist for those failure points, not just a list of pages to click.

Start with the failures that users notice first. A broken protected redirect, a form that submits twice, or a stale dashboard after mutation matters more than a low-value assertion that the footer exists.

## Build the Checklist Around Real Failure Points

Start with the flows that are expensive to ship broken: sign-in, sign-out, protected routes, the main form flow, one loading state, one error state, and at least one route that depends on server-rendered data. If those are covered, the checklist is already useful.

Write down what would block a release before you automate more tests. For many teams that means failed auth, broken navigation, unusable forms, or a route that renders locally but fails in a production build. Once those rules are visible, the checklist stops being decorative.

A practical starter sequence looks like this:

```ts
import { test, expect } from '@playwright/test'

test('home page loads', async ({ page }) => {
  await page.goto('/')
  await expect(page.getByRole('heading', { level: 1 })).toBeVisible()
})
```

That kind of first test should be boring. The checklist gets stronger when the first assertions are obvious and dependable.

## App Router Checks That Should Block Release

### 1. Route and navigation checks

Cover the routes that define the app: homepage or landing page, one dynamic route, one protected route, one 404 path, and one route that uses a loading state. Navigate through links instead of only calling `page.goto()` so you catch client-side routing problems too.

For App Router projects, include at least one direct refresh on a deep route. A page can pass client-side navigation but fail when loaded directly from the server.

### 2. Auth and session checks

Test signed-out access, successful sign-in, sign-out, and a restricted route. If your app has role-based access, test one allowed role and one denied role. Do not rely on a single stored login state unless you also verify the unauthenticated behavior.

Session bugs are expensive because they often pass locally and fail only after cookies, redirects, or middleware behave differently in production.

### 3. Form and server action checks

Every important form should have tests for valid submit, validation error, duplicate submit protection, and the post-submit UI. For server actions, assert the user-visible result instead of only waiting for a network call.

If a mutation should refresh cached data, verify that the changed value appears after submit without requiring a manual page refresh.

## Loading, Error, and Cache States

App Router makes loading and error UI first-class, so the checklist should test them on purpose. Trigger a slow data response or use a controlled test route to confirm `loading.tsx` appears. Trigger a handled failure to confirm the error boundary gives the user a usable next step.

Cache behavior needs the same discipline. Check one route that reads cached data, one route that should revalidate after a mutation, and one user-specific page that must not show another user's data. These tests catch the kind of bug that a generic smoke suite will never see.

## Reliable Tests Come from Better Contracts

Flaky end-to-end tests are often blamed on the tool, but the real issue is a weak contract with the app. When selectors are unstable, auth is ad hoc, and environment state is messy, any test runner feels unreliable. The checklist should therefore begin with contracts: stable locators, predictable seed data, isolated browser contexts, and a clear line between smoke tests and deeper scenario tests.

Once those contracts exist, Playwright becomes much more pleasant.

## How to Use the Checklist Without Turning It into Ritual

A checklist works only when it changes behavior. That means the items must be concrete enough to stop a bad release, a bad restore, or a bad test run. "Check everything" is not a checklist. "Confirm the database is reachable from the app container" is.

The second rule is ordering. Put the highest-risk items early. If one failed check should cancel the launch, it belongs near the top, not hidden in the middle. The third rule is evidence. A useful checklist item implies a verifiable proof: a successful request, a clean log line, a passing health endpoint, or a screenshot of a green status.

Teams start hating checklists when the list becomes ceremonial. Keep it short enough to use under pressure and precise enough to prevent avoidable mistakes.

## Common Mistakes and How to Avoid Them

The first mistake is testing only static page loads. App Router failures usually appear during navigation, redirects, mutation, refresh, or session changes.

The second mistake is letting selectors drift. Prefer role-based locators and stable labels. If the test needs brittle CSS selectors everywhere, the UI probably needs better accessible names.

The third mistake is using one all-powerful test user. Real apps break when signed-out users, restricted roles, and expired sessions follow different routes.

The fourth mistake is ignoring trace output. When a pre-launch check fails, keep the trace, screenshot, and console error so the failure can be fixed instead of re-run until it disappears.

## Troubleshooting and Verification

When something feels off, verify in this order.

1. **Environment**: confirm the test points to the intended build, database, feature flags, and auth provider.
2. **State**: confirm seed data exists and each test starts from a predictable browser context.
3. **Locator**: confirm the failing selector reflects real user-visible UI.
4. **App Router behavior**: confirm redirects, middleware, cache, and server actions behave the same in production mode.
5. **Evidence**: open the Playwright trace before guessing at the fix.

This sequence keeps Playwright failures useful. The goal is not a permanently green badge. The goal is a test suite that points to the release risk quickly.

## FAQ

### How do I use a playwright checklist nextjs app router without turning it into paperwork?

Keep the checklist short enough to use under pressure. If an item does not block a bad release or prove something important, it probably belongs in notes rather than in the main checklist.

### Which checks should happen before execution starts?

Confirm the environment, test data, auth state, feature flags, and deployment target before you start the run. Those assumptions cause more false confidence than Playwright itself.

### What evidence should each checklist item require?

Each checklist item should point to proof. That can be a passing test, a screenshot, a clean log line, or a specific request that confirms the app behaves the way the team expects.

### How often should the checklist be reviewed?

Review the checklist whenever a release incident exposes a missing step, and also after framework upgrades that change routing, caching, auth, or loading behavior. A checklist should age with the app, not sit frozen.

### Who should own checklist updates?

Ownership should sit with the team that feels the release pain first. In most cases that is the developer shipping the feature, with QA or platform review added for the high-risk items.

### What makes a checklist trustworthy during incidents?

A trustworthy checklist works during incidents because it is concrete, ordered, and already familiar to the team. You should be able to run it quickly without rewriting the steps while production is noisy.
