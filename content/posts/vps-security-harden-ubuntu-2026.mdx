## Why This Matters for Indie Developers

The threat model for indie developers is not sophisticated nation-state attacks. It's automated bots scanning the entire internet for default credentials and unpatched services — 24/7, hitting every IP within hours of provisioning.

A compromised VPS doesn't just take down your projects. It can send spam (getting your IP blacklisted), mine cryptocurrency (spiking your CPU bill), or exfiltrate API keys stored on the server.

The good news: basic hardening stops 99% of this. SSH key auth eliminates the entire class of password brute-force attacks, and the dedicated [SSH hardening guide](/blog/ssh-security-hardening-vps-2026) goes deeper if you want the access layer locked down before everything else. UFW drops all unrecognized traffic. fail2ban handles the rest. Let's go through each step.

Never run applications or day-to-day work as root. Create a user with sudo access:


## Step 2 of 8 — Set Up SSH Key Authentication
#
This is the most important step. Password auth can be brute-forced. Key auth cannot.
#
On your **local machine** (not the server):
#
```bash
# Generate a new SSH key pair if you don't have one
ssh-keygen -t ed25519 -C "your@email.com"
#
# Copy public key to server
ssh-copy-id -i ~/.ssh/id_ed25519.pub ankit@YOUR_SERVER_IP
#
# Test it works BEFORE proceeding
ssh ankit@YOUR_SERVER_IP
# Should log in without asking for a password
```
<Callout type="tip" title="Expected output">
#
Generating public/private ed25519 key pair.
Your identification has been saved in ~/.ssh/id_ed25519
#
</Callout>
<Callout type="danger" title="Critical: Test Key Login Before Disabling Passwords">
Do NOT disable password auth until you confirm key login works. Locking yourself out of a Hetzner VPS means using the browser-based KVM console — painful. Confirm key-based login works in a new terminal window first.
</Callout>

## Step 3 of 8 — Harden SSH Configuration
#
```bash
sudo nano /etc/ssh/sshd_config
```
<Callout type="tip" title="Expected output">
#

#
</Callout>
Change or add these lines:
#
```bash
PasswordAuthentication no
ChallengeResponseAuthentication no
PermitRootLogin no
Port 2222
AllowUsers ankit
MaxAuthTries 3
LoginGraceTime 20
X11Forwarding no
```
<Callout type="tip" title="Expected output">

#
</Callout>
Apply changes:
#
```bash
sudo systemctl restart sshd
```
<Callout type="tip" title="Expected output">
No error output means sshd restarted cleanly. Immediately open a new terminal and test your new port before closing this session.
</Callout>

Open a new terminal and test the new port before closing your current session:
#
```bash

```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
If it works, you can safely close the old session. All future SSH connections: `ssh -p 2222 ankit@YOUR_SERVER_IP`.
#
With SSH locked down, the firewall is next — deny everything by default and allow only what your apps need.
#
<Callout type="tip" title="Why Change the SSH Port?">
Changing from port 22 to any other port won't stop a determined attacker who scans all ports — but it eliminates the automated scanning noise. My auth.log went from 800+ failed attempts per day to under 20 after changing the port. Less noise means real threats are easier to spot.
</Callout>

## Step 4 of 8 — Configure UFW Firewall
#
UFW (Uncomplicated Firewall) is the standard Ubuntu firewall. Rule: deny everything by default, allow only what you need.
#
```bash
sudo apt install ufw -y
#
sudo ufw default deny incoming
sudo ufw default allow outgoing
#
# Allow your SSH port (replace 2222 with yours)
sudo ufw allow 2222/tcp
#
# Allow HTTP and HTTPS (for Nginx)
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
#
sudo ufw enable
sudo ufw status verbose
```
<Callout type="tip" title="Expected output">
`Status: active` with your allowed ports listed. If you see `inactive`, re-run `sudo ufw enable`.
</Callout>
#
<Callout type="warning" title="Docker Bypasses UFW">
Docker modifies iptables directly and can expose container ports even when UFW denies them. Always use `-p 127.0.0.1:PORT:PORT` (not `-p PORT:PORT`) in all Docker run commands to bind to localhost only. This ensures only [Nginx reverse proxy configuration](/blog/nginx-reverse-proxy-guide-2026) can reach your containers, not the public internet.
</Callout>

## Step 5 of 8 — Install and Configure fail2ban
#
fail2ban monitors log files and automatically bans IPs that show malicious patterns — like 5 failed SSH attempts in 10 minutes. It's why I see 50–200 IPs banned on a new server within the first 24 hours.
#
```bash
sudo apt install fail2ban -y
#
sudo cat > /etc/fail2ban/jail.local << 'EOF'
[DEFAULT]
bantime  = 1h
findtime  = 10m
maxretry = 5
banaction = ufw
#
[sshd]
enabled = true
port = 2222
logpath = /var/log/auth.log
maxretry = 3
#
[nginx-http-auth]
enabled = true
#
[nginx-limit-req]
enabled = true
EOF
#
sudo systemctl enable fail2ban
sudo systemctl start fail2ban
```
<Callout type="tip" title="Expected output">
`fail2ban.service` shows `active (running)`. If it fails, check `/var/log/fail2ban.log` for the error.
</Callout>
#
Check status after 24 hours:
#
```bash
sudo fail2ban-client status sshd
```
<Callout type="tip" title="Why this matters">
Fail2ban running on a new server typically bans 50–200 IPs within the first 24 hours — all automated scanners hitting the SSH port. Each ban is a brute-force attempt that didn't succeed.
</Callout>
#
With fail2ban active, enable automatic security updates so patches apply without manual intervention.

## Step 6 of 8 — Enable Automatic Security Updates
#
Security patches should apply automatically. You shouldn't need to manually run `apt upgrade` every time a CVE drops.
#
```bash
sudo apt install unattended-upgrades -y
sudo dpkg-reconfigure -plow unattended-upgrades
# Select "Yes" when prompted
#
# Verify
cat /etc/apt/apt.conf.d/20auto-upgrades
# Should show: APT::Periodic::Unattended-Upgrade "1";
```
<Callout type="tip" title="Expected output">
#
Reading package lists...
Setting up package-name...
Processing triggers...
#
</Callout>
This auto-applies security updates only — not regular package upgrades. Your application packages (Node.js, Python, Docker) aren't touched. The server only auto-reboots if a kernel update requires it.

## Nginx Security — Headers and Rate Limiting
#
Nginx hardening protects applications running behind the reverse proxy. Add these to your Nginx config:
#
```bash
# In /etc/nginx/nginx.conf, inside http { }
server_tokens off;
#
# Security headers in your server block
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy
  "strict-origin-when-cross-origin" always;
add_header Permissions-Policy \
  "camera=(), microphone=(), geolocation=()" always;
```
Add rate limiting to stop automated scanners on API endpoints:
#
```bash
# In nginx.conf http block - define the zone
limit_req_zone $binary_remote_addr \
  zone=api_limit:10m rate=10r/s;
#
# In your API location blocks
location /api/ {
    limit_req zone=api_limit burst=20 nodelay;
    proxy_pass http://localhost:3000;
}
```
Verify headers:
#
```bash
curl -I https://yourdomain.com | grep -E \
  "X-Frame|X-Content|Server"
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
The `Server` header should show nothing or just `nginx` without a version number.

## Troubleshooting Common Security Issues
#
**Locked out after changing SSH port**
#
Use Hetzner's browser-based KVM console (Control Panel → Server → Console). Log in as root, fix `/etc/ssh/sshd_config`, restart sshd, re-add the correct UFW rule.
#
**fail2ban not banning despite failed attempts**
#
Check the jail is looking at the right log file:
#
```bash
sudo fail2ban-client get sshd logpath
```
<Callout type="tip" title="Expected output">
#
fail2ban.service - Fail2Ban Service
   Active: active (running)
#
</Callout>
On some Ubuntu 24.04 setups, SSH logs go to `/var/log/auth.log` but fail2ban defaults to `journald`. Add `backend = auto` to `jail.local` to force file-based logging.
#
**UFW blocking Docker networking**
#
Docker and UFW have known conflicts. Install the `ufw-docker` package which handles the conflict automatically, or use `127.0.0.1` bindings in all Docker run commands.
#
**Automatic updates broke a package**
#
Check:
#
```bash
cat /var/log/unattended-upgrades/unattended-upgrades.log
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
Pin the package version in `/etc/apt/preferences.d/` to prevent future auto-updates on that specific package.

### Do I need all 8 steps or can I just do some?
#
At minimum: steps 2 (SSH keys), 3 (disable password auth), and 4 (UFW). These three eliminate 95% of the risk. fail2ban (step 5) is quick and adds meaningful protection. The rest is good practice but lower priority.
#
### Is a server firewall enough or do I also need Cloudflare?
#
Both serve different purposes. UFW/fail2ban protects at the server level. Cloudflare protects at the network level — absorbs DDoS traffic, hides your real IP, adds a WAF on paid plans. For indie projects, both together is ideal. Cloudflare free tier + UFW covers you against the threats you're actually likely to face.
#
### How do I know if my VPS has already been compromised?
#
Check `last` (recent logins), `who` (current sessions), `ps aux` (look for miners or unfamiliar processes), `crontab -l` (unexpected cron jobs), and `/tmp` directory (miners often drop files here). If anything looks suspicious, follow the [compromised VPS recovery guide](/blog/compromised-vps-recovery-2026) and restore from a Hetzner snapshot taken before the compromise.
#
### Should I change the SSH port?
#
It reduces log noise significantly but provides no real security benefit against a targeted attacker who scans all ports. I change it on every server because less noise makes real threats easier to spot.
#
### Is Hetzner's cloud firewall the same as UFW?
#
No. Hetzner provides a cloud-level firewall in the control panel that blocks traffic before it reaches your server. UFW is a firewall inside the server. Using both is best practice — Hetzner firewall as a first line, UFW as a second. The Hetzner firewall survives even if your server OS is misconfigured.
#
### How often should I update server packages?
#
Security packages: automatic via unattended-upgrades. Runtime packages (Node.js, Python, Docker): monthly. Full system: `sudo apt update && sudo apt upgrade` monthly during a low-traffic window. Always snapshot the server before a major update.

## Security Checklist — Run Monthly
#
After hardening, the work isn't done. Here's the monthly check I run on every Hetzner VPS to catch anything that's drifted:
#
```bash
# 1. Check for failed login attempts
grep "Failed password" /var/log/auth.log | tail -20
#
# 2. Check fail2ban is still active
sudo fail2ban-client status sshd
#
# 3. List all open ports
ss -tlnp
#
# 4. Check for unexpected processes
ps aux --sort=-%mem | head -20
#
# 5. Review UFW status
sudo ufw status verbose
#
# 6. Check SSL cert expiry
sudo certbot certificates
```
<Callout type="tip" title="Expected output for fail2ban check">
`sudo fail2ban-client status sshd` should show `Currently banned: N` with a non-zero number if any IPs have been blocked. If the command fails, fail2ban is not running — restart it with `sudo systemctl restart fail2ban`.
</Callout>
#
Run these once a month — takes under 5 minutes. If `ss -tlnp` shows a port you didn't open, that's the signal to investigate immediately. Unexpected open ports are the most common sign of compromise on shared VPS infrastructure.
