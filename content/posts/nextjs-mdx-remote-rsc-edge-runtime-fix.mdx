## Copy-paste fix for the next-mdx-remote/rsc Edge Runtime error

If you want the shortest path from error to recovery, start here.

```tsx filename="app/blog/[slug]/page.tsx"
// file-backed MDX routes should stay on Node.js runtime
export const runtime = "nodejs";

import { MDXRemote } from "next-mdx-remote/rsc";
import { getPostBySlug } from "@/lib/posts";

export default async function BlogPostPage({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = getPostBySlug(slug);

  return (
    <article>
      <MDXRemote source={post.content} />
    </article>
  );
}
```

That one line fixes the issue in a lot of real projects because it stops the route from inheriting Edge when the content path still needs Node.js APIs such as `fs` or `path`.

<Callout type="warning" title="Use this only for the right setup">
This fix is for file-backed MDX routes or any content path that still touches Node.js APIs. If your content is fully remote, precompiled, and Edge-safe, your architecture decision may be different.
</Callout>

<Callout type="info" title="Prerequisites">
You should already have a file-backed MDX route, access to the route file that renders the post, and a way to run `npm run build` locally or on your deployment target. This guide assumes App Router and a posts library that still touches `fs`, `path`, or frontmatter parsing on the server.
</Callout>

## Why the next-mdx-remote/rsc Edge Runtime error happens in Next.js 15

The important detail is not just `next-mdx-remote/rsc`. It is the full content path around it.

A typical blog route does more than render React:

- reads a file by slug
- parses frontmatter
- imports helpers from a local posts library
- passes the content into `MDXRemote`

That means the route is not only rendering JSX. It is also doing server-side work that usually depends on Node.js APIs such as `fs` and `path`.

That is where the failure starts. If the page or a parent layout pushes the route into Edge Runtime, the route now runs in an environment that does not support the same Node.js APIs your content path depends on.

So the practical diagnosis is usually simple:

`next-mdx-remote/rsc` is not the first thing to blame. The runtime choice is.

## Step 1: Force Node.js runtime on the route that renders the post

Put the runtime export directly on the route that renders the article. That is the smallest and safest fix because it removes guesswork at the page level.

```tsx filename="app/blog/[slug]/page.tsx"
// keep the MDX route on Node.js runtime
export const runtime = "nodejs";

import { MDXRemote } from "next-mdx-remote/rsc";
import { getPostBySlug } from "@/lib/posts";

export default async function BlogPostPage({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const post = getPostBySlug(slug);

  return (
    <article>
      <MDXRemote source={post.content} />
    </article>
  );
}
```

After this change, the article route renders under Node.js runtime even if you were testing other runtime behavior elsewhere in the app.
</Callout>

This is the safest fix because it removes ambiguity. Even if Node.js is the default runtime today, making it explicit protects the blog route when Edge was enabled higher in the tree.

## Step 2: Check parent layouts for inherited Edge Runtime

Before you change remark plugins, frontmatter parsing, or component mappings, inspect the route tree in this order:

1. `app/layout.tsx`
2. any segment layout above the blog route
3. `app/blog/[slug]/page.tsx`

Look for this line:

```ts filename="layout-or-page.tsx"
// this can push the route onto Edge
export const runtime = "edge";
```

That single export is often the whole problem.

The usual mistake is debugging the wrong layer. You strip down the MDX renderer, simplify content, or change helper functions, but the route still fails because it inherits Edge from a parent segment.

For most real projects, the stable split looks like this:

- keep middleware and tiny edge-safe request logic near Edge
- keep content routes that touch the filesystem on Node.js

That split is easier to understand, easier to test, and easier to ship. For more route-level fixes in the same cluster, keep browsing the [Web Development category](/category/web-dev) instead of generic tag archives.

## This fix will not work if your real problem is something else

This is the section most people need after they try the one-line fix and it still blows up.

The fix above will not solve the issue if:

- a parent layout still exports `runtime = "edge"` and the route is still inheriting it
- another import in the content path still pulls Node-only code into an Edge-only segment
- the real failure is an MDX compile or parse error and not a runtime mismatch
- your content is remote and Edge-safe, but a different helper is failing for unrelated reasons
- you are trying to keep the whole route on Edge without changing the architecture behind it

That is why the right debugging order matters. Fix the runtime mismatch first. Then separate it from content parsing, deployment errors, and route-level bugs.

## Node.js vs Edge for MDX routes: the practical rule

This is the part many articles avoid saying clearly.

If the route touches the filesystem, use Node.js.

That is the real rule.

Do not use Edge Runtime for MDX routes that:

- read content from disk
- depend on a filesystem-backed posts library
- import Node-only helpers
- parse files during render or server execution

For these routes, Node.js runtime is not a compromise. It is the correct environment.

## When `@next/mdx` may be the better fit

Tool choice matters here.

If you want first-party local MDX pages inside the App Router, `@next/mdx` is often the cleaner fit. It works well when your MDX files live directly in the app structure and behave more like route-level content.

If your MDX content behaves like data in a posts library, `next-mdx-remote/rsc` can still make sense. That is especially true when you already have slug lookup, frontmatter parsing, related post logic, and a content loader built around files or remote content.

The mistake is not using `next-mdx-remote/rsc`.

The mistake is using it on a route that still depends on Node.js APIs and then pushing that route into Edge.

## What to do if you really want Edge everywhere

Then you need a different architecture, not a one-line fix.

That usually means:

- moving away from filesystem reads during rendering
- fetching already-prepared content from a remote source
- removing Node-only dependencies from the content path
- separating edge-safe routes from Node-dependent content routes

That is a real architecture change. For most blogs, it is not worth the added complexity.

A boring Node.js content route is easier to maintain, easier to deploy, and easier to debug when something breaks.

## Troubleshooting checklist for stubborn runtime failures

If `runtime = "nodejs"` is already in place and the route still fails, work through this list in order:

1. confirm there is no parent `layout.tsx` or segment file still exporting `runtime = "edge"`
2. confirm the failing page is the same route you actually deployed
3. confirm your content loader still uses `fs`, `path`, `gray-matter`, or other Node-dependent code
4. confirm the production build completes without hidden MDX parse errors
5. confirm the live URL returns `200` after deploy

A lot of wasted time comes from mixing two different problems. One is a runtime mismatch. The other is an MDX parse issue. Fix the runtime first, then validate the article content separately.

## Common mistakes that waste time

### Fixing the wrong file

If the route inherits Edge from a parent layout, changing only a helper or sibling route will not fix anything. Start at the route tree, not your assumptions. Check `app/layout.tsx`, then each segment layout above the page, then the page itself. That order catches the real source faster than random edits inside the MDX renderer.

### Trusting `next dev` too much

Development mode is useful, but it is not the final truth for runtime behavior. A route can look healthy locally and still fail the moment you run a production build or hit the live server. Treat `next dev` as a fast feedback loop, not final validation.

### Assuming Edge is always faster

Not for this use case. A route that crashes in production is not a performance win. For a file-backed blog, the stable Node.js route is usually the better performance trade because it keeps the content path simple and predictable instead of adding cross-runtime complexity you do not need.

### Rewriting your MDX stack too early

Do the runtime fix first. Only rethink the content system if your actual architecture goals require it. Swapping libraries while the runtime is still wrong turns one bug into three. Stabilize the route, verify production, then decide whether the architecture still deserves a bigger change.

## Final takeaway

For a file-backed Next.js 15 blog, I would not fight this.

Keep `next-mdx-remote/rsc` routes on Node.js runtime when they touch disk or rely on Node APIs. Check parent layout inheritance, make the runtime explicit at the page level, rebuild in production mode, and verify the live URL.

That is the fix most teams actually need.

The only real exception is when your entire content path is already remote, precompiled, and Edge-safe from top to bottom.

## FAQ

### Does `next-mdx-remote/rsc` support Edge Runtime?

Not reliably for the common file-based blog setup. If the route depends on `fs`, `path`, or a filesystem-backed posts library, keep it on Node.js.

### Why does it work in development but fail after deploy?

Because development mode does not always expose the same runtime constraints that appear during a production build or live deployment.

### Where should I add `export const runtime = "nodejs"`?

Start with `app/blog/[slug]/page.tsx`. Then inspect parent layouts if the route still behaves like Edge is being inherited.

### Should I remove Edge Runtime from the whole app?

No. Only the routes that need Node.js APIs must avoid Edge. Other edge-safe routes can stay separate.

### Is this a Next.js bug or a `next-mdx-remote` bug?

In most real cases, it is a runtime mismatch first. Verify the environment before blaming MDX.

### Should I switch to `@next/mdx` instead?

Only if your content architecture fits route-level local MDX better. If MDX acts like content data in your posts system, `next-mdx-remote/rsc` can still be the better fit.
