## Why next-mdx-remote Breaks on Edge Runtime
#
`next-mdx-remote/rsc` reads MDX files from the filesystem using Node.js built-in modules — specifically `fs` and `path`. Edge runtime (Vercel Edge, Cloudflare Workers) does not have access to the Node.js runtime. It runs a stripped-down V8 environment with no filesystem access.
#
When Next.js tries to render your `/blog/[slug]` page on the edge, it attempts to import `next-mdx-remote/rsc`, which internally calls `fs.readFileSync` — and crashes immediately.
#
The error surface varies by environment:
#
- **Vercel:** explicit error — `The edge runtime does not support Node.js 'fs' module`
- **Self-hosted Next.js:** blank page, or a cryptic build error during `next build`
- **Dev mode:** works fine locally — `next dev` always runs Node.js regardless of runtime config
#
This is why the bug is easy to miss. It only surfaces in production.
#
<Callout type="warning" title="Why it passes in dev">
`next dev` ignores runtime configuration and always runs Node.js. The edge runtime restriction only applies during `next build` and production deploys. Always test with `next build && next start` before shipping.
</Callout>

## Step 1 of 3 — Find Where Edge Runtime Is Set
#
Check these 3 files in order — any one of them can cause the issue:
#
**Check 1 — root layout**
#
```tsx
// app/layout.tsx
export const runtime = 'edge' // ← this breaks all pages that use fs
```
**Check 2 — blog slug page**
#
```tsx
// app/blog/[slug]/page.tsx
export const runtime = 'edge' // ← directly breaks MDX rendering
```
**Check 3 — blog segment layout**
#
```tsx
// app/blog/layout.tsx
export const runtime = 'edge' // ← inherited by all blog routes
```
If you find `runtime = 'edge'` in any file that covers your blog post pages, that is the cause.
#
## Step 2 of 3 — Apply the Fix
#
**Fix A — Override to Node.js on the blog page (recommended)**
#
Add one line to your blog post page:
#
```tsx
// app/blog/[slug]/page.tsx
export const runtime = 'nodejs'  // ← add this
// blank line removed
export default async function BlogPost({ params }) {
  // your existing MDX rendering code
}
```
This explicitly sets Node.js runtime for this route, overriding any parent layout that sets edge.
#
<Callout type="tip" title="Expected output">
After adding this line and running `next build`, the build completes without the fs error. Verify with `curl -s -o /dev/null -w "%{http_code}" https://yourdomain.com/blog/your-post` — you should get `200`.
</Callout>
#
**Fix B — Remove edge runtime entirely**
#
If you set `runtime = 'edge'` for performance but don't actually use edge-specific features (geolocation, A/B at CDN layer), remove it. Node.js runtime with ISR and Cloudflare CDN delivers identical performance.
#
I run blixamo.com on Node.js runtime with Cloudflare CDN in front — hosted on a [Hetzner CPX22](https://www.vultr.com/?ref=9885468) class VPS. Pages load in 306ms globally. Edge runtime adds zero benefit for static blog content.
#
```tsx
// app/layout.tsx — remove this line
// export const runtime = 'edge'
```
**Fix C — Switch to @next/mdx for true edge compatibility**
#
If you genuinely need edge runtime everywhere, switch to `@next/mdx` which compiles MDX at build time — no filesystem reads at runtime.
#
```bash
npm install @next/mdx @mdx-js/loader @mdx-js/react
```
Trade-off: more complex config, less flexible for dynamic MDX content. For most blogs, Fix A is the right call.

## When to Actually Use Edge Runtime
#
Edge runtime is genuinely useful for:
#
- **Middleware** — auth checks, redirects, geolocation routing at the CDN layer
- **Lightweight API routes** that need sub-10ms cold starts with no Node.js dependencies
- **A/B testing logic** running at the CDN edge
#
It is the wrong choice for MDX blog pages. A blog post is static or ISR content. Rendered once, cached by Cloudflare, served instantly worldwide — no edge runtime needed.
#
<Callout type="warning" title="Never set runtime = edge globally">
Setting `export const runtime = 'edge'` in `app/layout.tsx` applies to every route. This breaks any route using `next-mdx-remote`, `fs`, `path`, `crypto`, or any Node.js-only library. Set edge runtime only on the specific routes that need it.
</Callout>

## Troubleshooting — Other Edge Runtime MDX Errors
#
### Module not found: Can't resolve 'fs'
#
Same root cause — a different import in your MDX pipeline calls `fs`. Check your plugins: `rehype-pretty-code`, `rehype-slug`, and custom remark plugins are common culprits.
#
### Build succeeds but page returns 500 in production
#
Edge runtime can fail silently during build. Check platform logs — Vercel Functions tab or PM2 logs on self-hosted. The actual error is always a missing Node.js module.
#
### Works on Vercel, breaks on self-hosted Next.js
#
Check `next.config.js` for experimental edge config:
#
```js
// next.config.js — remove if present
experimental: &#123;
  runtime: 'edge', // ← remove this
&#125;
```
### Works in dev, blank page in production
#
Classic edge runtime issue. Run `next build && next start` locally to reproduce before deploying.

## FAQ
#
### Does next-mdx-remote support edge runtime?
#
No. `next-mdx-remote` and `next-mdx-remote/rsc` both require Node.js runtime because they read and process MDX files using `fs` and other Node.js built-ins. As of 2026, there is no edge-compatible version.
#
### Will switching to Node.js runtime hurt performance?
#
No measurable impact for a blog. Node.js runtime with ISR and Cloudflare CDN delivers 200-400ms load times globally — identical to edge-rendered content. Cloudflare caches your HTML at the edge regardless of which runtime rendered it.
#
### Can I use edge runtime on some pages and Node.js on others?
#
Yes. Next.js supports per-route runtime configuration. Add `export const runtime = 'nodejs'` only on blog routes. Middleware and lightweight API routes can still use `runtime = 'edge'`.
#
### What is the difference between next-mdx-remote/rsc and next-mdx-remote?
#
`next-mdx-remote/rsc` is the React Server Component version for App Router. `next-mdx-remote` is the older Pages Router version. Both require Node.js runtime — neither works on edge.
#
### Does @next/mdx have the same edge runtime problem?
#
No. `@next/mdx` processes MDX at build time, not at runtime — so it works on edge runtime. Trade-off: requires more build configuration and is less flexible for dynamic MDX loading.
#
### Why does it work in development but break in production?
#
`next dev` always runs Node.js regardless of your runtime config. Edge runtime restrictions only apply during `next build` and production. Always test production builds locally before deploying.
#
### How do I confirm what runtime a page uses?
#
Check the `next build` output table. Edge runtime routes show `(Edge)`. After the fix, your blog routes should show `(SSG)` or `(Dynamic)`.
#
```bash
next build 2>&1 | grep -E "\(Edge\)|\(SSG\)|\(Dynamic\)" | grep blog
```
