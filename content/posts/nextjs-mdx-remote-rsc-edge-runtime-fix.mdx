This bug is annoying because the failure looks bigger than it is. Your Next.js app works locally, the MDX code looks fine, and then production throws `The edge runtime does not support Node.js 'fs' module` or just gives you a blank page with almost no signal.

The root cause is simple: `next-mdx-remote/rsc` needs Node.js runtime. If your blog route inherits edge runtime anywhere in the route tree, it breaks. The fix is usually one line.

<Callout type="info" title="TL;DR">
If `next-mdx-remote/rsc` is rendering content from disk, set `export const runtime = "nodejs"` on the affected route. Do not run that MDX route under edge runtime.
</Callout>

## Why next-mdx-remote/rsc Breaks on Edge Runtime

`next-mdx-remote/rsc` is not just a rendering helper. In the common blog setup, it works with content that is read from the filesystem, parsed, and then rendered inside a server component.

That means your route usually depends on Node.js APIs like:

- `fs`
- `path`
- file-based content loading in your posts library

Edge runtime does not support that environment. So if your route or a parent layout sets `runtime = "edge"`, the MDX page is running in the wrong place.

If you are using the content-on-disk architecture from [Build a Next.js MDX blog 2026](/blog/nextjs-mdx-blog-2026), this is exactly the kind of route that should stay on Node.js runtime.

## What the Error Usually Looks Like

The most obvious version is:

```text
Error: The edge runtime does not support Node.js 'fs' module.
```

But you can also see:

- build-time errors that mention unsupported modules
- blank production pages
- a route that works in dev but fails after deploy

That mismatch between local success and production failure is what wastes time. The app feels correct until the runtime environment changes.

## Step 1 of 3 — Find Where Edge Runtime Is Set

Check the route tree in this order:

1. `app/layout.tsx`
2. any segment layout above the blog route
3. `app/blog/[slug]/page.tsx`

You are looking for:

```tsx
export const runtime = "edge";
```

If that line exists anywhere above the MDX page, the blog route may inherit it.

That is the first thing I would inspect before touching the MDX code itself.

## Step 2 of 3 — Switch the Blog Route to Node.js Runtime

The normal fix is to set Node.js runtime directly on the blog page:

```tsx
export const runtime = "nodejs";
```

That line belongs on the route that renders the MDX content, typically:

```tsx
// app/blog/[slug]/page.tsx
export const runtime = "nodejs";
```

This is the safest fix because it is explicit. Even if a parent layout is using edge runtime for something else, the blog page opts back into the environment it actually needs.

## Step 3 of 3 — Rebuild and Verify the Runtime

After changing the runtime, rebuild the app and verify the affected page directly.

Use a production-style build check, not only local dev:

```bash
npm run build
```

Then verify the page loads:

```bash
curl -s -o /dev/null -w "%{http_code}" https://yourdomain.com/blog/your-post
```

If the page returns `200` and the build no longer reports the `fs` module issue, the runtime mismatch is fixed.

If you are self-hosting, pair this with the deployment path in [How to self-host Next.js on a VPS](/blog/how-to-self-host-nextjs-on-vps) so you are testing the same kind of environment you actually ship.

## When You Should Not Use Edge Runtime for MDX Routes

Do not use edge runtime for MDX routes that:

- read content from disk
- depend on a filesystem-backed posts library
- import Node-only libraries
- rely on build-time or server-side file parsing

This is one of those cases where edge runtime sounds faster than it is useful. For content routes, the simpler and more reliable choice is usually Node.js runtime plus caching.

That is also why [Next.js performance optimization 2026](/blog/nextjs-performance-optimization-2026) focuses more on ISR and caching than on forcing everything to the edge.

## What If You Really Want Edge Runtime?

Then you need to change the architecture, not just the config.

If you truly need edge runtime everywhere, move away from filesystem-driven MDX loading for those routes. That usually means:

- compile MDX at build time with a different setup
- avoid Node-only APIs in the rendering path
- separate edge routes from content routes

For most developer blogs, that tradeoff is not worth it. The clean answer is to keep the MDX route on Node.js.

## Common Mistakes Around This Fix

### Fixing the wrong file

If the route inherits edge runtime from a parent layout, changing only a sibling route will not help. Follow the route tree carefully.

### Trusting `next dev`

Local dev often hides runtime problems because it does not behave the same way as a production deployment. Always rebuild.

### Assuming edge runtime is a universal performance upgrade

It is not. For MDX content routes, edge runtime can add complexity without solving a real bottleneck.

## Summary

If `next-mdx-remote/rsc` is breaking with an edge runtime error, the problem is usually not MDX itself. The route is simply running under the wrong runtime.

The fix is:

- find where edge runtime is set
- set the MDX page to `runtime = "nodejs"`
- rebuild and verify

That is usually enough to get the route working again.

## FAQ

### Does next-mdx-remote/rsc support edge runtime?

Not in the common filesystem-based blog setup. If the route depends on Node.js APIs like `fs`, it needs Node.js runtime.

### Why does this work in development but fail in production?

Because local development often does not surface the same runtime constraints as the production build or hosting environment. The route can look fine until it is actually deployed under edge runtime.

### Where should I add `export const runtime = "nodejs"`?

Add it to the page or route segment that renders the MDX content, most often `app/blog/[slug]/page.tsx`. That makes the runtime explicit at the route level.

### Should I remove edge runtime from the whole app?

Not necessarily. Only the routes that need Node.js APIs must avoid edge runtime. It is fine to keep other routes or middleware on edge if they benefit from it.

### Is this a Next.js problem or an MDX problem?

It is mostly a runtime mismatch problem. Next.js is doing what you asked by running the route on edge, but the MDX rendering path depends on Node.js features that are unavailable there.

### What is the safest long-term fix?

For most blogs, keep content-rendering routes on Node.js runtime and use caching or ISR for performance. That is simpler than redesigning the whole MDX architecture around edge compatibility.
