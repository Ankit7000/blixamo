# Next.js 15 to 16 Migration Guide

A Next.js major upgrade is not just a package bump. It drags routing, React behavior, build tooling, and deployment assumptions along with it, which is why rushed upgrades create week-two bugs.

This guide focuses on the checks that matter most when you are moving a real app from Next.js 15 to 16.

> **TL;DR**
>
> - Upgrade in a branch, lock the current deploy target, and define rollback before touching packages.
> - Check the Next.js 16 removals first: async request APIs, `middleware` to `proxy`, `next lint`, Turbopack defaults, and image config changes.
> - Call the migration done only after `lint`, `build`, critical routes, images, and deploy logs all behave the same in production mode.

## Quick Plan

| Step | What matters |
|---|---|
| Preflight | Verify Node.js `20.9+`, TypeScript `5.1+`, and a clean rollback target. |
| Upgrade step | Run the codemod, then fix the breaking changes it cannot finish for you. |
| Validation | Lint, build, route checks, image checks, and one production-mode smoke pass. |
| Biggest mistake | Treating a green install as proof that the app is ready to ship. |

## What You Need Before You Start

- Node.js `20.9.0` or newer and TypeScript `5.1` or newer.
- A clean branch, a saved lockfile, and the current production commit hash.
- Access to CI, environment variables, and whatever VPS or platform path you use to ship the app.
- A short note that says how to restore the previous package versions and redeploy fast.

## Why Next.js 15 To 16 Migration Guide Matters

The risky part of a major Next.js upgrade is not running the codemod. It is the gap between "the packages installed" and "the app still behaves the same way under a production build." Routing assumptions, runtime boundaries, lint rules, and deployment defaults are where the real surprises usually live.

## Build the Upgrade Around Real Checks

Start by auditing the parts of the app most likely to move under a major upgrade: route handlers, `middleware.ts`, build scripts, lint scripts, image config, MDX tooling, and anything that depends on request-time APIs such as `cookies`, `headers`, `params`, or `searchParams`.

Define success before you upgrade. For most teams that means the app builds cleanly, the main routes render in production mode, and the release path stays boring after the version bump. If those checks fail, stop there and fix them before layering on unrelated cleanup.

A practical starter sequence looks like this:

```bash
npm install next@latest react@latest react-dom@latest eslint-config-next@latest
npx @next/codemod@canary upgrade latest
npm run lint
npm run build
```

That order should stay boring. A major upgrade is much safer when package changes, codemods, linting, and build verification happen in a predictable sequence.

## Breaking Changes Worth Checking First

### 1. Async request APIs must be async now

Next.js 15 allowed temporary compatibility in some places. Next.js 16 removes that fallback. Audit any usage of:
- `cookies()`
- `headers()`
- `draftMode()`
- `params`
- `searchParams`

If those are still read synchronously in pages, layouts, route handlers, or metadata files, update them before you trust the build.

### 2. `middleware` becomes `proxy`

If the app still uses the deprecated `middleware` convention, move it to `proxy` as part of the upgrade. This is easy to miss in apps that use auth, rewrites, or edge checks.

### 3. Turbopack is now the default

You no longer need `--turbopack` flags in `dev` or `build` scripts. Remove them. Also check for custom webpack config, because `next build` now using Turbopack by default can expose config paths that previously stayed quiet.

### 4. `next lint` is gone

Next.js 16 removes the old `next lint` command. Use the ESLint CLI directly in your scripts and CI. If your upgrade only fixes app code but leaves `npm run lint` broken in CI, the migration is not finished.

### 5. Image defaults changed

Review `next.config` if your app uses `next/image` heavily. Next.js 16 changes defaults around local query-string images, `minimumCacheTTL`, `qualities`, redirect limits, and deprecates `images.domains` in favor of `remotePatterns`.

### 6. Remove dead config while you are here

The codemod can clean up some deprecated config, but still check for removed flags such as `experimental_ppr` and any old unstable APIs that now have stable names.

## Validate the App Like a Real Deploy

The safest validation pass is simple and practical:
1. run `npm run lint`
2. run `npm run build`
3. start the app in production mode
4. load the top routes
5. test one form or mutation
6. test one image-heavy page
7. test auth or middleware/proxy behavior
8. check server logs for warnings that did not exist before

That is also a good time to compare against related deployment work such as [moving off Vercel](/blog/migrate-nextjs-vercel-to-vps-2026), [performance follow-up](/blog/nextjs-performance-optimization-2026), or a content-heavy app like this [MDX setup](/blog/nextjs-mdx-blog-2026).

## Common Mistakes and How to Avoid Them

The first mistake is upgrading Next.js without locking Node and TypeScript versions first. A framework problem is much easier to debug when the runtime is already correct.

The second mistake is trusting the codemod too much. It saves time, but it does not understand every app-specific assumption.

The third mistake is not testing images, auth, or metadata routes. Those are common places where an upgrade looks fine until production traffic hits the odd path.

The fourth mistake is coupling the framework upgrade with unrelated refactors. Keep the diff boring. Move stylistic cleanup to a later PR.

## Rollback Plan

Keep rollback boring:
1. restore the previous branch or tag
2. restore the old lockfile
3. reinstall previous versions
4. redeploy the last known good build
5. confirm the same smoke paths recover

Do not pair a database migration or big caching change with the same release unless you want rollback to become ambiguous.

## FAQ

### What is the safest way to start nextjs 15 to 16 migration guide?

Start with a branch, a locked runtime, and a saved rollback target. Then run the official codemod and fix the high-risk changes before you touch anything optional.

### Should I move everything in one cutover or in stages?

Stages are safer. Upgrade packages, fix the breaking changes, validate production mode, and only then clean up adjacent config.

### How do I reduce risk before switching traffic or ownership?

Lock dependency versions, save a clean branch, and decide which failed checks force a rollback before you change anything.

### What data should I back up first?

Back up the lockfile, environment settings, deployment config, and any app-specific compatibility notes. If you have custom scripts around build or caching, snapshot those too.

### How long should I keep the old system around?

Keep the previous package set and rollback notes around until the upgraded app has survived at least one normal release cycle.

### What is the cleanest rollback plan?

Keep the previous lockfile, dependency versions, and deploy instructions ready, and define the exact trigger for reverting before you start. A clean rollback should restore the last known good build without inventing new steps under pressure.

### What should I search for first in the codebase?

Search for `middleware`, `cookies(`, `headers(`, `draftMode(`, `searchParams`, `params`, and custom image config. Those are the highest-value upgrade checks.
