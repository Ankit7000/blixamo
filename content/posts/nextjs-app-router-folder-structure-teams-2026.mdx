The App Router does not become messy because the framework is confusing. It becomes messy because teams let route files, feature logic, shared UI, and server-only code bleed into each other until code review turns into archaeology.

This page is for teams that need a structure people can guess without asking in chat every day. It is not about winning a folder-style argument. It is about keeping changes reviewable, ownership clearer, and new feature work from turning `app/` into a junk drawer.

The structure below is the pattern I would use before a team starts adding more product areas, tests, background jobs, and shared UI. It keeps route composition in `app/`, feature ownership in `features/`, and reusable pieces in clearly named shared folders.

> **TL;DR**
>
> - Keep route files in `app/`, feature logic close to the feature, and shared code in clearly named shared directories.
> - Do not put every component in one giant `components/` folder or every helper in one giant `lib/` bucket without boundaries.
> - Separate server-only code, shared UI, and domain logic so code review can follow intent quickly.
> - Optimize for predictable placement and low review friction, not for the prettiest folder tree screenshot.

## When this structure is worth standardizing

Use a documented team structure when:

- more than one person is regularly changing the codebase
- route files are starting to contain feature logic, validation, and data plumbing all at once
- the team is adding tests, shared UI, and background logic around the same routes
- code review is slowing down because nobody can guess where related code lives

If the app is still a tiny one-person prototype, do not overengineer it. The structure should grow with team pressure, not ahead of it.

## The boundary that matters most

The most useful rule is simple:

- `app/` owns routing, layouts, pages, route handlers, metadata, and route-level composition
- feature folders own feature-specific UI and logic
- shared folders own code used across multiple features

Once the team agrees on that boundary, most folder debates get easier.

## A practical team-friendly shape

This is the baseline structure I would rather inherit than a fashionable but unclear variant:

```text
app/
  (marketing)/
    page.tsx
    pricing/page.tsx
  (app)/
    dashboard/page.tsx
    settings/page.tsx
  api/
    billing/route.ts
  layout.tsx
  globals.css

features/
  billing/
    components/
    server/
    validation/
  dashboard/
    components/
    hooks/
    queries/
  settings/
    components/
    actions/

components/
  ui/
  layout/

lib/
  server/
  client/
  utils/

tests/
  e2e/
  integration/
```

This shape is boring in the best way. A reviewer can see where route composition lives, where domain logic belongs, and where shared UI stops being feature-specific.

## What should stay inside `app/`

Keep `app/` narrow.

Good things to keep there:

- `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, and route handlers
- route-level metadata and route groups
- the thin composition layer that wires a route to feature code

Bad things to let spread through `app/`:

- giant business-logic helpers
- unrelated reusable components copied between routes
- validation rules and domain logic that should live with the feature
- utility files whose names make sense only to the person who created them

The more logic you bury inside route directories, the harder it becomes to change one feature cleanly across multiple routes.

## What should live in `features/`

`features/` is where the actual product shape should become visible.

A feature folder is useful when it groups:

- UI used only by that feature
- server actions, queries, or mutations tied to that feature
- validation schemas and mapping helpers
- feature tests if the team prefers colocated testing

The question is not "Should every project have a `features/` directory?" The better question is "Do we have enough feature-specific logic that route-only organization is now hurting review speed?" For most team projects, the answer becomes yes surprisingly early.

## What should stay clearly shared

Shared folders are where teams usually create sprawl if names are too vague.

Use shared directories for code that is genuinely reused:

- `components/ui` for reusable primitives
- `components/layout` for shared shells and wrappers
- `lib/server` for server-only helpers and integrations
- `lib/client` for client-safe helpers
- `lib/utils` for small general helpers that are not secretly domain logic

The rule I care about: if a helper or component is effectively owned by one feature, do not promote it to shared too early. Premature shared code becomes another hiding place for unrelated logic.

## Route groups, server code, and tests

Three team decisions pay off quickly:

### Use route groups for product shape, not as a dumping ground

Route groups such as `(marketing)` and `(app)` are useful when they reflect real layout or ownership boundaries. They are not useful if they only add another nesting layer without improving clarity.

### Mark server-only code intentionally

If a helper touches secrets, database access, or server-only SDKs, keep it in a clearly named server-only area. That makes accidental client imports easier to spot in review.

### Put tests where the team can actually maintain them

If end-to-end and integration tests are shared, a top-level `tests/` directory is often easier. If feature-level tests move with feature ownership, colocated tests may be better. Pick one default and document it.

## The review rule I would document for the team

A good structure should let a reviewer answer these questions quickly:

- which route changed
- which feature owns the behavior
- which shared code was affected
- whether any server-only or client-only boundary was crossed

If a simple dashboard change touches six directories with unclear names, the structure is not helping enough.

## Common mistakes

### Treating `app/` like the whole application

That works for very small projects, then becomes painful as soon as features need their own logic and tests.

### Making one giant `components/` bucket

This hides ownership and makes "shared" mean "nobody knows where it belongs."

### Promoting feature code to `lib/` too early

`lib/` should not become a second junk drawer for domain logic.

### Reorganizing constantly without a placement rule

A team does not need the perfect structure. It needs a stable one with clear defaults.

## A low-risk migration path if the current structure is already messy

Do not try to reorganize the entire repo in one pass.

A safer sequence is:

1. agree on placement rules first
2. move one active feature into the new pattern
3. keep route files thin as new work lands
4. create shared folders only after two or three real reuse cases
5. let older areas migrate gradually when touched

That keeps the team from spending a sprint on architecture theater instead of shipping.

## FAQ

### Should every Next.js App Router project use `features/`?

No. Small solo projects can stay route-first for a while. Teams usually benefit from `features/` once route files start collecting real domain logic.

### Where should server actions live?

Put them where ownership stays obvious. Route-specific actions can stay near the route or inside the feature. Shared domain actions should live with the feature or server-only layer that owns them.

### Is one big `lib/` directory a problem?

It becomes a problem when `lib/` turns into a vague home for unrelated code. Split it into clearer server, client, and utility boundaries before that happens.

### How do I know the structure is working?

New teammates can guess where changes belong, code review touches fewer unrelated files, and the team stops debating placement on every PR.
