# Build a Telegram Bot with Claude API in Python (2026)

The fastest way to build a useful Claude-powered bot is not to start with a web app. It is to use Telegram as the interface, keep the first version private, and run one small Python process on a VPS you already control. That gives you a real assistant without frontend work, auth flows, or another hosted tool in the middle.

The setup I would ship first is simple: `aiogram`, one Claude API call, an `.env` file for secrets, an allowlist so strangers cannot burn your credits, and a `systemd` service so the bot comes back after a reboot. You can add persistence, summarization, or richer commands later. The first version only needs to reply reliably.

<Callout type="info" title="TL;DR">
Build the first version as a private bot, not a public one. Use aiogram, keep secrets in `.env`, allow only your own Telegram user ID at first, and run the bot with `systemd`. Once the bot is stable for a week, then decide whether it deserves persistence, extra commands, or public users.
</Callout>

## What you are building

This guide builds a bot that:

- replies to direct messages using Claude
- limits use to your Telegram account at first
- runs as a background service on Linux
- survives SSH disconnects and host reboots

That is enough for a real personal assistant, internal tool bot, or operator helper.

## Step 1: Create the bot and collect the 3 values you need

Before you touch Python, get:

- the Telegram bot token from `@BotFather`
- your Claude API key
- your Telegram numeric user ID

The user ID matters because it lets you keep the bot private while you test it. That is the safest first move. Otherwise one leaked bot username can turn into avoidable API spend.

Your project directory can be as small as this:

```bash
mkdir -p /home/ankit/telegram-bot
cd /home/ankit/telegram-bot
python3 -m venv .venv
source .venv/bin/activate
pip install aiogram anthropic python-dotenv
```

Then create `.env`:

```bash
BOT_TOKEN=replace-me
ANTHROPIC_API_KEY=replace-me
ALLOWED_USER_ID=123456789
CLAUDE_MODEL=claude-haiku-3-5-20251001
```

<Callout type="tip" title="Verify it">
Keep `ALLOWED_USER_ID` set to your own Telegram account while testing. That single guard is worth more than a fancy rate limiter on day one because it blocks accidental public use completely.
</Callout>

## Step 2: Build the smallest useful bot

You do not need a 100-line starter just to prove the workflow. Start with a minimal reply loop that is easy to debug.

```python
import os
from dotenv import load_dotenv
from anthropic import Anthropic
from aiogram import Bot, Dispatcher, F
from aiogram.filters import CommandStart
from aiogram.types import Message
load_dotenv()
bot = Bot(os.environ["BOT_TOKEN"])
dp = Dispatcher()
client = Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])
allowed_id = int(os.environ["ALLOWED_USER_ID"])
@dp.message(CommandStart())
async def start(message: Message):
    if message.from_user.id == allowed_id:
        await message.answer("Bot is live. Send a prompt.")
@dp.message(F.text)
async def reply(message: Message):
    if message.from_user.id != allowed_id:
        return
    result = client.messages.create(
        model=os.environ["CLAUDE_MODEL"],
        max_tokens=400,
        messages=[{"role": "user", "content": message.text}],
    )
    await message.answer(result.content[0].text)
bot.run_polling(dp)
```

This version does not try to do everything. That is good. It proves the 4 things that matter first:

- Telegram can reach the bot
- your Claude key works
- the process can answer one real message
- outside users are ignored

Once that works, then add commands, memory, or message history.

## Step 3: Test the bot before you daemonize it

Run it directly first:

```bash
source .venv/bin/activate
python bot.py
```

Then test these cases from Telegram:

- `/start` should return a short confirmation
- a normal text prompt should get a Claude reply
- a second account should get nothing if the allowlist is working

If the bot fails here, do not move to `systemd` yet. A background service only hides bugs that are still easy to inspect in the foreground.

Common first-failure cases:

- wrong `BOT_TOKEN`
- wrong `ALLOWED_USER_ID`
- missing Claude credits or wrong API key
- using a model ID your account cannot access

## Step 4: Run it with systemd

Once the bot responds correctly in the foreground, make it persistent.

```ini
[Unit]
Description=Telegram Claude Bot
After=network.target
[Service]
User=ankit
WorkingDirectory=/home/ankit/telegram-bot
ExecStart=/home/ankit/telegram-bot/.venv/bin/python bot.py
Restart=always
EnvironmentFile=/home/ankit/telegram-bot/.env
[Install]
WantedBy=multi-user.target
```

Save that as `/etc/systemd/system/telegram-bot.service`, then:

```bash
sudo systemctl daemon-reload
sudo systemctl enable telegram-bot
sudo systemctl start telegram-bot
sudo systemctl status telegram-bot
```

What you want:

- service is `active (running)`
- restarts on failure
- logs are readable with `journalctl -u telegram-bot -f`

That is enough for a production-worthy personal bot on one VPS.

## Step 5: Add the 3 protections that matter most

You can keep piling on features, but the first useful hardening pass is smaller than people think.

### 1. Keep the bot private at first

This is still the best protection. One allowlist check prevents accidental spend and weird public messages while you are still learning what the bot should do.

### 2. Cap what one request can cost

Do not let the bot ask for unbounded responses. Keep `max_tokens` modest until you know what the real use cases are.

### 3. Log failures clearly

If you cannot see failed calls, timeouts, or process crashes, you do not have a bot. You have a background surprise waiting for the wrong day.

## When to add memory, summaries, and extra commands

Add features only after the base loop is stable.

Here is the order I would use:

1. add a `/clear` command
2. add simple in-memory history per user
3. add one useful command such as summarization
4. move history to SQLite only if the bot proves it needs persistence

That order matters because persistent storage is real complexity. If the bot is only helping you or a small internal group, in-memory state may be enough for a while.

## Failure cases that show up in real use

### The bot responds once, then becomes unreliable

That is often a process-management issue, not a Claude issue. Check `systemd` restarts and logs before you start changing prompts.

### The bot gets expensive faster than expected

That usually means too many long replies or too many users. Keep the bot private, set spending limits in Anthropic, and cap reply size before you chase deeper optimizations.

### The bot needs more workflow context

That is when you decide whether to add message history, pull in external context, or move the use case into an automation system like n8n instead of keeping everything in one Python bot.

## Which approach should you choose?

Use a Telegram bot with Claude and Python if:

- you want the fastest path to a useful assistant
- you are comfortable editing a small Python service
- you already have a VPS or Linux box to run it on

Use an automation-first setup instead if:

- the bot is only one step in a larger workflow
- you want trigger-based behavior more than conversational behavior
- non-Python teammates need to maintain it

My recommendation is still Python plus Telegram for the first version. It is simpler than it sounds, and it keeps the whole system understandable.

The honest exception is when the "bot" is really just an interface to a wider automation pipeline. In that case, a workflow tool can be the cleaner center of gravity.

## FAQ

### What is the easiest way to build a Telegram bot with Claude API in Python?

Use aiogram, keep secrets in `.env`, allow only your own Telegram user ID at first, and run the process with `systemd` after it works in the foreground.

### Should I use polling or webhooks?

For a simple personal or small internal bot, polling is easier to ship and debug. Move to webhooks only when you have a reason to optimize around that architecture.

### How do I keep strangers from using the bot?

Check the sender's Telegram user ID and ignore anyone who is not on your allowlist. That is the safest first control.

### Do I need a database for the first version?

No. Start without one unless persistent history is already a requirement. A bot that replies reliably without storage is better than a bigger bot that is harder to keep running.

### How much does a Claude-powered Telegram bot cost?

That depends on the model, reply length, and message volume. For budget planning, read [Claude API vs OpenAI Cost in India (2026)](/blog/claude-api-vs-openai-cost-india).

### Should I build this as a web app instead?

Only if the product really needs a custom UI. If the main goal is a useful assistant quickly, Telegram is the simpler first interface.
