# PM2 to Docker Compose Migration Guide in 2026

A PM2 to Docker Compose migration changes more than process management. Ports, env vars, persistence, [health checks that actually catch bad deploys](/blog/docker-compose-health-checks-production-2026), and restart behavior all move to a new layer, which is where most surprises hide.

This guide walks the migration in a sequence that keeps rollback easy while you prove the containerized setup works.

> **TL;DR**
>
> - Inventory everything PM2 currently handles before you write the first Compose file.
> - Move env loading, ports, volumes, health checks, and proxy targets one layer at a time.
> - Keep PM2 available until the containerized app has passed real traffic, restarts, and one rollback drill.

## Quick Plan

| Step | What matters |
|---|---|
| Inventory | Ports, env vars, startup command, logs, proxy target, and writable paths. |
| Compose baseline | Build, command, env file, restart policy, and health check. |
| Cutover | Switch traffic only after the container is healthy and repeatable. |
| Biggest mistake | Killing PM2 too early and losing the fastest rollback path. |

## What You Need Before You Start

- The current PM2 ecosystem or startup command.
- The port mapping, reverse-proxy target, and any volume or storage path the app depends on.
- Access to the current `.env` values or secret store.
- A rollback note that says how to put PM2 back on the old port or old target quickly.

## Why PM2 To Docker Compose Migration Guide Matters

Moving from PM2 to Docker Compose changes the shape of the deployment. The app process is only one part of it. Ports, persistent storage, env files, image builds, health checks, and restart behavior all move into a different operating model, which is why the migration can feel bigger than expected.

## Build the Cutover Around the Runtime Shift

Start by documenting what PM2 is doing today that Compose will need to replace: env loading, restart behavior, ports, volumes, reverse-proxy targets, and any background process assumptions. That is where the hidden migration work usually lives.

Define success before you start. For most teams that means the containerized app boots cleanly, receives traffic, stays healthy across a restart, and leaves you with a rollback that is still quick. If one of those is fuzzy, the migration is not ready yet.

A practical starter sequence looks like this:

```yaml
services:
  web:
    build: .
    env_file: .env
    restart: unless-stopped
    ports:
      - "3000:3000"
    command: npm run start
```

That first Compose file should stay simple. The goal is to prove the runtime model, not to redesign the whole stack on day one.

## What Changes When You Leave PM2

PM2 is mainly about keeping a Node process alive. Docker Compose defines the process, the filesystem, the environment, the network, and often the supporting services too. That means a PM2 migration is never just `pm2 delete` plus `docker compose up`.

The biggest mindset change is ownership:
- PM2 owns process restarts
- Docker owns the container lifecycle
- Compose owns service wiring
- your reverse proxy still owns public traffic

## Move the App Definition First

Start with the narrowest viable Compose setup: build image, start command, env file, port, and restart policy. Do not add extra services until the core app boots cleanly.

Then copy over the behavior PM2 previously hid from you:
- working directory assumptions
- startup command
- environment variable source
- log location
- memory or restart expectations

If you run more than one service on the VPS, compare the migration plan against [single-VPS service layout](/blog/multiple-projects-single-vps) before you expose ports or volumes casually.

## Handle Env Vars, Volumes, and Health Checks Explicitly

This is where most PM2 to Compose migrations go sideways. PM2 apps often depend on local assumptions that are invisible until the container starts fresh.

Check these one by one:
1. env vars load from the intended file or secret source
2. writable directories are mounted where the app expects them
3. uploads or cache paths survive container recreation
4. health checks prove readiness, not just process startup
5. the reverse proxy points at the new container port, not the old PM2 port

## Why People Move from PM2 Even When PM2 Works

PM2 is fine for many workloads. Teams usually move when they want cleaner dependency packaging, more consistent deployments across machines, or a workflow that treats app runtime and supporting services as one defined unit. Docker Compose becomes attractive when the deployment story needs to include database, cache, queue, or sidecars in a repeatable way.

That means the guide should compare operating models, not pretend the old approach was foolish.

## Cut Over Traffic Only After a Real Smoke Pass

Before changing the proxy target or public port, run the container locally on the VPS and test:
- homepage or app root
- one authenticated route
- one write path if the app supports writes
- restart behavior
- health endpoint
- logs during startup and under light traffic

Then switch traffic and keep PM2 untouched for a while. If the container crashes, leaks env vars, or serves the wrong assets, rollback should mean pointing traffic back to PM2, not rebuilding the old setup from memory.

## Common Mistakes and How to Avoid Them

The first mistake is moving traffic before volumes and env vars are verified. The app may boot and still fail real work.

The second mistake is treating `restart: unless-stopped` as a health strategy. Restarts help, but only a real health check tells the proxy whether the app is ready.

The third mistake is deleting PM2 config too early. Keep the old startup path until the new one survives real usage.

The fourth mistake is forgetting the reverse proxy. A lot of migration bugs are really Nginx or Caddy still pointing at the wrong place.

## Troubleshooting and Verification

If the migration feels unstable, verify in this order.

1. **Container config**: does the image, command, and env file match what PM2 used?
2. **State**: are volumes mounted to the correct write paths?
3. **Network**: does the reverse proxy point at the correct container port?
4. **Health**: does the check prove readiness instead of just process existence?
5. **Rollback**: can you point traffic back to PM2 immediately if needed?

If those checks are clear, the migration is usually safe enough to continue.

## FAQ

### What is the safest way to start pm2 to docker compose migration guide?

Start with one service, prove the container boots and stays healthy, then switch traffic only after the proxy and volumes are confirmed.

### Should I move everything in one cutover or in stages?

Move in stages. Service definition, environment variables, volumes, networking, and traffic cutover should each have their own checkpoint.

### How do I reduce risk before switching traffic or ownership?

Freeze unrelated changes, define the exact health checks that must pass, and decide which log or metric failure forces a rollback. That keeps a tooling migration from turning into a vague outage hunt.

### What data should I back up first?

Back up the PM2 ecosystem file, environment settings, Docker-related config, persistent volumes, and any reverse-proxy rules that point traffic at the app. Those are the pieces you need to restore quickly.

### How long should I keep the old system around?

Keep PM2 available until the Docker path has survived normal traffic, restarts, and at least one routine deploy. Removing the old path too early is how simple migrations become long recoveries.

### What is the cleanest rollback plan?

Keep the last known good PM2 config, the startup commands, and the old traffic target ready, and define the exact conditions that trigger the switch back. A rollback plan should restore service fast, not send you back into experimentation.

### What should I compare between PM2 and Compose before cutover?

Compare ports, env vars, logs, writable paths, restart behavior, health checks, and the reverse-proxy target. Those are the real migration surface.
