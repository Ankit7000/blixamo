The instinct when a VPS feels slow or unstable is to upgrade it. More RAM, more CPU, bigger disk. The upgrade takes five minutes and solves the immediate pain.

The problem is that half the time the VPS was not the bottleneck. A misconfigured app, a log file that was never rotated, or a Docker volume growing unchecked would have been a fifteen-minute fix. Instead you are paying 40% more per month for a server that now has the same underlying problem on slightly more headroom.

Before paying for a bigger server, I spend 30 minutes on this checklist. Here is exactly what I look at.

**TL;DR:** Check what signal is actually elevated. Memory issues and CPU issues look different. Disk issues are almost always fixable without an upgrade. App setup problems masquerade as resource constraints constantly. Only upgrade when the resource is genuinely maxed at normal load with no optimization options left.

## What Signals Tell Me the VPS Is Really the Bottleneck

The key word is "really." A VPS that spikes to 90% CPU during a deploy is not resource-constrained — it is doing a deploy. A VPS that sits at 85% CPU during normal traffic is a different situation.

I look for sustained elevated resource usage at baseline load, not peak load. If the resource is elevated while the server is doing nothing unusual, that is a real constraint. If it spikes during known heavy operations and returns to normal, that is expected behavior.

```bash
# Baseline load snapshot (run this during normal operation, not during a deploy)
uptime                     # load averages
free -h                    # memory
df -h /                    # disk
ps aux --sort=-%cpu | head -8   # top CPU consumers
ps aux --sort=-%mem | head -8   # top memory consumers
```

Run this when nothing unusual is happening. That snapshot is your baseline. Every diagnosis starts here.

## When RAM Is the Real Problem

RAM is the most commonly misdiagnosed resource on a small VPS. Linux uses free RAM for caching, so `free -h` showing low available memory is often completely normal.

The signals that RAM is actually the bottleneck:

**1. OOM kills happening**
```bash
sudo dmesg | grep -i "out of memory\|oom_kill" | tail -10
sudo grep -i "out of memory\|killed process" /var/log/syslog | tail -10
```
If you see recent OOM kills for your app process, RAM is a real constraint. If you see OOM kills for a cron job or a background script, fix the script first.

**2. Swap usage is significant and sustained**
```bash
free -h
# Look at the Swap line — if "used" is consistently above 200-300MB, that is a signal
```
Swap usage is not a crisis by itself, but sustained swap usage on a 4GB VPS means your working set is larger than available RAM. The question is whether that is fixable by reducing the working set or whether you genuinely need more RAM.

**3. App process RSS is growing over days without returning to baseline**
```bash
# Check RSS in MB for a specific process
ps -p $(pgrep -f "pm2.*blixamo") -o pid,rss,vsz,comm
# Run this same command over a few days and compare
```
Growing RSS over multiple days without a ceiling is a memory leak, not a RAM shortage. A leak will eventually exhaust any server. Fix the leak before upgrading.

**When to upgrade for RAM:** OOM kills happening on your main app process at normal traffic levels, with no memory leak present, and no obvious process that can be removed or optimized.

**When not to upgrade for RAM:** Swap is in use but app is stable. RSS is growing — that is a leak to fix. A background process you can remove (like a monitoring agent or a development tool) is eating significant RAM.

## When CPU Is the Problem

CPU problems on small VPS instances almost always come from one of three things: a runaway process, a poorly written cron job, or genuine traffic growth.

```bash
# Watch CPU usage in real time
top
# or
htop

# Find what ran at high CPU recently
sudo grep "CPU" /var/log/syslog | tail -20
```

**A runaway process** will appear as one process consistently at 90-100% CPU. Identify it with `top` sorted by CPU. Restart it or fix the underlying cause before drawing any capacity conclusions.

**A poorly written cron job** causes regular spikes. If the load spikes happen on a schedule (check `uptime` output over time), look at your crontab and the cron job logs.

```bash
crontab -l
sudo grep CRON /var/log/syslog | tail -20
```

**Genuine traffic growth** shows as a broad elevation of CPU during traffic hours that correlates with increased requests in your access logs. This is the rare case where CPU upgrade actually helps. But even here, check whether caching can absorb the load before paying for more CPU.

**When to upgrade for CPU:** Sustained load average above vCPU count during normal traffic hours, after eliminating runaway processes and inefficient cron jobs.

**When not to upgrade for CPU:** Spikes during deploys. Spikes during one specific cron job. One process at 100% CPU — that is a fix, not an upgrade.

## When Disk or I/O Is the Problem

Disk problems have two forms: running out of space, and I/O saturation.

**Running out of space** is almost never a VPS capacity problem. It is a cleanup problem.

```bash
# Where is space going?
df -h /
du -sh /* 2>/dev/null | sort -rh | head -10
du -sh /var/log/* | sort -rh | head -5
du -sh /var/lib/docker/* 2>/dev/null | sort -rh | head -5

# Docker-specific cleanup
docker system df
docker system prune  # removes stopped containers, dangling images, unused networks
```

In my experience, Docker image layers and unrotated logs account for 80% of unexpected disk usage on a small VPS. Clean those before concluding the disk is too small.

**I/O saturation** is different. This shows as processes waiting on disk reads/writes, which makes everything feel slow even when CPU and RAM look fine.

```bash
# Check I/O wait
iostat -x 1 5
# Look at the %iowait column — above 20% sustained is a problem
# Requires: sudo apt install sysstat
```

High I/O wait can mean the disk is genuinely slow (common on shared VPS hosts with oversold storage), or it can mean a process is doing excessive writes — logs, database writes, or a misconfigured app writing to disk too aggressively.

If I/O wait is the problem and cleanup/optimization does not resolve it, upgrading to a VPS with NVMe storage (not just more storage) is the right move, not more RAM or CPU.

## When Bad App or Process Setup Is the Problem

This is the most common situation. The server has enough resources, but something is using them inefficiently.

**Signs this is the real problem:**

- One specific process uses far more CPU or RAM than everything else combined
- Resource usage jumped after a deploy or a config change
- The VPS was fine last month with the same traffic

**Common causes I have seen:**

**Node.js memory leak.** The process grows in RSS over days. Restart it temporarily to confirm — if RSS drops to normal after restart, it is a leak. Fix the app, do not upgrade the server.

**n8n storing too much execution history.** n8n by default stores every workflow execution. This can grow to gigabytes. Set execution retention limits in n8n settings.

**Redis with no maxmemory set.** Redis will use all available RAM if you do not set a limit. Check: `redis-cli CONFIG GET maxmemory`. If it returns 0, set a limit appropriate for your server.

**PM2 log files never rotated.** PM2 logs to files that never rotate by default. `pm2-logrotate` is a module that fixes this. Check your log directory size: `du -sh ~/.pm2/logs/`.

**Docker containers running that you forgot about.** `docker ps` will show you. Old containers from experiments can consume significant resources.

These are all fixes, not upgrades.

## When to Scale Up vs Optimize vs Split Workloads

**Scale up (upgrade the VPS)** when:
- RAM: OOM kills on your main app at normal load, no memory leak, no removable processes
- CPU: sustained load above vCPU count at normal traffic, no runaway process
- Disk: genuinely running a data-heavy workload that has outgrown the disk tier, cleanup done
- I/O: confirmed I/O saturation with NVMe upgrade as the right solution

**Optimize first** when:
- Any resource spiked after a code change or config change
- One process is responsible for the majority of the resource use
- Resources were fine recently with the same workload
- You have not run cleanup on Docker or logs recently

**Split workloads** when:
- You are running multiple apps and one of them is responsible for the resource pressure
- Moving one app to its own VPS would solve the problem and cost less than upgrading the shared server
- You need resource isolation between projects for operational reasons

The [VPS setup guide](/blog/vps-setup-guide) and the [running multiple projects on a single VPS](/blog/multiple-projects-single-vps) article cover the setup patterns for both single-app and multi-app configurations.

## What I Check Before Paying for a Bigger Server

This is my actual checklist before I open the Hetzner console to resize:

```bash
# 1. Establish baseline — is this normal load?
uptime && free -h && df -h /

# 2. OOM kills? (memory)
sudo dmesg | grep -i oom | tail -5

# 3. Swap usage sustained? (memory)
free -h | grep Swap

# 4. Runaway process? (CPU)
ps aux --sort=-%cpu | head -5

# 5. Cron job culprit? (CPU)
sudo grep CRON /var/log/syslog | tail -10

# 6. Disk: what is actually taking space?
du -sh /var/log/* /var/lib/docker 2>/dev/null | sort -rh | head -10

# 7. I/O wait significant?
iostat -x 1 3 2>/dev/null | awk '/Device/{found=1} found'

# 8. Any process with suspicious growth?
ps aux --sort=-%mem | head -8
```

If I run through all eight checks and find no fixable root cause, and the resource in question is at sustained elevated levels during normal operation, then the VPS is genuinely too small. That is when I upgrade.

In my experience, running this checklist catches a fixable problem about 70% of the time.

## Troubleshooting Capacity Diagnosis

**The VPS feels slow but all metrics look normal**
Check time-to-first-byte from an external request, not just from inside the VPS. Cloudflare or DNS issues can cause slowness that looks like a VPS problem. Also check nginx response times in access logs.

**Memory looks fine but app keeps crashing**
If RSS is not growing but the process crashes, look at application-level errors, not resource constraints. Check `pm2 logs appname --lines 100` for errors before the crash.

**Disk cleaned up but performance still bad**
If you cleaned disk and I/O wait is still high, the disk itself may be the bottleneck. Hetzner CPX instances use NVMe by default — if you are on a CX (shared block storage) instance, the storage tier may be the issue.

**Resource usage was normal, then spiked after a deploy**
Roll back the deploy and check the diff. This is almost always a code or configuration regression, not a capacity issue.

## FAQ

### How do I know if my VPS upgrade is actually needed vs just convenient?

Run the eight-point checklist above. If you find a fixable root cause, fix it first and wait a week. If resource pressure returns and no fix addresses it, the upgrade is justified.

### Is it better to upgrade one big VPS or split into multiple small ones?

Depends on the workload. For apps that share state (Redis, a database), staying on one server is simpler. For fully independent apps, splitting gives you resource isolation and clearer cost attribution. The [multiple projects on a VPS](/blog/multiple-projects-single-vps) article covers this tradeoff.

### What is the cheapest signal that a VPS is genuinely too small?

Repeated OOM kills on the main app process during normal traffic. Everything else has a fix. OOM kills on the actual app at baseline load is the clearest indication the RAM tier is wrong.

### Should I upgrade proactively before hitting limits?

Only if you have a specific growth projection and a known headroom threshold. Upgrading proactively without data is usually paying for capacity you will not use. Monitor disk and memory trends over a month — if both are trending toward limits, plan an upgrade. Otherwise, wait for a real signal.

### Can I downgrade after upgrading if the upgrade did not help?

On Hetzner, you can resize up easily but not down to a smaller disk size (only down to equivalent disk). Plan your upgrade carefully. If you are not sure whether an upgrade will help, fix the most obvious optimization first and measure the impact before resizing.
