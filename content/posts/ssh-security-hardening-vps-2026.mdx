## This page is about SSH access, not full-server hardening

SSH is how you enter the server, deploy code, inspect logs, restart services, and recover when something breaks. If an attacker gets SSH access, they usually get more than a shell. They get your app files, your environment variables, your deploy path, and often your secrets.

That is why SSH hardening is not a "nice to have" for bigger teams. It is baseline hygiene for solo developers too. A cheap VPS gets scanned the same way an expensive server does. Bots do not care whether your site makes `€0` or `€10,000` a month. They only care whether the server answers on a known port and accepts weak access patterns.

The good news is that basic SSH hardening is not complicated. It is mostly a small set of boring rules applied in the right order. The order matters because beginners often lock themselves out by disabling passwords before key login is proven. This guide stays on that narrow problem. UFW, Fail2Ban, unattended upgrades, and exposed service audits belong to the broader [Ubuntu VPS hardening guide](/blog/vps-security-harden-ubuntu-2026), not to this SSH-only pass.

## Why SSH Is the Main Attack Point for Servers

For most Ubuntu VPS servers, SSH is the only intentional remote admin entry point. That makes it the first thing scanners hit. Attackers try weak passwords, reused passwords, old usernames like `root` and `admin`, and default configurations that were never tightened.

SSH is also attractive because it is high leverage. A successful web exploit may still need more steps to become full server access. A successful SSH login is already server access. That is why password SSH, root login, and loose account policy are so dangerous.

There is another reason SSH matters: developers use it constantly. Anything you use constantly tends to drift into convenience. You skip key setup "for now." You leave root login on because it is fast. You keep password auth enabled in case a key fails. Each one feels small. Together they turn a safe admin channel into a weak one.

My rule is simple: if SSH is public, it should be key-only, least-privilege, and easy to audit.

## Common SSH Security Mistakes

Most SSH incidents on beginner VPS setups come from a small set of repeat mistakes.

| Mistake | Why it is dangerous | Safer fix |
| --- | --- | --- |
| Logging in as `root` | Full server access in one step | Use a deploy user with `sudo` |
| Password login enabled | Brute force still matters | Use SSH keys only |
| No second-session test | Easy way to lock yourself out | Test key login before reload |
| Too many allowed users | More accounts to protect | Restrict with `AllowUsers` |
| Unlimited auth attempts | Bots get more chances | Set `MaxAuthTries 3` |
| Firewall enabled late | SSH is exposed with no guardrail | Add UFW early |
| No brute-force protection | Repeated attacks continue all day | Install Fail2Ban |
| Reusing old keys forever | Old access remains valid | Rotate keys when needed |
| Public DB or Docker ports | Lateral risk after shell access | Keep internal services private |

If you remember only one thing from this section, remember this: convenience is how weak SSH setups survive into production.

## Step-by-step SSH hardening flow

This is the order I would use on a new Ubuntu VPS today. Do not skip the login test step.

### Step 1: Create a Deploy User

Do not use `root` for day-to-day work. Create a normal user and give it `sudo`.

```bash
# Create a deploy user and add sudo access
sudo adduser deploy
sudo usermod -aG sudo deploy
```

<Callout type="tip" title="Expected output">
Ubuntu will ask for the new user's password and profile details. No error means the user was created correctly.
</Callout>

### Step 2: Set Up SSH Keys

Create the `.ssh` directory, then add the public key. If you already have a local key, copy the `.pub` content into `authorized_keys`.

```bash
# Prepare the deploy user's SSH directory
sudo mkdir -p /home/deploy/.ssh
sudo chmod 700 /home/deploy/.ssh
sudo chown deploy:deploy /home/deploy/.ssh
```

```bash
# Add your public key to authorized_keys
sudo nano /home/deploy/.ssh/authorized_keys
```

Paste your public key, save, then run:

```bash
# Lock down authorized_keys permissions
sudo chmod 600 /home/deploy/.ssh/authorized_keys
sudo chown deploy:deploy /home/deploy/.ssh/authorized_keys
```

<Callout type="warning" title="Do this before changing sshd_config">
Do not disable password login yet. First prove that `ssh deploy@YOUR_SERVER_IP` works from a separate terminal.
</Callout>

### Step 3: Test SSH Login Before Continuing

Open a new terminal and test the deploy user with the key:

```bash
# Test key-based login in a second terminal
ssh deploy@YOUR_SERVER_IP
whoami
```

You want `whoami` to print `deploy`. Keep the original session open while you test. If this step fails, stop and fix key permissions before changing SSH settings.

### Step 4: Disable Root Login

Now edit the SSH daemon config:

```bash
# Edit the SSH daemon config
sudo nano /etc/ssh/sshd_config
```

Set:

```text
PermitRootLogin no
```

This removes the most obvious high-value login target from the public internet.

### Step 5: Disable Password Login

In the same file, set:

```text
PasswordAuthentication no
PubkeyAuthentication yes
```

Password login is what makes brute-force attempts relevant. Keys make those password attacks useless.

### Step 6: Allow Only the Deploy User

Still in `sshd_config`, restrict SSH to the one user you actually want:

```text
AllowUsers deploy
```

This is simple and effective. If you only need one admin user, let only that user in.

### Step 7: Limit Login Attempts

Add:

```text
MaxAuthTries 3
X11Forwarding no
AllowTcpForwarding yes
```

`MaxAuthTries 3` cuts down noisy repeated attempts. `X11Forwarding no` removes something most VPS setups do not need. I keep `AllowTcpForwarding yes` because many deploy and tunnel workflows still need it.

### Step 8: Reload SSH Safely

Do not restart blindly. Validate the config first, keep the current shell open, and test from another session again after reload.

```bash
# Validate config, then reload SSH
sudo sshd -t
sudo systemctl reload ssh
```

<Callout type="danger" title="Safe reload rule">
Keep the current root or sudo session open until you confirm `ssh deploy@YOUR_SERVER_IP` still works after the reload.
</Callout>

Then test again:

```bash
# Confirm deploy login still works after reload
ssh deploy@YOUR_SERVER_IP
whoami
```

If that second test works, your SSH hardening path is in a safe state.

## Recommended SSH config example block

This is a clean baseline example for a single deploy user on Ubuntu:

```text
# /etc/ssh/sshd_config
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers deploy
MaxAuthTries 3
X11Forwarding no
AllowTcpForwarding yes
KbdInteractiveAuthentication no
UsePAM yes
```

This is not the most aggressive SSH config possible. It is the practical baseline I would trust on a beginner-friendly production VPS.

## What I verify after tightening SSH access

This is the SSH-specific verification pass I do before I move on to broader server hardening:

1. `ssh deploy@YOUR_SERVER_IP` works from a clean second terminal
2. `whoami` returns the expected non-root user
3. `ssh root@YOUR_SERVER_IP` is rejected
4. password login is rejected when key auth is missing
5. `sshd -t` passes with no config errors
6. the current session stays open until the second session is confirmed
7. port forwarding still works if this server genuinely needs it

Once those checks pass, then I move outward to UFW, Fail2Ban, unattended upgrades, and the rest of the box.

## SSH security checklist

Use this before you call the server "ready":

1. Deploy user exists and has `sudo`
2. SSH key login works in a separate session
3. Root SSH login is disabled
4. Password SSH login is disabled
5. Only the deploy user is allowed
6. `MaxAuthTries` is limited
7. `sshd -t` passes before reload
8. SSH reload succeeds without locking you out
9. A second login works after the reload
10. Root login fails after the reload
11. Password login fails after the reload
12. Forwarding behavior matches what the server actually needs

If two items matter most, they are these: key login must work before you harden, and the old session must stay open until the new login is confirmed after reload.

## If I Were Setting Up SSH on a New VPS Today

I would keep the workflow boring:

1. Create `deploy`
2. Add the SSH key
3. Test `ssh deploy@SERVER_IP` in a new terminal
4. Back up `sshd_config`
5. Disable root login
6. Disable password login
7. Restrict SSH to `deploy`
8. Set `MaxAuthTries 3`
9. Validate with `sshd -t`
10. Reload SSH
11. Test deploy login again
12. Move to the broader server hardening guide

I would not change the SSH port on day one unless I had already confirmed firewall rules and recovery access. Port changes can cut noise, but they are not the first win. Keys, no root login, no password login, and least-privilege users matter more.

## Conclusion: SSH Security Best Practices

SSH hardening does not need enterprise tooling to be effective. On a small Ubuntu VPS, the basics stop most real SSH risk: use a deploy user, use keys, disable root login, disable password login, allow only the user you trust, and limit login attempts.

The most important best practice is the safe order of operations. Test key login first. Keep the current session open. Validate before reload. Confirm login again after reload. That one habit prevents the most common beginner disaster: locking yourself out while trying to be secure.

If your server runs a real site, treat SSH as production infrastructure, not as a quick admin shortcut. It is the front door. Then move on to the broader Ubuntu hardening pass for firewall rules, bans, updates, and exposed services.

## FAQ

### Is disabling root SSH login enough to secure a VPS?

No. It helps, but it is only one layer. You still want SSH keys, password login disabled, a restricted user list, a firewall, and Fail2Ban. Disabling root login removes an obvious target. It does not secure the full SSH path by itself.

### Should I change the SSH port on a VPS?

It is optional. Changing the port can reduce log noise from basic scanners, but it is not a substitute for keys, no root login, and no password login. I treat port changes as a later cleanup step, not the first hardening step.

### How do I avoid locking myself out while hardening SSH?

Always test key login in a separate session before disabling passwords. Keep your current session open. Run `sudo sshd -t` before reload, then test `ssh deploy@YOUR_SERVER_IP` again after reload. Do not close the old session until the new one works.

### Should I leave `AllowTcpForwarding` on?

Only if you actually use SSH tunnels, remote forwards, or deploy workflows that depend on it. If you never use forwarding, turn it off. If you do use it, leave it on deliberately instead of copying a config block without understanding the tradeoff.
