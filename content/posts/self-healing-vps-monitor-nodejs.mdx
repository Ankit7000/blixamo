## Why Naive Retry Loops Fail

The standard approach to self-healing: detect problem → run fix → restart service. Simple — until a service restarts, fails again within seconds, triggers the monitor, restarts again, and fails again. Forty restart attempts in an hour. The actual problem — corrupted build, full disk, misconfigured env var — never gets fixed. The restart loop makes it worse by burying the real error in log noise.

Bayesian reasoning fixes this with one insight: every fix attempt is evidence.

If you've tried restarting PM2 six times and it worked twice, P(restart fixes this) ≈ 0.44. If you've tried it six times and it worked zero times, P ≈ 0.13. In the second case, "keep restarting" is the wrong move — escalate and let a human investigate.

The formula I use is Laplace smoothing: `P = (successes + 1) / (attempts + 2)`. The +1 and +2 avoid zero probabilities and give new fix types a fair starting prior of 0.5. After one successful fix, P jumps to 0.67. After three failures with no successes, P collapses to 0.20 — below my escalation threshold of 0.25.

The checker handles article quality, error scanning, and infra health. The supervisor is the meta-layer — it watches the checker's output, detects if the checker is hung, and applies heals at the infrastructure level.

<Callout type="danger" title="LAW 1: Supervisor Never Spawns Itself">

## Step 2 of 7 — The Confidence Function
#
This is the core Bayesian piece — 3 lines of code:
#
```bash
// Laplace-smoothed Bayesian confidence
// P(fix_works) starts at 0.5, updates on every attempt
function confidence(fixRecord) {
  if (!fixRecord) return 0.5
  const { attempts = 0, successes = 0 } = fixRecord
  return (successes + 1) / (attempts + 2)
}
```
What this gives you at various data points:
#
| Attempts | Successes | P(fix works) | Action |
|---|---|---|---|
| 0 | 0 | 0.50 | Try it (uniform prior) |
| 1 | 1 | 0.67 | Try it |
| 3 | 3 | 0.80 | High confidence |
| 3 | 0 | 0.20 | Stop — escalate |
| 6 | 1 | 0.25 | Borderline — escalate |
| 10 | 8 | 0.75 | Reliable fix |
#
The threshold is `MIN_CONFIDENCE = 0.25`. Below that, the supervisor stops attempting the fix and sends an alert. This is LAW 3.

## Step 3 of 7 — Anti-Loop Laws
#
Six laws are enforced in strict order before any heal action runs:
#
```bash
// LAW 1 — Supervisor NEVER spawns itself or the checker
// LAW 2 — Each type: max 3 attempts per 2h (thrash gate)
// LAW 3 — Each fix type: P < 0.25 → stop, escalate
// LAW 4 — Heal budget: max 5 per 24h across all types
// LAW 5 — 3 consecutive same issues → escalate, stop
// LAW 6 — 3 consecutive supervisor crashes → reboot VPS
```
Thrash detection (LAW 2) uses the rolling timestamps array:
#
```bash
function isThrashing(state, fixType) {
  const f = state.fixes[fixType]
  if (!f) return false
  const recentAttempts = (f.timestamps || [])
    .filter(t => Date.now() - t < 2 * 3600000).length
  return recentAttempts >= 3
}
```
If the same fix has been attempted 3+ times in the last 2 hours, it's thrashing — stop and escalate regardless of confidence.

## Step 4 of 7 — Evidence Collection
#
Each supervisor run collects a snapshot of 9 signals:
#
```bash
function collectEvidence(state) {
  let pm2Online = false
  try {
    const list = 
      JSON.parse(run('pm2 jlist 2>/dev/null').out)
    pm2Online = list
      .find(p => p.name === 'blixamo')
      ?.pm2_env?.status === 'online'
  } catch {}
//
  const n8nHealth  = 
    run('curl -s -o /dev/null'
      + ' -w "%{http_code}"'
      + ' http://localhost:5678/healthz'
    )
  const redisPing  = 
    run('docker exec redis redis-cli ping 2>/dev/null')
  const nginxAct   = run('systemctl is-active nginx')
  const watcherAct = 
    run('systemctl is-active blixamo-watcher')
  const cacheHdr   = 
    run(
      'curl -sI https://blixamo.com/ 2>/dev/null'
      + ' | grep -i "cdn-cache-control"'
    )
  const buildOk    = 
    fs.existsSync(
      `${BLIXAMO_DIR}/.next/required-server-files.json`
    )
  const checkerHung = fs.existsSync(LOCK_FILE) &&
    Date.now() - fs.statSync(LOCK_FILE).mtimeMs > 25 * 60000
//
  return {
    pm2Online,
    n8nDown:       n8nHealth.out !== '200',
    redisPong:     redisPing.out === 'PONG',
    nginxDown:     nginxAct.out  !== 'active',
    watcherActive: watcherAct.out === 'active',
    buildMissing:  !buildOk,
    cacheMissing:
      !cacheHdr.out.toLowerCase().includes('no-store'),
    checkerHung,
  }
}
```
Each signal maps directly to a hypothesis and a heal action:
#
```bash
function buildHypotheses(evidence) {
  const hyps = []
  if (!evidence.pm2Online)
    hyps.push({ id: 'restartPm2', prior: 0.95 })
  if (evidence.nginxDown)
    hyps.push({ id: 'restartNginx', prior: 0.90 })
  if (!evidence.redisPong)
    hyps.push({ id: 'restartRedis', prior: 0.85 })
  if (!evidence.watcherActive)
    hyps.push({ id: 'restartWatcher', prior: 0.85 })
  if (evidence.n8nDown)
    hyps.push({ id: 'restartN8n', prior: 0.88 })
  if (evidence.buildMissing)
    hyps.push({ id: 'cleanBuild', prior: 0.80 })
  if (evidence.cacheMissing)
    hyps.push({ id: 'fixNginxCache', prior: 0.75 })
  if (evidence.checkerHung)    hyps.push({
      id: 'clearCheckerLock',
      prior: 0.90
    })
  return hyps
}
```

## Step 5 of 7 — The Heal Actions
#
Eight heal functions. Each returns `true` on success and `false` on failure:
#
```bash
const HEALS = {
  restartPm2: {
    label: 'Restart PM2 blixamo',
    run: () => run('pm2 restart blixamo').ok,
  },
//
  restartNginx: {
    label: 'Restart nginx',
    run: () => {
      const test = run('nginx -t 2>&1')
      if (!test.out.includes('ok')) return false
      return run('systemctl restart nginx').ok
    }
  },
//
  restartRedis: {
    label: 'Restart Redis docker container',
    run: () => {
      run('docker restart redis')
      const r = 
        run(
          'sleep 5 && docker exec redis'
          + ' redis-cli ping 2>/dev/null'
        )
      return r.out === 'PONG'
    }
  },
//
  cleanBuild: {
    label: 'Clean Next.js build',
    run: () => {
      run(`rm -rf ${BLIXAMO_DIR}/.next`)
      const b = 
        run(
          `cd ${BLIXAMO_DIR}`
          + ` && npm run build 2>&1 | tail -3`,
          120000
        )
      if (!b.ok || b.out.includes('error')) return false
      return run('pm2 restart blixamo').ok
    }
  },
//
  fixNginxCache: {
    label: 'Fix Cloudflare CDN-Cache-Control header',
    run: () => {
      const conf = 
        fs.readFileSync(
          '/etc/nginx/sites-enabled/blixamo', 'utf8'
        )
      const needle = 'CDN-Cache-Control "no-store"'
      const hasHeader = conf.includes(needle)
      if (hasHeader) return true  // idempotent
      const fixed = conf.replace(
        /location \/ \{/,
        'location / {\n'
        + '  add_header CDN-Cache-Control "no-store";'
      )
      fs.writeFileSync(
        '/etc/nginx/sites-enabled/blixamo',
        fixed
      )
      return run('nginx -t 2>&1').out.includes('ok') &&
             run('systemctl reload nginx').ok
    }
  },
//
  clearCheckerLock: {
    label: 'Clear hung checker lock file',
    run: () => {
      run(`rm -f ${LOCK_FILE}`)
      run('pkill -f "blixamo-checker.js" 2>/dev/null 
        || true')
      return true
    }
  },
}
```
<Callout type="tip" title="Why cleanBuild Has a 120-Second Timeout">
Next.js builds on a CPX22 take 60–90 seconds. The default run() timeout of 20 seconds would kill it mid-build, leaving a corrupted .next directory — which would then trigger the buildMissing hypothesis on the next supervisor run. Set the timeout explicitly for any long-running heal action.
</Callout>

## Step 6 of 7 — The Main Loop
#
The heal loop processes hypotheses in order, checking all 4 gates before attempting any fix:
#
```bash
for (const hyp of hyps) {
  const fixRecord = state.fixes[hyp.id]
  const conf    = confidence(fixRecord)
  const thrash  = isThrashing(state, hyp.id)
  const budget  = budgetAvailable(state)
//
  if (conf < MIN_CONFIDENCE) {
    sendAlert(state, `Low confidence: ${hyp.label}`, ...)
    continue
  }
//
  if (thrash) {
    sendAlert(state, `Thrash loop: ${hyp.label}`, ...)
    continue
  }
//
  if (!budget) {
    sendAlert(state, 'Self-heal budget exhausted', ...)
    break
  }
//
  useBudget(state)
  const success = await HEALS[hyp.id].run()
  recordAttempt(state, hyp.id, success)
//
  if (!success) {
    sendAlert(state, `Heal failed: ${hyp.label}`, ...)
  }
}
```

## FAQ
#
### Does this work without n8n?
#
Yes. The supervisor alerts via n8n webhook because n8n is already running on my VPS. Replace the `sendAlert()` function with a direct SMTP call using `nodemailer`, or write alerts to a file and tail it. The Bayesian logic and heal actions are entirely independent of the alerting mechanism.
#
### What happens if the supervisor itself crashes?
#
`supervisorHealth.consecutiveCrashes` tracks this. Each run wraps the main function in a try/catch — on crash, it increments the counter and exits with code 1. At 3 consecutive crashes, it sends a final alert and triggers `shutdown -r +1` (VPS reboot with 1-minute delay). In production I've never hit this.
#
### Can I run this on a non-Hetzner VPS?
#
Yes. The script checks services specific to my setup: PM2 (blixamo), Docker containers (n8n, Redis), and systemd services (blixamo-watcher). Replace the evidence collection with whatever services you run. The Bayesian engine, state management, and anti-loop laws are generic Node.js with no Hetzner-specific code.
#
### Why not use Monit or PM2's built-in watch?
#
PM2 restarts crashed processes automatically — it handles the simple case. Monit is excellent for rule-based monitoring. What neither does is Bayesian confidence tracking: they'll keep retrying a failing action indefinitely. The budget cap, thrash detection, and confidence gating are the specific problems this system solves that generic tools don't address.
#
### How much RAM does the supervisor use?
#
Node.js process peak: ~40MB during the run. It exits immediately after completing, holding zero RAM between runs. The state file stays under 10KB for a year of history. Total overhead: negligible.
#
### Is the state file safe to delete?
#
Yes. Deleting it resets all confidence history and budgets — the supervisor starts fresh with 0.5 priors on every fix type. The only downside is losing accumulated evidence about which fixes work reliably on your specific setup.
