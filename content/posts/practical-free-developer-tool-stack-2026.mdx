I do not build a "free stack" by collecting one tool from every category. I build it by asking a harsher question: which tools still feel worth opening after the project gets messy?

For a small project, I want the stack to stay boring. That means one tool for API work, one path for docs, one way to stay sane in Git, one diagram pair, and one database GUI that does not turn into another subscription decision too early.

<Callout type="info" title="TL;DR">
If I were starting a small project today, I would use **Bruno**, **MkDocs Material** or **Nextra**, **Lazygit**, **Excalidraw**, **Mermaid**, and **DBeaver**. That stack stays light, cheap, and easy to explain to the next person.
</Callout>

## Who this is for

- Solo developers.
- Small teams where one person still understands most of the system.
- Side projects and client work where tooling should support delivery, not become its own budget line.

## Skip this if

- Your team already needs heavier permissions, comments, and audit trails.
- The main problem is enterprise coordination, not developer speed.

## What I keep in the stack from day one

### API work: Bruno first, Hoppscotch only when I need it

My default is **Bruno** because I want API collections to live close to the code. Plain files, local-first behavior, and Git-friendly review matter more to me than a glossy workspace.

I add **Hoppscotch** only for quick one-off checks. That is the first example of how I keep the stack practical: one default, one optional specialist. Not 4 overlapping clients.

If API testing is the whole problem, [Best Free API Testing Tools for Developers in 2026](/blog/best-free-api-testing-tools-2026) is the page I would read next.

### Docs: MkDocs Material unless the project is already deep in Next.js

For docs, I care about whether I will still update them in 3 months. That is why I usually start with **MkDocs Material**. It is fast, Markdown-first, and does not ask me to turn documentation into a side frontend project.

The exception is **Nextra**. If the project already lives inside a Next.js content workflow, I would rather keep docs in the same ecosystem than invent a separate docs stack just to be "pure."

If the docs choice itself is blocking you, go deeper with [Best Free Documentation Tools for Developers in 2026](/blog/best-free-documentation-tools-2026).

### Git: normal Git plus Lazygit

I have no interest in building a complicated Git toolchain for a small project. I use Git as the source of truth and **Lazygit** as the speed layer for staging, review, and branch cleanup.

That combination is enough. I do not want 3 Git GUIs installed just because they are free.

### Diagrams: Excalidraw for thinking, Mermaid for keeping

I want two diagram modes:

- **Excalidraw** when I am still figuring things out.
- **Mermaid** when the diagram should live in docs or a README.

That split keeps me honest. Rough thinking stays rough. Repeatable documentation stays in text.

### Database work: DBeaver

For small projects I want one database GUI that does not make me think twice. That is **DBeaver**. It is not the prettiest tool here, but it handles daily inspection, query work, and boring admin tasks well enough that I rarely need more.

If interface polish matters more than flexibility, I would try TablePlus. But for a safe free default, DBeaver is still my answer.

## What I deliberately do not add early

### I do not add a heavy docs portal too soon

If the project has 8 pages of docs, I do not need a grand documentation platform. I need something I will keep current.

### I do not add multiple tools in the same lane

A free stack gets expensive fast when it adds decision fatigue. One default per job is the rule.

### I do not pay to avoid small discomfort too early

Some friction is fine. Recurring cost should buy back real time, not emotional relief from having one more subscription.

## When I would pay later

I would upgrade parts of this stack only when the problem changes from "help me ship" to "help us coordinate."

That usually shows up as:

- shared permissions become annoying
- review history matters more
- free-tier limits interrupt normal work
- one missing feature creates manual cleanup every week

That is the point where "free" stops being cheap in time.

## Why this stack works for small projects

This stack works because every tool has one clear job:

- **Bruno** for durable API work
- **MkDocs Material** or **Nextra** for docs
- **Lazygit** for faster Git flow
- **Excalidraw** for quick architecture thinking
- **Mermaid** for diagrams that should live in the codebase
- **DBeaver** for database inspection

I can explain every choice in one sentence. That is the real test.

## What I would change for a bigger team

I would not force this exact stack onto a team of 15. Once more people are touching the same workflows, the questions change. Permissions, handoffs, approvals, and structured review start to matter more than the elegance of a lean local setup.

That is when I would revisit docs, API tooling, and maybe the database layer first. Those are the lanes where collaboration overhead shows up earliest.

## Final recommendation

For a small project, I would keep the stack lean on purpose: **Bruno**, **MkDocs Material** or **Nextra**, **Lazygit**, **Excalidraw**, **Mermaid**, and **DBeaver**. That is enough to do real work without turning tooling into another project to manage.

The honest exception is the docs layer. If the project already runs on Next.js and MDX, I would choose **Nextra** over MkDocs because the ecosystem fit matters more than staying language-agnostic. Otherwise, MkDocs is still the cleaner default.
