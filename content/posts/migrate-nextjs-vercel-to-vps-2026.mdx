Moving a Next.js app off Vercel is not mostly about `next build`. It is about replacing everything Vercel was quietly doing for you: environment handling, build/runtime parity, image behavior, cron jobs, SSL, and rollback.

This runbook assumes the app already works on Vercel and the real risk is the cutover, not learning how to host Next.js from scratch.

![Migration runbook cover showing the Vercel-to-VPS move for a Next.js app](/images/posts/migrate-nextjs-vercel-to-vps-2026/desktop-cover.jpg)
*This overview visual frames the migration as a controlled cutover, not a blind redeploy.*

> **TL;DR**
>
> - Export the real Vercel assumptions first: env vars, domains, cron jobs, image behavior, and callback URLs.
> - Make the VPS build, boot, and answer locally before you touch public DNS.
> - Recreate the boring platform layer on purpose: proxy, SSL, scheduled tasks, uploads, and webhook targets.
> - Keep Vercel serving until the VPS survives the first real production checks without surprises.

## Migration Decision Table

| Concern | What to verify before you leave Vercel | VPS replacement |
| --- | --- | --- |
| Environment variables | Which vars are needed at build time vs runtime | `.env`, Compose envs, or Coolify env settings |
| Custom domain and SSL | Current DNS records, redirect rules, and cert assumptions | Nginx, Caddy, or Coolify-managed proxy |
| Scheduled tasks | Vercel cron jobs or serverless schedules | `cron`, Coolify scheduled jobs, or external scheduler |
| Image handling | Whether you rely on default `next/image` behavior and cache | Self-hosted Next.js image optimizer and cache/storage plan |
| Rollback | What "go back now" means in the first 10 minutes | Keep Vercel serving until the VPS is proven |

## When this approach is the right choice

Use this runbook when:

- your Next.js app has outgrown Vercel pricing or platform limits
- you want one VPS to host the app plus adjacent services
- you are comfortable owning the deploy, proxy, and rollback path
- you need a migration checklist, not a generic "self-host Next.js" overview

Do not use this page if you are still deciding whether a VPS is the right hosting model. Start with [How to Self-Host Next.js on a VPS](/blog/how-to-self-host-nextjs-on-vps) or [Deploy Next.js on Hetzner with Coolify](/blog/deploy-nextjs-coolify-hetzner) first, then come back when the migration is real.

## Prerequisites and assumptions

- working access to the Vercel project, domain DNS, and current environment variables
- a VPS with Docker Compose or a platform layer like Coolify already chosen
- one clear deployment path for the new host
- a health check or at least one URL you can trust during verification
- a rollback owner and a low-traffic cutover window

If the VPS still needs prep work, stop here and use [How to Set Up a VPS for Web Hosting in 2026](/blog/vps-setup-guide) plus [Next.js VPS Prelaunch Checklist in 2026](/blog/nextjs-vps-prelaunch-checklist-2026).

## Step 1: Audit the Vercel-specific assumptions

Before you deploy anything, write down what the app currently depends on.

Your audit should answer:

- which env vars are needed during `next build`
- which env vars are only needed at runtime
- whether any route depends on Vercel-only features such as scheduled functions or edge behavior
- where uploads, generated files, or cache data live today
- whether background jobs, webhooks, or third-party callbacks point at the Vercel URL

The dangerous migration is the one where the team remembers only the app code and forgets the platform behavior around it.

## Step 2: Pick one VPS deploy path and stop changing architecture mid-migration

This is where teams create unnecessary risk. They start with "move from Vercel to a VPS" and halfway through it becomes "also switch package manager, also change proxy, also rebuild CI, also move databases."

Do not do that.

Pick one deployment path:

- Coolify if you want a UI-led deployment layer and easier app management
- raw Docker Compose if you want fewer moving parts and you already operate containers comfortably

Either path is fine. What is not fine is changing both the hosting model and the deployment model in the same cutover without a reason.

If you are leaning toward Coolify, pair this with [Fix Failed Next.js Deploys in Coolify (2026)](/blog/fix-coolify-deploy-failures-nextjs-2026). If you are staying close to the metal, keep [Docker Compose on VPS 2026 - Production Setup Guide](/blog/docker-compose-production-vps-2026) open.

## Step 3: Make the app build and boot on the VPS before touching DNS

Your first success condition is simple: the app builds on the VPS, starts cleanly, and answers locally.

For a plain Node-based deployment, the minimum proof looks like this:

```bash
npm ci
npm run build
PORT=3000 HOSTNAME=0.0.0.0 npm run start
curl -I http://127.0.0.1:3000
```

If you are deploying through Docker, the same rule applies. The container must build, listen on the expected port, and answer locally before the public domain moves.

Common migration misses here:

- the build succeeds locally but not on the VPS because build-time env vars are missing
- the app listens on the wrong port or only on `localhost`
- the app writes to local disk for something that used to be handled elsewhere
- a route depends on a background job that has not been recreated on the VPS

Before you cut DNS, prove the VPS stack is answering on the target runtime and not just "deployed" on paper.

## Step 4: Recreate the surrounding platform behavior

The app is only one part of the migration. Rebuild the boring platform pieces explicitly:

1. add the env vars to the new runtime
2. configure the reverse proxy and SSL
3. recreate scheduled jobs
4. confirm any webhook targets now point at the new domain
5. verify image optimization, file writes, and cache directories behave the way you expect

This is also where a compact migration checklist helps:

| Check | Done when |
| --- | --- |
| Build parity | `npm run build` succeeds on the VPS |
| Runtime parity | the app starts and serves the main path locally |
| Domain parity | the production hostname serves the new app after cutover |
| Background jobs | scheduled tasks run on the VPS, not on the old platform |
| Rollback parity | you can point traffic back without guessing |

If search and analytics matter for this app, keep [Google Search Console for Self-Hosted Next.js](/blog/google-search-console-self-hosted-nextjs) nearby after the move.

## Step 5: Cut over traffic with rollback still available

Do not destroy the old path before the new one is trusted.

The safest cutover sequence is:

1. reduce DNS TTL ahead of time if you control it
2. verify the new stack on the VPS using a temporary host or direct IP/proxy route where possible
3. move the production domain
4. re-run the core user path immediately
5. keep Vercel intact until you are satisfied with the new host

The point is not elegance. The point is making rollback cheap.

## Verification: what proves the migration actually worked

Do not call the move done until these checks pass:

- homepage or primary route returns `200`
- login or auth callback works if the app has one
- `next/image` or other asset-heavy pages load correctly
- background jobs and webhooks hit the new environment
- logs stay clean after the first real traffic
- SSL and redirect behavior match the old production behavior
- at least one rollback step has been written down, not assumed

This is the proof point that matters most: the production hostname is serving the new stack cleanly after cutover, with logs that match that claim.

## Common migration failures

### Build-time and runtime env vars were mixed together

This is the most common failure in Next.js migrations. The build works in one environment and the runtime fails in another, or vice versa. Audit both sets separately.

### The app boots, but the proxy points at the wrong port

A green deploy is not enough if the reverse proxy expects a different internal port or the app is bound incorrectly.

### Vercel cron jobs were forgotten

If a route or workflow relied on Vercel scheduling, nothing on the VPS will replace it by accident. Recreate it intentionally.

### The team removed Vercel too early

Do not kill the old environment because the first test passed once. Watch the app under real traffic first.

### The migration also changed too many unrelated things

Changing hosting, proxy, runtime, background jobs, and database topology in one wave makes debugging much harder than it needs to be.

## Rollback plan

Your rollback plan should fit in a few lines:

1. switch traffic back to Vercel
2. disable or pause new VPS-only jobs if they create duplicate actions
3. keep the VPS stack for investigation, not for live writes
4. capture the failure cause before retrying

If you need a dedicated rollback checklist after the cutover, use [How to Roll Back a Failed Next.js VPS Deploy in 2026](/blog/rollback-failed-nextjs-vps-deploy-2026).

## FAQ

### What usually breaks first when moving Next.js from Vercel to a VPS?

Usually the first break is not the framework itself. It is missing env vars, a wrong proxy target, forgotten cron jobs, or image/cache assumptions that were previously hidden by the platform.

### Should I migrate directly to raw Docker Compose or use Coolify?

Use the path you are most likely to operate calmly. If you want a UI and easier app management, Coolify is a reasonable step. If you already run containers directly, Docker Compose is often simpler.

### Do I need to move DNS before I know the VPS build works?

No. Build and test locally on the server first. DNS is part of cutover, not part of initial validation.

### Can I keep Vercel live while I test the VPS?

You should. Keeping the old path intact is what makes rollback cheap.

### How long should I keep Vercel live after cutover?

Keep it live until the VPS has passed the core checks you actually care about: main route, auth callback if relevant, scheduled jobs, uploads or image handling, and at least one period of real traffic without new errors. The cheap rollback window is worth more than deleting the old deployment immediately.
