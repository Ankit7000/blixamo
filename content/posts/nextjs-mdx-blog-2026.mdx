<Callout type="warning" title="next-mdx-remote vs @next/mdx — Critical Distinction">
`next-mdx-remote` compiles and renders MDX content stored on disk at runtime — the right choice for a blog with frontmatter, tags, and ISR. `@next/mdx` treats `.mdx` files as page routes. For a dynamic blog, always use `next-mdx-remote`.
</Callout>

## Step 2 of 6 — The Posts Library (lib/posts.ts)
#
This is the most important file in the project. Get this right and everything else is simple.
#
```bash
import fs from 'fs'
import path from 'path'
import matter from 'gray-matter'
import readingTime from 'reading-time'
//
const POSTS_DIR = path.join(process.cwd(), 'content/posts')
//
export interface Post {
  slug:           string
  title:          string
  description:    string
  date:           string
  updatedAt:      string
  author:         string
  category:       string
  tags:           string[]
  keyword:        string
  featured:       boolean
  featuredImage: /images/nextjs-mdx-blog-2026.jpg
  readingTime:    string
  excerpt:        string
  series?:        string
  seriesPart?:    number
}
//
export function getAllPosts(): Post[] {
  const files = 
    fs.readdirSync(POSTS_DIR)
      .filter(f => f.endsWith('.mdx'))
  return files
    .map(filename => {
      const raw = 
        fs.readFileSync(
          path.join(POSTS_DIR, filename), 'utf8'
        )
      const { data, content } = matter(raw)
      const rt = readingTime(content)
      return {
        slug: data.slug || filename.replace('.mdx', ''),
        title:         data.title || '',
        description:   data.description || '',
        date:          data.date || '',
        updatedAt:     data.updatedAt || data.date || '',
        author:        data.author || 'Ankit Sorathiya',
        category:      data.category || 'tutorials',
        tags:          data.tags || [],
        keyword:       data.keyword || '',
        featured:      data.featured || false,
        featuredImage: /images/nextjs-mdx-blog-2026.jpg
        readingTime:   rt.text,
        excerpt:       data.excerpt || '',
        series:        data.series || undefined,
        seriesPart:    data.seriesPart || undefined,
      } as Post
    })
    .sort((a, b) =>
      new Date(b.date).getTime()
        - new Date(a.date).getTime()
    )
}
//
export function getPostBySlug(
  slug: string
): { post: Post; content: string } | null {
  const filePath = path.join(POSTS_DIR, `${slug}.mdx`)
  if (!fs.existsSync(filePath)) return null
  const raw = fs.readFileSync(filePath, 'utf8')
  const { data, content } = matter(raw)
  const rt = readingTime(content)
  return {
    post: { ...data, slug, readingTime: rt.text } as Post,
    content
  }
}
//
export function getAllTags(): string[] {
  return [...new Set(
    getAllPosts().flatMap(p => p.tags)
  )].sort()
}
```
`getPostBySlug` returns both the parsed `Post` metadata AND the raw `content` string separately — the content is the MDX body without frontmatter, which gets compiled and rendered by `next-mdx-remote`.

## Step 3 of 6 — The Blog Post Page
#
```bash
// app/blog/[slug]/page.tsx
import { MDXRemote } from 'next-mdx-remote/rsc'
import { getAllPosts, getPostBySlug } from '@/lib/posts'
import { notFound } from 'next/navigation'
import type { Metadata } from 'next'
import rehypePrettyCode from 'rehype-pretty-code'
import rehypeSlug from 'rehype-slug'
import remarkGfm from 'remark-gfm'
//
type Props = { params: Promise<{ slug: string }> }
//
export async function generateStaticParams() {
  return getAllPosts().map(post => ({ slug: post.slug }))
}
//
export async function generateMetadata(
  { params }: Props
): Promise<Metadata> {
  const { slug } = await params
  const result = getPostBySlug(slug)
  if (!result) return {}
  const { post } = result
  return {
    title: post.title,
    description: post.description,
    alternates: {
      canonical: `https://blixamo.com/blog/${slug}`
    },
    openGraph: {
      title: post.title,
      description: post.description,
      type: 'article',
      publishedTime: post.date,
      images: [{
        url: `https://blixamo.com${post.featuredImage}`,
        width: 1200, height: 630
      }],
    },
  }
}
//
export default async function BlogPost({ params }: Props) {
  const { slug } = await params
  const result = getPostBySlug(slug)
  if (!result) notFound()
  const { post, content } = result
//
  return (
    <article style={{
      maxWidth: '780px',
      margin: '0 auto',
      padding: '2rem 1rem'
    }}>
      <h1>{post.title}</h1>
      <p>{post.date} · {post.readingTime}</p>
      <MDXRemote
        source={content}
        options={{
          mdxOptions: {
            remarkPlugins: [remarkGfm],
            rehypePlugins: [
              rehypeSlug,
              [rehypePrettyCode, {
                theme: 'github-dark',
                keepBackground: true
              }],
            ],
          },
        }}
      />
    </article>
  )
}
//
export const revalidate = 3600
```
The `revalidate = 3600` tells Next.js to keep cached HTML for 1 hour before regenerating. This is ISR — Incremental Static Regeneration. At build time Next.js pre-renders all post pages. When a visitor hits a page after the revalidation period, Next.js regenerates it in the background while serving the cached version. The next visitor gets the fresh version.

## Step 4 of 6 — The ISR + Cloudflare Fix (Critical)
#
This is the fix every other tutorial skips. It will silently break your blog if you miss it.
#
When Cloudflare sits in front of your Nginx server, it caches HTML responses by default. Your `revalidate: 3600` ISR setting is ignored — Cloudflare serves its own cached version indefinitely. New articles take days to appear. Updated content never updates. The page shows HTTP 200 so nothing looks broken.
#
The fix is two Nginx directives that tell Cloudflare to never cache HTML:
#
```bash
server {
    listen 443 ssl;
    server_name blixamo.com www.blixamo.com;
#
    # HTML pages — tell Cloudflare NOT to cache (ISR fix)
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
#
        proxy_hide_header Cache-Control;
        add_header Cache-Control
          "public, s-maxage=0, must-revalidate";
        add_header CDN-Cache-Control "no-store";
    }
#
    # Static assets — Cloudflare SHOULD cache forever
    location /_next/static/ {
        proxy_pass http://localhost:3000;
        add_header Cache-Control
          "public, max-age=31536000, immutable";
        add_header CDN-Cache-Control
          "public, max-age=31536000, immutable";
    }
#
    # Public images — 30 day Cloudflare cache
    location /images/ {
        proxy_pass http://localhost:3000;
        proxy_hide_header Cache-Control;
        add_header Cache-Control "public, max-age=2592000";
        add_header CDN-Cache-Control
          "public, max-age=2592000";
    }
}
```
The `proxy_hide_header Cache-Control` removes Next.js's own header before adding yours. Without it, you end up with duplicate Cache-Control headers and undefined browser behavior.
#
Always test before reloading Nginx:
#
```bash
nginx -t && systemctl reload nginx
```
<Callout type="tip" title="Expected output">
`nginx: syntax is ok / test is successful` before reload. After reload, test with `curl -I https://yourdomain.com/blog/any-post | grep -i cdn-cache` — you should see `CDN-Cache-Control: no-store`.
</Callout>

## Step 5 of 6 — PM2 Configuration for Production
#
Blixamo runs via PM2, not [Docker Compose in production](/blog/docker-compose-production-vps-2026). PM2 is lighter than Docker for a single Next.js app on a VPS — it runs the process directly without container overhead.
#
```bash
// ecosystem.config.js
module.exports = {
  apps: [{
    name:        'blixamo',
    script:      'node_modules/.bin/next',
    args:        'start',
    cwd:         '/var/www/blixamo',
    instances:   1,
    exec_mode:   'fork',
    autorestart: true,
    max_memory_restart: '512M',
    wait_ready:  true,
    listen_timeout: 10000,
    env: { NODE_ENV: 'production', PORT: 3000 }
  }]
}
```
`wait_ready: true` tells PM2 to wait for Next.js to signal readiness before marking the process online — prevents the "PM2 shows online but site returns 502" issue. `max_memory_restart: '512M'` prevents memory leaks from accumulating.
#
```bash
pm2 start ecosystem.config.js
pm2 save        # persist across reboots
pm2 startup     # configure startup on boot
```

## Step 6 of 6 — Auto-Generated Sitemap
#
The sitemap at `app/sitemap.ts` reads all posts and generates proper XML automatically. No plugin, no manual updates:
#
```bash
import { getAllPosts, getAllTags } from '@/lib/posts'
import type { MetadataRoute } from 'next'
//
const SITE = 'https://blixamo.com'
//
export default function sitemap(): MetadataRoute.Sitemap {
  const posts = getAllPosts()
//
  const staticPages = [
    { url: SITE, lastModified: new Date(), priority: 1.0 },
    {
      url: `${SITE}/blog`,
      lastModified: new Date(),
      priority: 0.9
    },
  ]
//
  const postPages = posts.map(p => ({
    url: `${SITE}/blog/${p.slug}`,
    lastModified: new Date(p.updatedAt || p.date),
    priority: p.featured ? 0.95 : 0.85,
  }))
//
  return [...staticPages, ...postPages]
}
```
The sitemap auto-includes new posts the moment they appear in `content/posts/`. Submit it to Google Search Console once and it stays current.

const nextConfig = {
  transpilePackages: ['rehype-pretty-code', 'shiki'],

## FAQ
#
### What is the difference between next-mdx-remote and @next/mdx?
#
`next-mdx-remote` compiles and renders MDX stored as files at runtime — right for blogs with frontmatter, tags, categories, and ISR. `@next/mdx` treats `.mdx` files as page routes. For a dynamic blog, use `next-mdx-remote`.
#
### Can I use this stack on Vercel?
#
Yes — the pipeline works identically on Vercel. You lose the Rs 465/month total cost advantage (Vercel Pro starts at $20/month). The ISR + Cloudflare fix in Step 4 is not needed on Vercel since it handles ISR natively. If budget matters, self-host on Hetzner.
#
### How long does a Next.js build take on Hetzner CPX22?
#
Blixamo with 17 articles builds in approximately 35–45 seconds on CPX22 (2 vCPU, 4GB RAM). At 50 articles, expect 60–90 seconds. The bottleneck is page pre-rendering, not compilation. Add 2GB swap to prevent OOM kills during large builds.
#
### How do I deploy a new article without rebuilding everything?
#
For MDX-only changes, ISR handles freshness automatically. For the new post to appear in the blog index and sitemap (which use `getAllPosts()` at build time), you need a rebuild. On CPX22, `npm run build && pm2 restart blixamo` completes in under 60 seconds.
#
### What is ISR and why does it matter?
#
ISR (Incremental Static Regeneration) serves pre-built HTML instantly — page loads under 200ms — while keeping content fresh without a full site rebuild. With `revalidate: 3600`, Next.js regenerates the page in the background when visited after 1 hour. You get static site performance with fresh content.
#
---
#
### How do I add syntax highlighting to code blocks in MDX?
#
Use `rehype-pretty-code` with a Shiki theme. Install it with `npm install rehype-pretty-code shiki`, then add it to your `next.config.ts` MDX options: `[rehypePrettyCode, { theme: 'github-dark', keepBackground: true }]`. This gives you token-level highlighting for 200+ languages with zero client-side JavaScript. The `keepBackground: true` option preserves the theme's background colour inside code blocks so they don't inherit your page background unexpectedly.
