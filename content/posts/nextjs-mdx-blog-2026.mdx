If you want a developer blog with Next.js and MDX in 2026, the cleanest setup is still App Router plus `next-mdx-remote/rsc`, with posts stored on disk and a small posts library handling frontmatter, reading time, related posts, and metadata.

I built blixamo.com this way because I wanted the blog to behave like an app repo, not a CMS subscription. The real challenge is not rendering MDX once. The real challenge is picking the right MDX approach, keeping metadata clean, and deploying the site without breaking revalidation or search visibility.

## Who This Setup Is For

This setup makes sense if:

- you are a developer comfortable with files, Git, and TypeScript
- you want full control over layout, metadata, and internal linking
- you do not need a non-technical editor on day one
- you want the blog to live inside the same codebase as the rest of the site

## Why Build a Next.js MDX Blog Instead of Using WordPress?

For developers, the biggest advantage is control. Your content lives in Git. You can review diffs, keep metadata close to the code, and build the exact page structure you want without relying on plugins or a separate CMS workflow.

The cost profile is also better if you are already comfortable with a VPS. If deployment is the part you are unsure about, pair this article with [Deploy Next.js on Hetzner with Coolify](/blog/deploy-nextjs-coolify-hetzner) or [How to self-host Next.js on a VPS](/blog/how-to-self-host-nextjs-on-vps).

## When to Use next-mdx-remote vs @next/mdx

This is the first decision that confuses most tutorials.

| Option | Choose it when | Avoid it when |
| --- | --- | --- |
| `next-mdx-remote/rsc` | your MDX files are content data inside a real blog system | you want route files to be the content layer |
| `@next/mdx` | you want MDX pages directly in the app router | you need a stronger posts library with frontmatter-driven helpers |
| CMS + MDX | non-technical editors need a UI | you want the smallest and cheapest stack |

For a real blog with frontmatter, categories, related posts, featured images, and custom metadata, `next-mdx-remote/rsc` is the better fit. It lets you keep a proper posts library instead of turning every MDX file into route definitions.

## Step 2: Build the Posts Library

Your posts library should read files, parse frontmatter, compute reading time, and expose helpers like `getAllPosts()` and `getPostBySlug()`.

If this exact setup works in dev but breaks after deploy, the [next-mdx-remote/rsc Edge Runtime error](/blog/nextjs-mdx-remote-rsc-edge-runtime-fix) guide covers the Node.js runtime mismatch that usually causes it.

![MDX frontmatter and content structure used in the production blog workflow](/images/posts/nextjs-mdx-blog-2026/04-mdx-frontmatter.png)
*File-backed MDX only stays clean when frontmatter is consistent and the posts library is the single source of truth.*

```ts
import fs from "fs";
import path from "path";
import matter from "gray-matter";
import readingTime from "reading-time";

const POSTS_DIR = path.join(process.cwd(), "content/posts");

export function getAllPosts() {
  return fs
    .readdirSync(POSTS_DIR)
    .filter((file) => file.endsWith(".mdx"))
    .map((file) => {
      const slug = file.replace(/\.mdx$/, "");
      const raw = fs.readFileSync(path.join(POSTS_DIR, file), "utf8");
      const { data, content } = matter(raw);
      return {
        slug,
        ...data,
        readingTime: readingTime(content).text,
      };
    });
}
```

This is the layer that later powers your homepage, category pages, related posts, sitemap, and RSS.

## Step 3: Render MDX in App Router

```tsx
import { MDXRemote } from "next-mdx-remote/rsc";
import remarkGfm from "remark-gfm";
import rehypeSlug from "rehype-slug";
import rehypePrettyCode from "rehype-pretty-code";
import { getAllPosts, getPostBySlug } from "@/lib/posts";
import { notFound } from "next/navigation";

export async function generateStaticParams() {
  return getAllPosts().map((post) => ({ slug: post.slug }));
}

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const result = getPostBySlug(slug);
  if (!result) notFound();

  return (
    <article>
      <MDXRemote
        source={result.content}
        options={{
          mdxOptions: {
            remarkPlugins: [remarkGfm],
            rehypePlugins: [
              rehypeSlug,
              [rehypePrettyCode, { theme: "github-dark" }],
            ],
          },
        }}
      />
    </article>
  );
}
```

## Step 5: Make Code Blocks and Tables Worth Reading

Most tutorials stop after "it renders." That is not enough for a real developer blog.

Use `rehype-pretty-code` so code blocks are readable without client-side highlighting. Make sure your prose styles handle tables, captions, inline code, and mobile overflow properly.

## Step 6: Deploy It Without Breaking Revalidation

A Next.js MDX blog can run almost anywhere, but self-hosting adds one common failure point: proxy caching can fight HTML revalidation if your CDN or reverse proxy is too aggressive.

My preferred path for this kind of blog is:

1. build on the server
2. run with PM2 or a platform like Coolify
3. proxy through Nginx
4. cache static assets heavily
5. avoid stale HTML policies that hide content updates

## What Broke in Production for Me

These were the real breakpoints that separated a working demo from a stable blog.

### A file-backed MDX route inherited Edge runtime

The route looked fine locally and then broke only after a real production build. The underlying problem was simple: the content pipeline still depended on Node.js APIs while the route inherited Edge runtime higher up the tree.

### HTML caching hid fresh content

Static assets should be cached hard. HTML should not. Once the proxy or CDN keeps old HTML around too aggressively, revalidation stops feeling real even though the app build is correct.

### Frontmatter drift made the site feel inconsistent

The render path can be fine while the publication still looks sloppy because featured images, descriptions, or tags are inconsistent. That is why the posts library matters as much as the MDX renderer.

## Common Mistakes When Building a Next.js MDX Blog

### Mixing route-driven MDX with content-driven MDX

If you treat posts as route files too early, categories, related posts, and frontmatter-driven layouts become harder than they need to be.

### Letting metadata drift

If half your posts have featured images and half do not, the whole publication starts feeling inconsistent.

### Ignoring mobile reading experience

If code blocks overflow badly or headings feel cramped, the content feels lower quality even when the writing is strong.

### Overengineering the CMS layer

For a solo or developer-led blog, MDX on disk is usually enough for a long time.

## FAQ

### Is Next.js MDX better than WordPress for a developer blog?

For developers, often yes. The biggest wins are Git-based content, predictable deployment, and full control over rendering and metadata.

### Should I use next-mdx-remote or @next/mdx?

Use `next-mdx-remote/rsc` when your MDX files are content stored on disk and loaded through a posts library. Use `@next/mdx` when your MDX files are route-level content.

### Can I self-host a Next.js MDX blog on a VPS?

Yes. A small VPS can handle a Next.js MDX blog well if you build it correctly and keep the caching setup sane.

### Do I need a CMS for a Next.js blog in 2026?

Not always. If you are the main writer and comfortable with Git, MDX files on disk are enough for a lot of developer blogs.

### What should I build after the blog is working?

After the core blog works, add a sitemap, RSS, structured metadata, related posts, and stronger internal linking.

## Final Takeaway

The best Next.js MDX blog setup in 2026 is still the one that stays simple: MDX on disk, a small posts library, `next-mdx-remote/rsc`, real metadata, and a deployment setup that respects revalidation. That gives you a fast, flexible blog without dragging in a CMS you do not need yet.
