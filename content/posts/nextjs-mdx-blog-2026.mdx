## Why Build a Next.js MDX Blog Instead of Using WordPress?

If you are a developer, the biggest advantage is not speed. It is control.

Your content lives in the same Git workflow as the rest of your project. You can review post diffs, branch content changes, version metadata, and keep custom components without fighting a plugin ecosystem. That matters once the blog becomes part of your product, documentation, or growth engine.

The cost profile is also better if you are already comfortable with a VPS. A small self-hosted setup often costs less than a managed CMS stack with the same flexibility. If deployment is the part you are unsure about, pair this article with [How to self-host Next.js on a VPS](/blog/how-to-self-host-nextjs-on-vps) or the more concrete [Deploy Next.js on Hetzner with Coolify](/blog/deploy-nextjs-coolify-hetzner).

## next-mdx-remote vs @next/mdx

This is the first decision that confuses most tutorials.

Use `@next/mdx` when your `.mdx` files are basically routes. Use `next-mdx-remote/rsc` when your `.mdx` files are content data you want to load, parse, and render inside a custom blog system.

For a real blog with frontmatter, categories, related posts, featured images, and custom metadata, `next-mdx-remote/rsc` is the better fit. It lets you keep a proper posts library instead of turning every MDX file into a route definition.

That separation is what makes sitemap generation, RSS, related posts, and category pages much easier later.

## Recommended Project Structure

Keep the content and the rendering logic separate from day 1.

```text
app/
  blog/
    [slug]/
      page.tsx
  page.tsx
  sitemap.ts
components/
  blog/
    AuthorBio.tsx
    PostHeader.tsx
content/
  posts/
    first-post.mdx
lib/
  posts.ts
  site.ts
mdx-components.tsx
public/
  images/
    posts/
```

The important idea is that `content/posts/*.mdx` is just data. Your app code reads the files, parses frontmatter, computes reading time, and renders the result into the UI you want.

## Step 1 of 6 — Install the Core Packages

Start with a plain Next.js app, then add the MDX parsing and rendering packages you actually need.

```bash
npx create-next-app@latest my-blog --typescript --app
cd my-blog
npm install next-mdx-remote gray-matter reading-time
npm install remark-gfm rehype-slug rehype-autolink-headings
npm install rehype-pretty-code shiki
```

You do not need a complicated CMS SDK for the first version. `gray-matter` handles frontmatter, `reading-time` gives you a useful metadata field, and `next-mdx-remote/rsc` handles rendering cleanly in App Router.

## Step 2 of 6 — Build the Posts Library

Your posts library is the core of the whole system. It should read files, parse frontmatter, compute reading time, and expose helpers like `getAllPosts()` and `getPostBySlug()`.

```ts
import fs from "fs";
import path from "path";
import matter from "gray-matter";
import readingTime from "reading-time";
const POSTS_DIR = path.join(process.cwd(), "content/posts");
export function getAllPosts() {
  return fs
    .readdirSync(POSTS_DIR)
    .filter((file) => file.endsWith(".mdx"))
    .map((file) => {
      const slug = file.replace(/\.mdx$/, "");
      const raw = fs.readFileSync(path.join(POSTS_DIR, file), "utf8");
      const { data, content } = matter(raw);
      return {
        slug,
        ...data,
        readingTime: readingTime(content).text,
      };
    });
}
export function getPostBySlug(slug: string) {
  const raw = fs.readFileSync(
    path.join(POSTS_DIR, `${slug}.mdx`),
    "utf8"
  );
  const { data, content } = matter(raw);
  return {
    post: {
      slug,
      ...data,
      readingTime: readingTime(content).text,
    },
    content,
  };
}
```

This is the layer that later powers your homepage, blog index, related posts, sitemap, and RSS feed. Keep it boring and predictable.

## Step 3 of 6 — Render MDX in App Router

Once the posts library works, the blog route becomes straightforward.

```tsx
import { MDXRemote } from "next-mdx-remote/rsc";
import remarkGfm from "remark-gfm";
import rehypeSlug from "rehype-slug";
import rehypePrettyCode from "rehype-pretty-code";
import { getAllPosts, getPostBySlug } from "@/lib/posts";
import { notFound } from "next/navigation";
export async function generateStaticParams() {
  return getAllPosts().map((post) => ({ slug: post.slug }));
}
export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const result = getPostBySlug(slug);
  if (!result) notFound();
  return (
    <article>
      <MDXRemote
        source={result.content}
        options={{
          mdxOptions: {
            remarkPlugins: [remarkGfm],
            rehypePlugins: [
              rehypeSlug,
              [rehypePrettyCode, { theme: "github-dark" }],
            ],
          },
        }}
      />
    </article>
  );
}
```

The main reason this approach scales well is that metadata generation and content rendering stay separate. You can render the article while also using the same frontmatter for title tags, OG tags, breadcrumbs, and category pages.

## Step 5 of 6 — Make Code Blocks and Tables Worth Reading

Most MDX blog tutorials stop after "it renders." That is not enough for a developer publication.

Good technical content needs readable code blocks, tables that do not explode on mobile, and callouts for warnings and shortcuts. `rehype-pretty-code` is a strong default because it gives you server-rendered syntax highlighting without adding client-side weight.

Also make sure your prose styles handle:

- inline code
- ordered and unordered lists
- wide tables with a mobile scroll wrapper
- images and captions
- callouts for TL;DR and warnings

## Step 6 of 6 — Deploy It Without Breaking Revalidation

A Next.js MDX blog can run almost anywhere, but self-hosting introduces one common problem: proxy caching can fight ISR or revalidation behavior if your CDN is too aggressive with HTML.

That does not mean self-hosting is a bad idea. It means you need a clean deployment path and sane cache headers. My preferred path for this kind of blog is:

1. build on the server
2. run with PM2 or a platform like Coolify
3. proxy through Nginx
4. cache static assets heavily
5. avoid stale HTML policies that hide content updates

## Common Mistakes When Building a Next.js MDX Blog

### Mixing route-driven MDX with content-driven MDX

If you treat blog posts as route files too early, adding categories, related posts, or frontmatter-driven layouts becomes harder than it needs to be. Keep content as content.

### Letting metadata drift

If half your posts have featured images and half do not, your social sharing, card previews, and article pages start looking inconsistent fast. Standardize early.

### Ignoring mobile reading experience

Most readers will hit your posts on a phone first. If code blocks overflow badly or headings feel cramped, the whole blog feels lower quality even if the content is strong.

### Overengineering the CMS layer

For a solo or developer-led blog, MDX on disk is usually enough. You can add a CMS later if a non-technical editor actually needs it.

## FAQ

### Is Next.js MDX better than WordPress for a developer blog?

For developers, yes in many cases. A Next.js MDX blog gives you Git-based content, custom components, predictable deployment, and better control over performance. WordPress is easier for non-technical editors, but it is usually less pleasant if you want the blog to behave like part of your app.

### Should I use next-mdx-remote or @next/mdx?

Use `next-mdx-remote/rsc` when your MDX files are content stored on disk and you want to load them through a posts library. Use `@next/mdx` when your MDX files are route-level content. For a custom blog system, `next-mdx-remote/rsc` is usually the better fit.

### Can I self-host a Next.js MDX blog on a VPS?

Yes. A small VPS can handle a Next.js MDX blog well if you build it correctly, run it with PM2 or a platform layer, and put Nginx in front of it. That is one of the cheapest ways to keep full control over the stack.

### How do I add syntax highlighting to a Next.js MDX blog?

The cleanest approach is `rehype-pretty-code` with Shiki. It gives you server-rendered syntax highlighting, good themes, and no client-side highlighting overhead.

### Do I need a CMS for a Next.js blog in 2026?

Not always. If you are the main writer and you are comfortable with Git, MDX files on disk are enough for a lot of developer blogs. Add a CMS later only if your workflow genuinely needs non-technical editing or complex collaboration.

### What should I build after the blog is working?

After the core blog works, add a sitemap, RSS, structured metadata, related posts, and stronger internal linking. Those upgrades improve both discoverability and reader retention without forcing a full rebuild of the content model.
