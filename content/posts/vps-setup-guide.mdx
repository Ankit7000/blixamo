A fresh VPS is powerful for one reason: nothing gets in your way. That is also why people break them so quickly. The difference between a useful web-hosting server and a future disaster is usually the first hour: user setup, SSH rules, package updates, firewall, runtime install, and reverse proxy configuration.

This VPS setup guide is the practical baseline I would use before putting a real app online. Get the boring but high-leverage setup right first, then add app-specific deployment layers. ## What this VPS setup guide covers

This walkthrough assumes:

- Ubuntu 22.04 or 24.04
- SSH access to a fresh VPS
- a web app you eventually want to deploy
- a domain you can point later if needed

The goal is not to make the server perfect in one pass. The goal is to make it safe, stable, and ready for web hosting.

## Quick setup checklist

| Stage | Why it matters |
| --- | --- |
| update packages | avoids starting from outdated software |
| create non-root user | reduces avoidable risk |
| lock down SSH | removes the easiest attack paths |
| configure firewall | limits exposed services |
| install Node.js and PM2 | prepares app runtime |
| install Nginx | handles web traffic and reverse proxying |
| add SSL | makes the site production-safe |

## The first production-ready baseline I want

My baseline is not "the server exists." It is this:

- non-root deploy user works over SSH keys
- root login and password auth are off
- UFW exposes only SSH, HTTP, and HTTPS
- Node.js and PM2 are ready if the workload needs them
- Nginx is serving either the default page or the app cleanly
- the box is simple enough that I can explain it from memory

That last point matters. If the first day setup already feels confusing, the first incident will feel worse.

## Step 1: first login, updates, and a non-root user

SSH into the server as root the first time:

```bash
ssh root@YOUR_VPS_IP
```

Then update the system:

```bash
apt update
apt upgrade -y
```

<Callout type="tip" title="Expected result">
`apt update` should finish without repository errors, and `apt upgrade -y` should complete cleanly before you add more services.
</Callout>

Now create a deploy user and give it sudo access:

```bash
adduser deploy
usermod -aG sudo deploy
rsync --archive --chown=deploy:deploy ~/.ssh /home/deploy
```

The reason to do this immediately is simple: you do not want daily app work happening as root.

## Step 2: lock down SSH before you forget

Edit the SSH daemon config:

```bash
nano /etc/ssh/sshd_config
```

Set these values:

```text
PermitRootLogin no
PasswordAuthentication no
```

Then reload SSH:

```bash
systemctl restart ssh
```

Before you close the current root session, open a second terminal and verify that the new user can log in with SSH keys. Do not lock yourself out by testing this last.

## Step 3: add a basic firewall

Ubuntu usually ships with UFW available. Use it.

```bash
ufw allow OpenSSH
ufw allow 80
ufw allow 443
ufw enable
ufw status
```

<Callout type="tip" title="Expected result">
`ufw status` should show `Status: active` with rules for `OpenSSH`, `80`, and `443`.
</Callout>

At this stage you usually only need SSH, HTTP, and HTTPS exposed publicly. Do not open random app ports unless you know exactly why they need to be public.

If `ufw enable` warns about interrupting existing SSH connections, read the prompt carefully and make sure `OpenSSH` is already allowed before you confirm.

## Step 4: install Node.js if you are hosting Node apps

If your workload includes Next.js, Node APIs, or automation tools, install a current LTS version:

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt-get install -y nodejs
node --version
npm --version
```

<Callout type="tip" title="Expected result">
You want a current Node.js 20 LTS release and a working `npm --version` output before you move on.
</Callout>

If the server is not for Node workloads, you can skip this step. But for most Blixamo-style deployments, Node.js is part of the stack.

## Step 5: install PM2 for process management

PM2 is the simplest reliable way to keep Node apps running after crashes or reboots.

```bash
npm install -g pm2
pm2 startup
```

<Callout type="tip" title="Expected result">
PM2 should print a startup command for your init system. Save it and run it later after the app is live.
</Callout>

Follow the command PM2 prints, then later run `pm2 save` after your app is live.

If you know you are deploying a Next.js app next, [How to Self-Host Next.js on a VPS](/blog/how-to-self-host-nextjs-on-vps) covers the app-specific PM2 setup.

## Step 6: install and enable Nginx

Nginx is the default answer for basic web hosting on a VPS because it is fast, stable, and easy to reason about.

```bash
apt install nginx -y
systemctl enable nginx
systemctl start nginx
nginx -v
```

<Callout type="tip" title="Expected result">
`systemctl start nginx` should complete silently and `nginx -v` should print the installed version immediately.
</Callout>

Once Nginx is up, visiting the server IP in a browser should show the default welcome page. That tells you the web server layer is alive before you put an app behind it.

## Step 7: deploy an app and confirm localhost first

Clone or copy the app onto the server:

```bash
git clone https://github.com/you/your-app.git /var/www/myapp
cd /var/www/myapp
npm install
npm run build
```

Then start it with PM2:

```bash
pm2 start npm --name "myapp" -- start
pm2 save
```

Before touching Nginx config, verify the app directly:

```bash
curl -I http://localhost:3000
pm2 status
```

Expected checks:

```text
HTTP/1.1 200 OK
X-Powered-By: Next.js

┌────┬───────┬─────────┬────────┐
│ id │ name  │ mode    │ status │
├────┼───────┼─────────┼────────┤
│ 0  │ myapp │ fork    │ online │
└────┴───────┴─────────┴────────┘
```

That single check saves a lot of wasted debugging time.

<Callout type="info" title="Stop here and confirm this worked">
If `curl -I http://localhost:3000` does not return a real response, do not open Nginx files yet. Fix the app process first, then add the proxy.
</Callout>

## Step 8: connect Nginx to the app

Create an Nginx site config at `/etc/nginx/sites-available/myapp`:

```nginx
server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}
```

Enable it and validate the config:

```bash
ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/
nginx -t
systemctl reload nginx
```

<Callout type="tip" title="Expected result">
`nginx -t` should end with `syntax is ok` and `test is successful`. If it fails, fix Nginx before you request SSL.
</Callout>

If you need multiple apps on one box later, [Nginx Reverse Proxy Setup in 2026](/blog/nginx-reverse-proxy-guide-2026) is the guide to read next.

## Failure recovery notes I use on a fresh VPS

These are the first recovery moves I use when the setup looks wrong:

- SSH problem after config change: keep the original root session open until the deploy user logs in successfully from a second terminal
- UFW problem: confirm `OpenSSH` was allowed before enablement, then re-check from another terminal instead of guessing
- app not starting under PM2: run the start command directly once, then move back to PM2 after you know the app itself works
- Nginx problem: run `nginx -t` before every reload and never debug SSL until plain HTTP is already correct

Fresh-server incidents are usually sequencing errors, not mysterious infrastructure magic.

## Step 9: add SSL with Certbot

Install Certbot and let it update the Nginx config:

```bash
apt install certbot python3-certbot-nginx -y
certbot --nginx -d yourdomain.com -d www.yourdomain.com
```

Once it succeeds, test the final site over HTTPS. Do not treat the setup as finished until the certificate works and the redirect is clean.

## Final verification checklist

Before you call the server ready, I would run this exact checklist:

```bash
whoami
sudo ufw status
node --version
pm2 status
curl -I http://localhost:3000
nginx -t
systemctl status nginx --no-pager
curl -I https://yourdomain.com
```

What you want to see:

- `whoami` returns your non-root deploy user
- UFW is active with only the expected ports open
- Node.js is the version you planned for
- PM2 shows the app `online`
- `localhost:3000` responds before you blame Nginx
- Nginx config validates cleanly
- the Nginx service is active
- the final HTTPS domain returns a valid response

## Common mistakes during VPS setup

### Doing everything as root forever

That works until it does not. Create the non-root deploy user early and use it for day-to-day work.

### Pointing Nginx at an app that is not running

Always test `localhost:3000` first. If the app is down, Nginx will only give you a cleaner-looking error page.

### Skipping basic hardening

Package updates, SSH key auth, and firewall rules are not optional extras. They are the baseline.

### Installing too much too early

Do not turn a clean VPS into a cluttered experiment on day one. Get one app working first, then add monitoring, databases, and background tooling deliberately.

## Summary

A good VPS setup guide for web hosting should leave you with:

- a non-root deploy user
- SSH locked down
- firewall enabled
- Node.js and PM2 ready if needed
- Nginx installed and working
- SSL configured
- one clear path for future deploys

That is enough to host real projects without overengineering the first day.

## FAQ

### How much RAM do I need for basic web hosting on a VPS?

For a small site or one lightweight app, 1 GB can work. For more comfortable headroom, 2 GB is a better minimum. The right answer depends on your app, background jobs, and database strategy.

### Which VPS provider is best for this setup?

Hetzner, DigitalOcean, and Vultr are all fine starting points. The best choice depends on region, price, and whether you care more about cost efficiency or broader data center options.

### Should I install Docker during the first setup?

Only if you already know you need it. For many first deployments, Node.js + PM2 + Nginx is simpler and easier to debug.

### What should I do right after the server is live?

Set up backups, monitoring, log review, and security hardening next. Getting the app online is only the first half of responsible server ownership.

A VPS becomes valuable when it is boring in the right ways: predictable, secure enough, easy to redeploy, and easy to inspect when something breaks. That is what this setup gets you.
