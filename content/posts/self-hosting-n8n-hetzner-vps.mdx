## Step 2 of 6 — Create the n8n Data Directory and Environment File
#
n8n stores all workflow data, credentials, and execution history in a data directory. Set this up before running the container so data persists across restarts and updates.
#
```bash
mkdir -p /home/n8n-data
chown -R 1000:1000 /home/n8n-data
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
Create the environment file:
#
```bash
cat > /home/n8n.env << 'EOF'
N8N_HOST=n8n.yourdomain.com
N8N_PORT=5678
N8N_PROTOCOL=https
WEBHOOK_URL=https://n8n.yourdomain.com/
NODE_ENV=production
N8N_LOG_LEVEL=info
EXECUTIONS_DATA_PRUNE=true
EXECUTIONS_DATA_MAX_AGE=336
GENERIC_TIMEZONE=Asia/Kolkata
TZ=Asia/Kolkata
N8N_ENCRYPTION_KEY=your-random-32-char-string-here
EOF
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
<Callout type="warning" title="Three Settings That Matter">
Set `N8N_HOST` and `WEBHOOK_URL` to your actual subdomain — webhooks break without this. `EXECUTIONS_DATA_MAX_AGE=336` keeps execution history for 14 days then auto-prunes — without this the SQLite database grows indefinitely and eventually causes OOM. `N8N_ENCRYPTION_KEY` must be set before first run — you can't change it after credentials are stored.
</Callout>

## Step 3 of 6 — Run n8n With Docker
#
```bash
docker run -d \
  --name n8n \
  --restart unless-stopped \
  -p 127.0.0.1:5678:5678 \
  --env-file /home/n8n.env \
  -e EXECUTIONS_MODE=queue \
  -e QUEUE_BULL_REDIS_HOST=host.docker.internal \
  -e QUEUE_BULL_REDIS_PORT=6379 \
  --add-host=host.docker.internal:host-gateway \
  -v /home/n8n-data:/home/node/.n8n \
  n8nio/n8n
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
Key flags explained:
#
- `-p 127.0.0.1:5678:5678` — binds to localhost only; Nginx proxies to it, internet cannot reach it directly
- `--restart unless-stopped` — survives server reboots without manual intervention
- `EXECUTIONS_MODE=queue` — enables Redis queue mode for reliable, non-blocking workflow execution
- `--add-host=host.docker.internal:host-gateway` — lets the container reach the host's Redis instance
- `-v /home/n8n-data:/home/node/.n8n` — persists all data outside the container so updates don't wipe workflows
#
Check it started:
#
```bash
docker logs n8n --tail 20
# Should see: editor at http://localhost:5678
```
<Callout type="tip" title="Expected output">
`Editor is now accessible via: http://localhost:5678` confirms n8n started. Memory usage: `docker stats n8n --no-stream` should show ~80MB idle.
</Callout>
#
Memory usage: ~80MB idle, ~180MB with active workflows queued.

## Step 4 of 6 — Configure Nginx With SSL
#
Point your subdomain DNS A record to your VPS IP first, then:
#
```bash
cat > /etc/nginx/sites-available/n8n << 'EOF'
server {
    listen 80;
    server_name n8n.yourdomain.com;
    return 301 https://$host$request_uri;
}
#
server {
    listen 443 ssl;
    server_name n8n.yourdomain.com;
#
    ssl_certificate
      /etc/letsencrypt/live/n8n.yourdomain.com/fullchain.pem;
    ssl_certificate_key
      /etc/letsencrypt/live/n8n.yourdomain.com/privkey.pem;
#
    location / {
        proxy_pass http://127.0.0.1:5678;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For \
          $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 88400s;
        proxy_send_timeout 88400s;
        chunked_transfer_encoding on;
    }
}
EOF
#
ln -s /etc/nginx/sites-available/n8n \
  /etc/nginx/sites-enabled/
nginx -t
#
# Get SSL certificate
certbot --nginx -d n8n.yourdomain.com --non-interactive \
  --agree-tos -m your@email.com
#
systemctl reload nginx
```
<Callout type="tip" title="Expected output">
`nginx -t` output: `syntax is ok / test is successful`. If SSL fails, ensure DNS A record is pointing to your VPS IP and wait 2 minutes for propagation.
</Callout>
#
<Callout type="tip" title="Why proxy_read_timeout 88400s?">
n8n webhooks can have long-running connections. Without a high timeout, Nginx kills connections mid-workflow. 88400 seconds (24.5 hours) is effectively "never timeout" for any realistic workflow duration.
</Callout>

## Step 6 of 6 — Automatic Updates and Backups
#
Set up weekly auto-updates with a simple shell script:
#
```bash
cat > /usr/local/bin/update-n8n.sh << 'EOF'
#!/bin/bash
docker pull n8nio/n8n
docker stop n8n && docker rm n8n
docker run -d \
  --name n8n \
  --restart unless-stopped \
  -p 127.0.0.1:5678:5678 \
  --env-file /home/n8n.env \
  -e EXECUTIONS_MODE=queue \
  -e QUEUE_BULL_REDIS_HOST=host.docker.internal \
  -e QUEUE_BULL_REDIS_PORT=6379 \
  --add-host=host.docker.internal:host-gateway \
  -v /home/n8n-data:/home/node/.n8n \
  n8nio/n8n
echo "n8n updated at $(date)"
EOF
#
chmod +x /usr/local/bin/update-n8n.sh
#
# Run weekly on Sunday at 3am IST
(crontab -l 2>/dev/null; echo "0 3 * * 0 \
  /usr/local/bin/update-n8n.sh >> /var/log/n8n-update.log 2>&1") | crontab -
#
# Daily backup of n8n data directory
(crontab -l 2>/dev/null; echo "0 2 * * * tar -czf \
  /root/n8n-backup-\$(date +\%Y\%m\%d).tar.gz /home/n8n-data && find /root -name 'n8n-backup-*.tar.gz' -mtime +7 -delete") | crontab -
```

#
(command runs silently — no output means success)
#
</Callout>
7 days of n8n backups on a CPX22 typically use under 100MB. Also create a Hetzner snapshot monthly from the control panel — it's free for one snapshot per server.

## Troubleshooting Common Issues
#
**Workflows stop executing after a few hours**
#
Almost always an OOM kill. Verify:
#
```bash
dmesg | grep -i kill
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
If confirmed: add a 2GB swap file or upgrade to CPX31. Also ensure `EXECUTIONS_DATA_PRUNE=true` is set — uncontrolled execution history growth is the most common cause.
#
**Webhooks return 502 from external services**
#
The `WEBHOOK_URL` in your env file doesn't match your actual domain. Restart with `WEBHOOK_URL` set to exactly `https://n8n.yourdomain.com/` including the trailing slash.
#
**n8n login page shows but workflows don't save**
#
Permissions issue on the data directory:
#
```bash
chown -R 1000:1000 /home/n8n-data
docker restart n8n
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
**Queue mode not working — workflows run sequentially**
#
Redis not reachable from the Docker container. Test:
#
```bash
docker exec n8n ping host.docker.internal
```

#
(command runs silently — no output means success)
#
</Callout>
If it doesn't resolve, verify `--add-host=host.docker.internal:host-gateway` is in your docker run command.
#
**Certbot SSL renewal fails**

Check the auto-renewal timer:

systemctl status certbot.timer

### How much RAM does n8n use on a 4GB VPS?
#
At idle: ~80MB. Queue mode active: ~180MB. During complex workflow execution: peaks at 250–350MB. On a CPX22 running n8n alongside a Next.js app and FastAPI backend, total memory stays under 2.5GB with 1.5GB headroom. I've run this setup for over a year without OOM issues.
#
### Can I use n8n to replace WhatsApp automation on Make or Zapier?
#
Yes. I built a full WhatsApp AI assistant using n8n + WAHA + Claude API. The complete guide is at [WhatsApp AI assistant with n8n](/blog/whatsapp-ai-assistant-n8n-claude-api) — runs entirely on the same Hetzner VPS.
#
### How do I back up n8n workflows to git?
#
From n8n 0.214+, export all workflows via the CLI:
#
```bash
docker exec n8n n8n export:workflow --all \
  --output=/home/node/.n8n/workflows/
```
<Callout type="tip" title="Expected output">
#
(command runs silently — no output means success)
#
</Callout>
This creates individual JSON files you can commit to a git repo. Add this to your weekly update script.
#
### What happens to running workflows when I update n8n?
#
With queue mode, in-progress executions are queued in Redis. If you stop n8n mid-execution, queued items resume when n8n restarts. Actively running executions will fail. Best practice: update during low-traffic hours and check the execution queue is empty first.
#
### Can n8n connect to local services like Razorpay or MSG91?
#
Any service with a REST API works via n8n's HTTP Request node. Razorpay webhooks work as n8n Webhook triggers. MSG91, Cashfree, Shiprocket, and other regional payment/messaging APIs all integrate the same way.
#
### How do I migrate from Make or Zapier to n8n?
#
The migration isn't automatic — you rebuild workflows node by node. Start with your simplest 2-3 node workflows to learn n8n's interface. Run both old and new workflows in parallel for 3–5 days to validate output matches. Cancel your cloud subscription only after 2 weeks of stable n8n operation. The full cost math for why it's worth it is in the [n8n vs Make vs Zapier comparison](/blog/n8n-vs-make-vs-zapier-indie-dev).
