## Why Razorpay Over Stripe for Indian Indie Developers

Stripe works in India but with major friction: Indian businesses need to apply for Stripe India (separate from global Stripe), and as of March 2026, Stripe India approval takes 2–4 weeks. For a solo indie developer shipping fast, that delay kills momentum.

Razorpay is Indian-first. You sign up, complete KYC in 2–3 business days, and start accepting payments. UPI — the dominant Indian payment method at 12+ billion monthly transactions — is natively supported with zero MDR (merchant discount rate). Cards and net banking cost 2% per transaction.

The other option worth mentioning: PayU and CCAvenue exist but their developer experience is significantly worse than Razorpay. Their APIs feel like they were designed in 2012 and haven't changed since. Razorpay's API is clean, the docs are current, and their SDK is actively maintained.

For an Indian indie developer building a SaaS or digital product, Razorpay is the obvious first choice. I integrated it into a subscription billing system for a client in 2025 — 4 hours from npm install to first real transaction.

## Step 2 of 6 — Create a Payment Order (Server-Side)
#
Razorpay requires a two-step flow: first create an order on the server, then confirm payment on the client. This is what prevents payment manipulation — the order amount is set server-side and can't be changed by a clever user.
#
```bash
// app/api/payment/create-order/route.ts
// Creates a Razorpay order for the given amount
//
import Razorpay from 'razorpay'
import { NextResponse } from 'next/server'
//
const razorpay = new Razorpay({
  key_id: process.env.RAZORPAY_KEY_ID!,
  key_secret: process.env.RAZORPAY_KEY_SECRET!,
})
//
export async function POST(req: Request) {
  try {
    const { amount, currency = 'INR', receipt }
      = await req.json()
//
    // amount is in paise (1 INR = 100 paise)
    const order = await razorpay.orders.create({
      amount: amount * 100,
      currency,
      receipt: receipt || `receipt_${Date.now()}`,
    })
//
    return NextResponse.json({
      orderId: order.id,
      amount: order.amount,
      currency: order.currency,
    })
  } catch (error) {
    console.error('Razorpay order creation failed:', error)
    return NextResponse.json(
      { error: 'Failed to create order' },
      { status: 500 }
    )
  }
}
```
<Callout type="tip" title="Why this matters">
Razorpay amounts are in paise, not rupees. Rs 499 becomes `49900` in the API. Getting this wrong is the most common mistake — your checkout will show "Rs 49,900" instead of "Rs 499". Always multiply by 100 when sending to Razorpay, divide by 100 when displaying to users.
</Callout>

## Step 3 of 6 — Add the Checkout Button (Client-Side)
#
Load the Razorpay checkout script and trigger it when the user clicks pay:
#
```bash
// components/PaymentButton.tsx
'use client'
//
import { useState } from 'react'
//
interface Props {
  amount: number      // in rupees (e.g. 499)
  productName: string
  onSuccess: (paymentId: string) => void
}
//
export default function PaymentButton(
  { amount, productName, onSuccess }: Props
) {
  const [loading, setLoading] = useState(false)
//
  const handlePayment = async () => {
    setLoading(true)
//
    // Step 1: create order server-side
    const orderRes = 
      await fetch('/api/payment/create-order', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        amount,
        receipt: `order_${Date.now()}`
      }),
    })
    const { orderId, amount: orderAmount }
      = await orderRes.json()
//
    // Step 2: open Razorpay checkout
    const options = {
      key: process.env.NEXT_PUBLIC_RAZORPAY_KEY_ID,
      amount: orderAmount,
      currency: 'INR',
      name: 'Your App Name',
      description: productName,
      order_id: orderId,
      handler: async (response: {
        razorpay_payment_id: string
        razorpay_order_id: string
        razorpay_signature: string
      }) => {
        // Step 3: verify payment server-side
        const verifyRes = 
          await fetch('/api/payment/verify', {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify(response),
        })
        const { success } = await verifyRes.json()
        if (success) onSuccess(response.razorpay_payment_id)
      },
      prefill: { name: '', email: '', contact: '' },
      theme: { color: '#3B82F6' },
    }
//
    // Load Razorpay script dynamically
    const script = document.createElement('script')
    const rzpSrc =
      'https://checkout.razorpay.com/v1/checkout.js'
    script.src = rzpSrc
    script.onload = () => {
      const rzp = new (window as any).Razorpay(options)
      rzp.open()
      setLoading(false)
    }
    document.body.appendChild(script)
  }
//
  return (
    <button
      onClick={handlePayment}
      disabled={loading}
      style={{
        padding: '12px 24px',
        background: '#3B82F6',
        color: '#fff',
        border: 'none',
        borderRadius: '6px',
        cursor: 'pointer'
      }}
    >
      {loading ? 'Loading...' : `Pay Rs ${amount}`}
    </button>
  )
}
```

## Step 4 of 6 — Verify Payment Signature (Server-Side)
#
After the user completes payment, Razorpay returns a signature. Always verify this signature server-side before granting access or updating your database. Never trust client-side payment confirmation.
#
```bash
// app/api/payment/verify/route.ts
// Verifies Razorpay payment signature after checkout
//
import crypto from 'crypto'
import { NextResponse } from 'next/server'
//
export async function POST(req: Request) {
  const {
    razorpay_payment_id,
    razorpay_order_id,
    razorpay_signature,
  } = await req.json()
//
  // Verify signature using HMAC SHA256
  const body = `${razorpay_order_id}|${razorpay_payment_id}`
  const expectedSignature = crypto
    .createHmac('sha256', process.env.RAZORPAY_KEY_SECRET!)
    .update(body)
    .digest('hex')
//
  const isValid = expectedSignature === razorpay_signature
//
  if (!isValid) {
    return NextResponse.json(
      { success: false, error: 'Invalid signature' },
      { status: 400 }
    )
  }
//
  // Payment verified — update your database here
  // mark user as paid, create subscription record, etc.
//
  return NextResponse.json({
    success: true,
    paymentId: razorpay_payment_id
  })
}
```
<Callout type="danger" title="Signature verification is non-negotiable">
Skipping signature verification means any user can send a fake payment ID and get access for free. The 4 lines of crypto code above are the difference between a secure payment system and one that gets abused within days of launch. Never ship without this.
</Callout>

## Step 5 of 6 — Set Up Webhooks for Reliable Payment Confirmation
#
The checkout handler fires when the user's browser completes payment. But what if their browser closes mid-payment? Or they lose network? Webhooks catch these cases — Razorpay sends a POST to your server for every payment event regardless of browser state.
#
```bash
// app/api/payment/webhook/route.ts
// Handle Razorpay webhook events for payment tracking
//
import crypto from 'crypto'
import { NextResponse } from 'next/server'
//
export async function POST(req: Request) {
  const body = await req.text()
  const signature = 
    req.headers.get('x-razorpay-signature') || ''
//
  // Verify webhook signature
  const expectedSig = crypto
    .createHmac(
      'sha256',
      process.env.RAZORPAY_WEBHOOK_SECRET!
    )
    .update(body)
    .digest('hex')
//
  if (expectedSig !== signature) {
    return NextResponse.json(
      { error: 'Invalid signature' },
      { status: 400 }
    )
  }
//
  const event = JSON.parse(body)
//
  switch (event.event) {
    case 'payment.captured':
      const payment = event.payload.payment.entity
      // Payment successful — provision access
      const amt = payment.amount / 100
      console.log(
        `Payment: ${payment.id}, Rs ${amt}`
      )
      // await provisionUserAccess(payment.contact, payment.id)
      break
//
    case 'payment.failed':
      const failed = event.payload.payment.entity
      console.log(
        `Payment failed: ${failed.id},`
        + ` reason: ${failed.error_description}`
      )
      break
//
    case 'refund.created':
      console.log(
        `Refund: ${event.payload.refund.entity.id}`
      )
      break
  }
//
  return NextResponse.json({ received: true })
}
```
<Callout type="tip" title="Why this matters">
A user pays on mobile, their WiFi drops after payment clears but before the redirect. Without webhooks, your system never gets notified — the user paid but gets no access. With webhooks, Razorpay retries the POST to your server until it gets a 200 response. This is why webhooks are mandatory for production billing.
</Callout>
#
In Razorpay Dashboard → Settings → Webhooks → Add webhook:
- URL: `https://yourdomain.com/api/payment/webhook`
- Secret: generate a random string, add to `.env.local` as `RAZORPAY_WEBHOOK_SECRET`
- Events: `payment.captured`, `payment.failed`, `refund.created`
#
<Callout type="tip" title="Expected output">
When you trigger a test payment in Razorpay's dashboard, your webhook endpoint should receive the POST and log the event. Check your server logs: `pm2 logs yourapp --lines 20` to see the console.log output from the webhook handler.
</Callout>

## Troubleshooting Common Issues
#
**Payment popup opens but stays on loading screen**
#
Most common cause: wrong `order_id` passed to Razorpay options. The `order_id` must be the value from your server-side order creation response (`orderId` from the API route), not a custom value you generate. Log `orderId` after the fetch to verify it's a valid Razorpay order ID starting with `order_`.
#
**Webhook not receiving events**
#
Check two things: the webhook URL must be publicly accessible (not localhost — use ngrok for local testing), and the Content-Type for the webhook must be read as raw text before parsing. Using `req.json()` instead of `req.text()` will cause signature verification to fail because JSON parsing can alter whitespace.
#
**Payment captured but verification returns invalid signature**
#
The signature is computed from `orderId|paymentId` — note the pipe `|` separator. If you accidentally concatenate without the pipe or in the wrong order, signatures won't match. Log `body` in the verify route to confirm the format.
#
**KYC approval delayed beyond 3 business days**
#
Razorpay KYC typically approves in 1–2 business days for individual developers with a valid PAN and bank account. Common delays: bank account name doesn't exactly match PAN card name, or cancelled cheque image is unclear. Upload a clear photo of a cancelled cheque (not a screenshot of net banking).

## FAQ
#
### Can I accept payments without KYC in test mode?
#
Yes. Razorpay test mode works without KYC — you can build and test the full payment flow immediately after signup. KYC is required only to accept real money in live mode.
#
### What Indian payment methods does Razorpay support?
#
Razorpay supports UPI (including Google Pay, PhonePe, Paytm), all major Indian debit and credit cards (Visa, Mastercard, RuPay), net banking for 100+ Indian banks, EMI, and wallets. This covers approximately 95% of Indian payment preferences in a single integration.
#
### Is Razorpay better than Stripe for an Indian indie developer?
#
For accepting INR payments from Indian users: Razorpay. The approval process is faster (2–3 days vs 2–4 weeks for Stripe India), UPI support is native, and the developer experience is well-suited to indie projects. For accepting international payments (USD, EUR) from global users: Stripe or Paddle. Many Indian indie developers use both: Razorpay for Indian users, Stripe for international.
#
### How do I handle subscriptions with Razorpay?
#
Razorpay has a Subscriptions API that handles recurring billing with automatic UPI mandate creation. The setup is more complex than one-time payments — create a Plan, create a Subscription linked to the plan, and Razorpay handles the recurring charge. The webhook events `subscription.activated`, `subscription.charged`, and `subscription.cancelled` cover the full lifecycle.
#
### Can I use Razorpay with Supabase or Firebase for the user database?
#
Yes. After verifying payment in the webhook handler, call your database to update the user's subscription status. For Supabase: use the Supabase server client in the webhook route. For Firebase: use the Firebase Admin SDK. The webhook handler is just a server-side API route — any Node.js database client works.
#
### What happens to payments if my server goes down during checkout?
#
The user's checkout session is managed by Razorpay's servers, not yours. If your server goes down after the user pays but before the checkout handler fires, the webhook will retry the event to your server every 15 minutes for 24 hours. This is why the webhook handler is more reliable than the client-side callback for provisioning access.
