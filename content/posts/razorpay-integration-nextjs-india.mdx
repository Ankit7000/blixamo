I added Razorpay to a Next.js product because waiting for the perfect billing setup would have delayed the launch. The practical job was simple: accept INR payments, keep the checkout familiar for Indian users, and make sure payment confirmation could not be faked from the browser.

That means 4 things matter more than the rest of the integration:

1. Create every order on the server.
2. Open Checkout on the client with the `order_id` you created.
3. Verify the signature on the server before you mark anything paid.
4. Use webhooks so a missed callback does not create a false failure.

<Callout type="info" title="TL;DR">
- The safe Razorpay flow in Next.js is `create order -> open checkout -> verify signature -> confirm with webhook`.
- Amounts are sent in the smallest currency unit, so `499` rupees becomes `49900`.
- Do not trust the browser callback alone for fulfillment. Use server-side verification and webhooks.
</Callout>

<Callout type="info" title="My setup">
- Next.js App Router
- Node.js 18+
- Razorpay Standard Checkout
- Test mode first, live keys only after webhook verification works
</Callout>

## Who this is for

- India-first SaaS or digital product founders using Next.js.
- Developers who need UPI, cards, net banking, and wallets behind one checkout.
- Teams that want a straightforward one-time payment flow before building subscriptions or invoicing layers.

## Skip this if

- Your product is not India-focused and Stripe already fits your business entity and payout setup.
- You want to trust client-side "payment success" events without server verification.
- You have not planned what happens when checkout succeeds but your app misses the callback.

## Fastest path

Use Razorpay Standard Checkout first. It covers the common India payment methods, fits a lean launch, and gives you a clear integration shape:

1. Server route creates the order.
2. Client opens Checkout with that order.
3. Server route verifies the returned signature.
4. Webhook confirms the final payment state and drives fulfillment.

That is enough for most early-stage digital products. Do not start by abstracting payment providers or designing a big billing engine.

## Step 1 - Install the SDK and add environment variables

Install the Node SDK:

```bash
npm install razorpay
```

Create the environment variables:

```bash
# .env.local
RAZORPAY_KEY_ID=rzp_test_xxxxx
RAZORPAY_KEY_SECRET=your_test_secret
RAZORPAY_WEBHOOK_SECRET=your_webhook_secret
NEXT_PUBLIC_RAZORPAY_KEY_ID=rzp_test_xxxxx
```

`NEXT_PUBLIC_RAZORPAY_KEY_ID` is safe in the browser. `RAZORPAY_KEY_SECRET` and `RAZORPAY_WEBHOOK_SECRET` are server-only values and should never be exposed to client code.

## Step 2 - Create the order on the server

Razorpay expects you to create an order server-side and pass the returned `order_id` to Checkout. Keep the amount authoritative on the server, not in the browser.

```ts
// app/api/razorpay/create-order/route.ts
import { NextResponse } from "next/server";
import Razorpay from "razorpay";

const razorpay = new Razorpay({
  key_id: process.env.RAZORPAY_KEY_ID!,
  key_secret: process.env.RAZORPAY_KEY_SECRET!,
});

export async function POST(request: Request) {
  const { amountInRupees, receipt } = await request.json();

  const order = await razorpay.orders.create({
    amount: amountInRupees * 100,
    currency: "INR",
    receipt,
  });

  return NextResponse.json({
    orderId: order.id,
    amount: order.amount,
    currency: order.currency,
  });
}
```

### Why this matters

- Razorpay amounts are in the smallest currency unit.
- The server decides the amount and receipt.
- The browser gets only the values it needs to open Checkout.

## Step 3 - Open Razorpay Checkout in the client

Use the returned `orderId` to launch Checkout from a client component:

```tsx
"use client";

type PaymentButtonProps = {
  amountInRupees: number;
  receipt: string;
};

export function PaymentButton({
  amountInRupees,
  receipt,
}: PaymentButtonProps) {
  async function handleClick() {
    const orderRes = await fetch("/api/razorpay/create-order", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ amountInRupees, receipt }),
    });

    const { orderId, amount, currency } = await orderRes.json();

    const options = {
      key: process.env.NEXT_PUBLIC_RAZORPAY_KEY_ID,
      order_id: orderId,
      amount,
      currency,
      name: "Your Product Name",
      description: "Order payment",
      handler: async (response: {
        razorpay_order_id: string;
        razorpay_payment_id: string;
        razorpay_signature: string;
      }) => {
        await fetch("/api/razorpay/verify", {
          method: "POST",
          headers: { "Content-Type": "application/json" },
          body: JSON.stringify(response),
        });
      },
      theme: { color: "#111827" },
    };

    const RazorpayCheckout = (window as any).Razorpay;
    const checkout = new RazorpayCheckout(options);
    checkout.open();
  }

  return <button onClick={handleClick}>Pay now</button>;
}
```

Load the Razorpay script once in your layout or payment page:

```tsx
<script src="https://checkout.razorpay.com/v1/checkout.js" async />
```

Do not mark the purchase complete in the browser. The handler should only pass the returned values back to your server.

## Step 4 - Verify the payment signature on the server

Razorpay returns `razorpay_payment_id`, `razorpay_order_id`, and `razorpay_signature` after a successful checkout. Verify them on the server before you grant access, create a license, or mark an order paid.

```ts
// app/api/razorpay/verify/route.ts
import crypto from "crypto";
import { NextResponse } from "next/server";

export async function POST(request: Request) {
  const {
    razorpay_order_id,
    razorpay_payment_id,
    razorpay_signature,
  } = await request.json();

  const expectedSignature = crypto
    .createHmac("sha256", process.env.RAZORPAY_KEY_SECRET!)
    .update(`${razorpay_order_id}|${razorpay_payment_id}`)
    .digest("hex");

  const isValid = expectedSignature === razorpay_signature;

  if (!isValid) {
    return NextResponse.json({ ok: false }, { status: 400 });
  }

  // Persist the payment attempt here.
  // Do not fulfill the order yet if your webhook is the source of truth.
  return NextResponse.json({ ok: true });
}
```

### Common mistake

Do not use the browser as your source of truth. The browser tells you what the user saw. Your server should decide whether the payment is authentic.

## Step 5 - Add a webhook for final payment state

Webhooks protect you from missed callbacks, retry edges, and late authorization cases. Use them for order fulfillment or final status transitions.

```ts
// app/api/razorpay/webhook/route.ts
import crypto from "crypto";
import { NextResponse } from "next/server";

export async function POST(request: Request) {
  const rawBody = await request.text();
  const signature = request.headers.get("x-razorpay-signature");

  const expectedSignature = crypto
    .createHmac("sha256", process.env.RAZORPAY_WEBHOOK_SECRET!)
    .update(rawBody)
    .digest("hex");

  if (!signature || signature !== expectedSignature) {
    return NextResponse.json({ ok: false }, { status: 400 });
  }

  const event = JSON.parse(rawBody);

  if (event.event === "payment.captured" || event.event === "order.paid") {
    // Mark the order fulfilled here.
    // Make this idempotent so retries do not create duplicate fulfillment.
  }

  return NextResponse.json({ ok: true });
}
```

### What success looks like

- Checkout opens with the right INR amount.
- Your verify route returns success after a test payment.
- Your webhook route receives and validates the event.
- Fulfillment is idempotent, so the same event does not create duplicate access.

## Common failure points

### Amount mismatch

If the amount looks wrong in Checkout, you probably forgot that Razorpay uses the smallest currency unit. `499` should become `49900`, not `499`.

### Exposed secret

If `RAZORPAY_KEY_SECRET` or `RAZORPAY_WEBHOOK_SECRET` appears in client code, fix that first. Those values belong on the server only.

### Missing idempotency

Webhooks can retry. If your fulfillment logic creates licenses, credits, or subscriptions, make that operation idempotent.

### Trusting the callback too early

The Checkout handler is useful, but it is not enough by itself for production fulfillment. Verify the signature on the server and use webhooks for the final state.

## Verification checklist

Before you switch to live keys:

1. Make a test payment in Razorpay test mode.
2. Confirm the server created the order and stored the receipt you expect.
3. Confirm your verify route accepts the returned signature.
4. Confirm the webhook route validates the raw body with the webhook secret.
5. Confirm a duplicate webhook does not create duplicate fulfillment.

## Rollback / restore check

If payments are going live for the first time, keep a manual fallback for a day or two. That can be as simple as a "contact us to complete payment" backup path or a manual invoice flow while you confirm webhook delivery and fulfillment logs on real traffic.
