### CPU: 3 vCPUs vs 1 burstable vCPU

AWS Lightsail uses "burstable" T-series instances (T3 or T4g). This means you get full CPU for short bursts but are throttled when you exceed your CPU credit balance. ## When AWS Lightsail Wins

Hetzner is not the right answer for everyone. Lightsail makes sense when:

**You're deeply in the AWS ecosystem.** If your app already uses S3, RDS, CloudFront, SES, and SNS from the same AWS account, adding a Lightsail instance gives you VPC peering to all of those services. Hetzner requires external services or self-hosting equivalents.

**You need compliance certifications.** AWS Lightsail inherits AWS's SOC 2, ISO 27001, and HIPAA BAA support. Hetzner has some certifications but not the full AWS compliance portfolio.

**Your team already knows AWS.** Operational familiarity matters in production. If your entire team knows CloudWatch and AWS Console, switching to Hetzner + Coolify has a learning curve cost.

**You're in a region Hetzner doesn't cover.** Hetzner currently has datacenters in Germany, Finland, and the USA. If you need Singapore, Sydney, or SÃ£o Paulo, Hetzner isn't an option.

## Troubleshooting Migration Issues
**Problem: App works on Lightsail but crashes on Hetzner**
Most likely cause: missing environment variables. Lightsail stores env vars in its dashboard. They don't transfer automatically.
```bash
# On Lightsail, export all env vars
printenv | grep -v "^_" > lightsail_env.txt
# Copy to Hetzner manually, don't SCP (contains secrets)
```
**Problem: Database connection refused after migration**
Check that your app's `DATABASE_URL` points to the new Hetzner database, not still pointing to the Lightsail managed DB.
**Problem: SSL certificate error after DNS switch**
Let's Encrypt needs to verify domain ownership via HTTP. If Cloudflare proxy is on, the ACME challenge fails. Set DNS to DNS-only during cert provisioning.
