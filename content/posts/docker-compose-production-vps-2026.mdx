# Docker Compose on VPS in 2026: Production Setup Guide

Docker Compose is still a good production choice on a VPS when your stack is small enough to stay understandable. The setup I trust is simple: one app directory, named volumes for state, health checks on the services that matter, a reverse proxy in front, and a backup routine you have actually tested. If you skip any of those pieces, the stack works right up until the first bad reboot, bad deploy, or full disk.

I would use Compose on a VPS for 2 to 5 services on one machine. I would not use it if the real problem is multi-node orchestration or team-level platform abstractions. For the small-VPS case, though, Compose stays one of the cleanest ways to ship a stack you can still debug at 2 AM.

<Callout type="info" title="TL;DR">
Use Docker Compose on a VPS when you want simple multi-service deployment without committing to a heavier platform. The production basics are non-negotiable: named volumes, health checks, restart policies, reverse proxying, and a real restore test. If one of those is missing, fix that before you add more services.
</Callout>

## Quick production checklist

| Requirement | Why it matters |
|---|---|
| Named volumes | Keeps state safe across container recreation |
| Health checks | Stops app containers from racing the database |
| Restart policies | Brings services back after host reboot |
| Reverse proxy | Handles public traffic, TLS, and sane ports |
| Backup plus restore test | Proves the stack is recoverable, not just running |

## When Docker Compose is the right choice

Compose is the right answer when:

- one VPS can handle the workload
- you want to understand every moving part
- the stack is a few services, not a platform
- you care about cheap, boring operations more than clever orchestration

It is a bad answer when the team really needs horizontal scaling, service discovery across many nodes, or policy-heavy platform automation. That is not a Compose problem. That is just a different job.

## Step 1: Prepare the VPS before you write any Compose file

Do the boring host work first. That includes Docker, Compose, swap, and a place on disk where the stack will live.

```bash
# install Docker and Compose plugin
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# add 2 GB swap on a small VPS
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
```

<Callout type="tip" title="Verify it">
Run `docker --version`, `docker compose version`, and `swapon --show`. If any of those checks fail, stop here. Compose problems get much harder to debug when the host itself is half-configured.
</Callout>

On a small VPS, swap is not optional if you pull images, build containers, or run bursts of work. It is not there to make bad sizing okay forever. It is there to stop a short memory spike from killing the stack.

## Step 2: Use a Compose file that is boring on purpose

This is the part most tutorials overcomplicate. The goal is not a huge Compose file. The goal is a readable one that another developer can inspect quickly.

```yaml
services:
  app:
    image: ghcr.io/example/app:latest
    restart: unless-stopped
    env_file: .env
    depends_on:
      db:
        condition: service_healthy
  db:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB: app
      POSTGRES_USER: app
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app"]
      interval: 10s
      timeout: 5s
      retries: 5
volumes:
  postgres_data:
```

This file does 4 important things:

- keeps secrets out of the YAML by using `.env`
- stores database state in a named volume
- waits for Postgres to become healthy before the app depends on it
- restarts services automatically after a host reboot

What I would not do:

- put secrets directly in `compose.yaml`
- use bind mounts for database state unless I had a specific reason
- skip health checks and hope the startup order behaves
- expose every service directly to the public internet

## Step 3: Validate before you start containers

The fastest way to avoid a dumb outage is to validate the file before `up -d`.

```bash
mkdir -p /opt/my-stack
cd /opt/my-stack
printf "POSTGRES_PASSWORD=replace-me\n" > .env
docker compose config -q
```

<Callout type="tip" title="Expected result">
`docker compose config -q` should exit cleanly with no output. If it prints interpolation errors, missing env values, or invalid keys, fix those first. Do not debug them after the containers are already running.
</Callout>

That one check catches more production mistakes than people think: missing environment variables, broken YAML, invalid health-check syntax, and typo-level config drift.

## Step 4: Start the stack and verify the right things

Bring the stack up, then verify service health, not just container existence.

```bash
docker compose up -d
docker compose ps
docker compose logs --tail=50
```

What you want to see:

- the app container is `Up`
- the database is `healthy`
- logs show the app using the expected database host
- there is no restart loop

If the app depends on a database or queue, an `Up` status alone is not enough. A container can be alive and still be failing its real job.

## Step 5: Put a reverse proxy in front of the stack

Do not publish your app container directly if the service is meant for the public web. Put Nginx or another reverse proxy in front so TLS, host routing, and public ports stay predictable.

```nginx
server {
    server_name app.example.com;
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

After that:

```bash
sudo nginx -t
sudo systemctl reload nginx
```

If you expose the container directly on a public port, you lose a clean place to manage TLS and traffic rules. That is fine for a private service. It is lazy for a public one.

For the full host-level routing layer, [Nginx Reverse Proxy Setup in 2026](/blog/nginx-reverse-proxy-guide-2026) is the better companion article.

## Step 6: Backups and restore tests are part of the setup

Most Compose guides stop after `up -d`. That is too early. A production setup is not complete until you know how to recover it.

For a database-backed stack, I want 2 things:

- a scheduled dump or backup routine
- a restore test to a temporary target

```bash
# example Postgres backup
docker compose exec -T db \
  pg_dump -U app app | gzip \
  > /var/backups/app-$(date +%F).sql.gz
```

Then prove you can restore it. A backup you have never restored is only a theory.

## Troubleshooting that actually matters

### The app starts before the database is ready

That usually means the dependency exists but health checks do not. Add a real health check to the database and wait on `service_healthy`.

### The stack comes back empty after recreation

That is usually a volume mistake. Check whether the service used a named volume or a bind mount pointed at the wrong place.

### A deploy "works" but the app is broken

That is a verification problem, not a Compose problem. Check logs, health checks, and one real request path after every deploy.

### The VPS runs out of disk over time

That is often unused images, logs, or stale backup files. Watch disk growth before it turns into a surprise outage.

### The stack is too fragile for one VPS

That is the point where you decide whether to simplify the services, split the workloads, or move to a different platform. Do not keep patching a topology that no longer fits the machine.

## Which should you choose: raw Compose or a platform?

Use raw Compose if you want:

- the smallest moving surface
- readable deployment state
- full control over file layout and updates
- cheap, predictable infrastructure

Use a platform like Coolify if you want:

- easier app onboarding
- less manual reverse-proxy work
- a friendlier surface for repeated deploys

I still like raw Compose when the stack is small and the operator is comfortable. It is easier to trust because there is less magic between the config and the running services.

The honest exception is a team or founder who wants faster repeatability with less command-line glue. That is where a platform starts earning its overhead.

## FAQ

### Is Docker Compose okay for production on a VPS in 2026?

Yes. It is still a good production choice for small multi-service stacks on one VPS, as long as you use named volumes, health checks, restart policies, a reverse proxy, and tested backups.

### How much RAM do I need for Docker Compose on a VPS?

That depends on the services, but 2 to 4 GB is a realistic starting point for a small app plus a database. Add swap as a safety buffer on small machines.

### Should I use bind mounts or named volumes?

Use named volumes for stateful services unless you have a specific reason not to. They are safer for ordinary production use and easier to reason about during container recreation.

### Do I need Nginx in front of Docker Compose?

For public web traffic, usually yes. A reverse proxy gives you a cleaner place to handle TLS, host routing, and public entry points.

### What is the biggest production mistake with Docker Compose?

Treating `docker compose up -d` as the finish line. The real production work is validation, health checks, backups, restore tests, and host-level traffic handling.

### Should I use Docker Compose or Kubernetes for a side project?

Use Docker Compose unless you genuinely need multi-node orchestration. Kubernetes adds operational overhead that most side projects do not need.
