A decent VPS can host a production Next.js app for a fraction of the monthly cost if you are comfortable with Node.js, PM2, Nginx, and basic Linux administration.

## When self-hosting Next.js on a VPS makes sense

Self-hosting is usually the right move when one or more of these are true:

- you want predictable monthly cost
- you need root access or custom Nginx behavior
- you plan to run multiple apps on one machine
- you want tighter control over logs, caching, and background jobs
- you already know enough Linux to maintain the server safely

If you just need a demo online tonight, Vercel is simpler. If you want long-term control over cost and infrastructure, a VPS wins quickly.

## What you need before you start

Prepare these first:

- an Ubuntu 22.04 or 24.04 VPS
- a domain pointed at the server IP
- SSH access
- a working Next.js app that already builds locally

Before touching the server, run a local production build:

```bash
npm run build
```

If the app fails locally, fixing it on the VPS will be slower and more frustrating.

If your blog uses local MDX files, also verify those routes are [not inheriting Edge runtime in production](/blog/nextjs-mdx-remote-rsc-edge-runtime-fix) before you assume the VPS setup is the problem.

## Step 1 of 8: point your domain to the VPS

In your DNS panel, add A records for the root domain and `www`:

```text
A   yourdomain.com      -> YOUR_VPS_IP
A   www.yourdomain.com  -> YOUR_VPS_IP
```

Wait for propagation, then verify:

```bash
nslookup yourdomain.com
```

If `nslookup` does not return your server IP yet, do not move on to SSL. Nginx and Certbot become harder to troubleshoot when DNS is still stale.

## Step 2 of 8: SSH in and install Node.js

Connect to the box:

```bash
ssh root@YOUR_VPS_IP
```

Install Node.js 20 LTS and confirm the version:

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt-get install -y nodejs
node --version
npm --version
```

At this point, you want a stable LTS runtime, not a random distro package that lags behind for months.

## Step 3 of 8: install PM2 and Nginx

PM2 keeps the app alive. Nginx handles the public HTTP traffic, compression, and reverse proxying.

```bash
npm install -g pm2
apt-get update
apt-get install -y nginx
pm2 --version
nginx -v
```

If you plan to host several apps later, this combination scales surprisingly well. For a deeper reverse proxy setup, [Nginx Reverse Proxy Setup in 2026](/blog/nginx-reverse-proxy-guide-2026) goes further.

## Step 4 of 8: upload or clone the Next.js app

You can copy the project from your machine:

```bash
scp -r "C:\path\to\your-app" root@YOUR_VPS_IP:/var/www/myapp
```

Or clone from GitHub directly on the server:

```bash
cd /var/www
git clone https://github.com/youruser/your-nextjs-app.git myapp
```

The GitHub route is usually cleaner because future deploys are simpler.

## Step 5 of 8: install dependencies and build for production

Move into the project directory and build it:

```bash
cd /var/www/myapp
npm install
npm run build
```

You want the build to finish with zero errors before you introduce PM2 or Nginx into the debugging chain.

If the build fails, the usual reasons are:

- environment variables missing on the server
- case-sensitive file paths that worked on Windows
- TypeScript errors ignored during local development
- native dependency mismatch between local machine and server

## Step 6 of 8: run the app with PM2

Create an `ecosystem.config.js` file in the project root:

```js
module.exports = {
  apps: [
    {
      name: 'myapp',
      script: 'node_modules/.bin/next',
      args: 'start',
      instances: 'max',
      exec_mode: 'cluster',
      env: {
        NODE_ENV: 'production',
        PORT: 3000,
      },
    },
  ],
}
```

Start the app and persist it:

```bash
pm2 start ecosystem.config.js
pm2 save
pm2 startup
```

Then confirm the process is online:

```bash
pm2 status
curl -I http://localhost:3000
```

Do not continue to Nginx until `localhost:3000` responds correctly. Otherwise you will end up debugging the proxy while the app itself is still broken.

## Step 7 of 8: configure Nginx as the reverse proxy

Create the site config:

```bash
nano /etc/nginx/sites-available/myapp
```

Paste this basic configuration:

```nginx
server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;

    gzip on;
    gzip_types text/plain text/css application/json application/javascript;

    location /_next/static/ {
        proxy_pass http://localhost:3000;
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_cache_bypass $http_upgrade;
    }
}
```

Enable it and validate the config:

```bash
ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/
nginx -t
systemctl reload nginx
```

Visit `http://yourdomain.com`. If the app loads over plain HTTP, the core setup is working.

## Step 8 of 8: add SSL with Certbot

Install Certbot and request the certificate:

```bash
apt-get install -y certbot python3-certbot-nginx
certbot --nginx -d yourdomain.com -d www.yourdomain.com
```

Certbot will update the Nginx config and set up renewal automatically. Once that finishes, test the final site on HTTPS.

## How to deploy updates without downtime

Once the site is live, the update flow is straightforward:

```bash
cd /var/www/myapp
git pull
npm install
npm run build
pm2 reload myapp --update-env
```

Use `reload`, not `restart`, if you want cleaner deploys with less visible interruption.

If you want to keep the app lean after launch, [Next.js Performance Optimization in 2026](/blog/nextjs-performance-optimization-2026) is a good follow-up once the deployment is stable.

## Common problems and how to fix them

### Build works locally but fails on the VPS

This usually means the server environment is not equivalent to local development. Check Node.js version first, then environment variables, then file path casing.

### Nginx shows 502 Bad Gateway

That almost always means PM2 is not actually serving the app on port 3000. Check:

```bash
pm2 status
pm2 logs myapp
curl -I http://localhost:3000
```

If `localhost:3000` fails, Nginx is only showing you the symptom.

### SSL request fails

Do not troubleshoot Certbot until DNS is confirmed. Most SSL setup failures happen because the domain is not really pointing at the box yet.

## Quick summary

To self-host Next.js on a VPS safely:

- confirm the app builds locally first
- install Node.js, PM2, and Nginx
- get the app responding on `localhost:3000`
- put Nginx in front of it
- add SSL only after HTTP works
- use `pm2 reload` for future deploys

That order keeps the setup understandable and easier to debug.

## FAQ

### Is self-hosting Next.js on a VPS cheaper than Vercel?

Usually yes, especially once you host more than one app or you want predictable cost. The tradeoff is that you are now responsible for updates, monitoring, backups, and security.

### Can I self-host Next.js without Docker?

Yes. PM2 + Nginx is still a solid way to run a Next.js app on a VPS. Docker can be helpful, but it is not required for a reliable deployment.

### Which VPS provider is best for this setup?

Any provider with stable Ubuntu images works. Hetzner is popular because of cost efficiency, but DigitalOcean and Vultr are also fine if they fit your region or support needs.

### Should I use PM2 cluster mode for every app?

Not automatically. It can help on multi-core servers, but start simple and only keep cluster mode if your app and runtime behavior actually benefit from it.

## Conclusion

If you want lower monthly cost and more control, self-hosting Next.js on a VPS is absolutely worth learning. I would choose it for projects where logs, reverse proxy behavior, background jobs, or cost predictability matter more than one-click convenience.

Use Vercel when speed matters most. Use a VPS when control and economics matter more. For many developer-run projects, that tradeoff becomes obvious after the first real hosting bill.
