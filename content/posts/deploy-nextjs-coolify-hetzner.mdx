## Step 1 — Create Your Hetzner VPS

Log into [Hetzner Cloud Console](https://console.hetzner.cloud) and create a new project.

**Recommended server for Next.js:**

| Server | vCPU | RAM | Storage | Price |
|--------|------|-----|---------|-------|
| CPX21 | 3 | 4GB | 80GB | €5.19/month |
| CPX31 | 4 | 8GB | 160GB | €10.52/month |

CPX21 is the sweet spot for 1-3 Next.js apps. If you're running 5+ projects or have a high-traffic app, go CPX31.

**Setup steps:**

1. Click **Create Server**
2. Location: **Nuremberg** or **Helsinki** (lowest latency for India via Cloudflare)
3. Image: **Ubuntu 24.04**
4. Type: **Shared vCPU → CPX21**
5. SSH Key: Add your public key (highly recommended over password auth)
6. Click **Create & Buy Now**

Your server will be ready in about 30 seconds. Note the IP address — you'll need it.

### Add Swap Space First

This is the step most tutorials skip and you'll regret it. Next.js builds are memory-hungry. On a 4GB server, a build can spike to 3.5GB RAM. Without swap, the process gets OOM-killed mid-build.

SSH into your server and run:

```bash
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

Verify it worked:

```bash
free -h
# Swap: 2.0Gi should now show
```

I learned this the hard way — 13 PM2 restarts in a week, all OOM kills during builds. 2GB swap costs nothing and saves hours of debugging.

## Step 2 — Install Coolify

Coolify installs via a single curl command. SSH into your server:

```bash
ssh root@YOUR_VPS_IP
```

Then run the official installer:

```bash
curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash
```

This installs Docker, Docker Compose, and the Coolify application itself. Takes about 3-5 minutes depending on server speed.

When it finishes, you'll see something like:

```
Coolify is installed!
Access it at: http://YOUR_VPS_IP:8000
```

Open `http://YOUR_VPS_IP:8000` in your browser. You'll see the Coolify setup wizard.

### First-Time Coolify Setup

1. **Create admin account** — set your email and password
2. **Instance settings** — set your instance domain if you have one (e.g. `coolify.yourdomain.com`), otherwise skip
3. **Server** — Coolify auto-detects your localhost. Click **Validate Server** — it should pass all checks

You're now inside the Coolify dashboard.

## Step 3 — Connect GitHub

In the Coolify sidebar, go to **Sources** → **Add** → **GitHub App**.

Follow the GitHub OAuth flow:
1. Name your GitHub App (e.g. "Coolify - My Server")
2. Authorize the repositories you want to deploy
3. You'll be redirected back to Coolify with the connection confirmed

You should now see your GitHub organization/account listed under Sources.

> **Private repos work fine.** Coolify uses a GitHub App (not a personal token), so it has proper scoped permissions.

---

## Step 4 — Create a New Project and Application

In Coolify:

1. Click **Projects** → **New Project** → name it (e.g. "My Next.js App")
2. Inside the project, click **New Resource** → **Application**
3. Select **GitHub** as your source
4. Choose your repository and branch (usually `main`)
5. Build pack: **Nixpacks** (Coolify auto-detects Next.js and configures the build correctly)

### Configure the Application

Coolify will show you a settings page for your app. Key fields:

**General tab:**
- **Name**: Your app name
- **Domains**: Enter your domain here (e.g. `myapp.com` or `app.myapp.com`). Coolify will auto-provision SSL via Let's Encrypt.
- **Port**: `3000` (Next.js default)

**Build tab:**
- **Build Command**: `npm run build` (auto-detected, usually correct)
- **Start Command**: `npm run start` (auto-detected)
- **Base Directory**: `/` unless your Next.js app is in a subdirectory

**Health Check tab:**
- Path: `/`
- Interval: `30` seconds
- Leave other defaults

## Step 5 — Add Environment Variables

Click the **Environment Variables** tab. Add your `.env.local` values here.

```
DATABASE_URL=your_database_url
NEXT_PUBLIC_API_URL=https://api.yourdomain.com
NEXTAUTH_SECRET=your_secret_here
NEXTAUTH_URL=https://yourdomain.com
```

> **Important:** Variables prefixed with `NEXT_PUBLIC_` are embedded at build time, not runtime. Make sure these are set correctly before your first deploy — you can't inject them after the build.

For Supabase projects specifically:

```
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your_anon_key
SUPABASE_SERVICE_ROLE_KEY=your_service_role_key
```

## Step 6 — Deploy

Click the **Deploy** button. Coolify will:

1. Pull your code from GitHub
2. Run `npm install`
3. Run `npm run build`
4. Start the Next.js server
5. Route traffic through its built-in proxy (Traefik)
6. Provision SSL certificate via Let's Encrypt (if you added a domain)

Watch the build logs in real time — Coolify shows the full output so you can catch errors immediately.

A typical Next.js build on CPX21 takes **2-4 minutes**.

When it finishes, you'll see **Running** status with a green dot. Your app is live.

## Step 7 — Configure Your Domain DNS

If you added a custom domain, point your DNS to your Hetzner server IP:

```
Type: A
Name: @ (or subdomain like "app")
Value: YOUR_VPS_IP
TTL: 300
```

For Cloudflare users — set the proxy to **DNS only (grey cloud)** initially until SSL is provisioned, then you can enable the orange cloud.

SSL certificate provisioning takes 1-5 minutes after DNS propagates. Coolify handles the Let's Encrypt challenge automatically.

## Step 8 — Enable Auto-Deploy on Git Push

In your Coolify application settings, go to the **Webhooks** tab. You'll see a webhook URL like:

```
https://coolify.yourdomain.com/webhooks/source/github/events/manual
```

In your GitHub repository:
1. Go to **Settings** → **Webhooks** → **Add webhook**
2. Paste the Coolify webhook URL
3. Content type: `application/json`
4. Events: **Just the push event**
5. Save

Now every `git push` to your main branch triggers an automatic deploy. Zero-downtime — Coolify builds the new version, waits for it to be healthy, then switches traffic.

## Step 9 — Add Persistent Storage (If Needed)

If your app writes files to disk (uploads, SQLite, etc.), you need persistent storage. Container filesystems are ephemeral — they reset on each deploy.

In Coolify, go to your application → **Storages** tab → **Add Storage**:

- **Name**: `uploads` (or whatever makes sense)
- **Source path on host**: `/data/your-app/uploads`
- **Destination path in container**: `/app/public/uploads`

This mounts a host directory into your container so data persists across deploys.

For databases, don't use the container filesystem at all — use a managed service (Supabase, PlanetScale, Neon) or create a separate Coolify database resource.

## Step 10 — Set Up Multiple Apps on One Server

One Hetzner CPX21 can comfortably run 3-5 Next.js apps simultaneously. Here's how I structure my setup:

```
coolify.blixamo.com     → Coolify dashboard (port 8000, internal)
blixamo.com             → This blog (Next.js)
aitools-for-devs.com    → AI tools directory (Next.js)
api.myapp.com           → FastAPI backend (Python)
n8n.blixamo.com         → n8n automation (Docker)
```

Coolify's built-in Traefik proxy handles routing between all of these based on domain names. No manual nginx configuration needed.

For each new app:
1. Create a new resource in the same project (or a new project)
2. Set its domain
3. Deploy

Traefik picks it up automatically. No port conflicts, no nginx files to edit.

> If you want more control over your nginx configuration (like I do for this blog), you can also run your own nginx alongside Coolify. Just use ports 8001+ for Coolify and keep nginx on 80/443. See my [guide on running multiple projects on one VPS](/blog/multiple-projects-single-vps) for the full setup.

## Troubleshooting Common Errors

### Build fails with "JavaScript heap out of memory"

Your server ran out of RAM during the build. The swap fix from Step 1 handles this. If you already have swap and still see this:

```bash
# Increase Node.js memory limit
NODE_OPTIONS=--max-old-space-size=3072 npm run build
```

Add `NODE_OPTIONS=--max-old-space-size=3072` as an environment variable in Coolify.

### "Port 3000 already in use"

Another process is using port 3000. Find and kill it:

```bash
lsof -i :3000
kill -9 PID_FROM_ABOVE
```

Or just set Coolify to use a different port (3001, 3002, etc.) in the application settings.

### SSL certificate not provisioning

Most common cause: DNS hasn't propagated yet, or Cloudflare orange cloud is blocking the Let's Encrypt HTTP challenge.

```bash
# Check if DNS resolves correctly from VPS
dig yourdomain.com +short
# Should return your Hetzner IP
```

Set Cloudflare to grey cloud (DNS only), wait 5 minutes, trigger a redeploy in Coolify.

### App crashes after deploy with "Module not found"

Usually means a package is in `devDependencies` that should be in `dependencies`. Next.js production builds need everything in `dependencies`.

```bash
# Check your package.json
# Move the missing package from devDependencies to dependencies
npm install your-package --save
```

### Coolify dashboard shows "Unhealthy"

The health check is failing. Check:
1. Your app is actually starting (check the deploy logs)
2. The health check path (`/`) returns 200
3. Port is set correctly (default: 3000)

You can temporarily disable health checks in the app settings to confirm the app is running.

### "NEXT_PUBLIC_ variables showing as undefined"

These are baked in at build time. If you added them after the first build, trigger a fresh deploy — Coolify → your app → **Deploy** button.

## Coolify vs Doing It Manually (PM2 + nginx)

I run both setups — Coolify for some apps, manual PM2+nginx for this blog. Here's when each makes sense:

| | Coolify | Manual PM2+nginx |
|---|---|---|
| Setup time | 30 mins | 2-3 hours |
| Auto-deploy | ✅ Built-in | ❌ Manual or scripted |
| SSL | ✅ Automatic | ✅ Certbot |
| Custom nginx config | ❌ Limited | ✅ Full control |
| Resource overhead | ~300MB RAM for Coolify itself | ~0 overhead |
| Multi-app routing | ✅ Automatic (Traefik) | ✅ Manual nginx blocks |
| Rollback | ✅ One click | ❌ Manual git reset + rebuild |

For most Next.js apps, Coolify wins on convenience. For this blog, I use manual PM2+nginx because I need precise Cache-Control header control for ISR + Cloudflare to work correctly — something Traefik's defaults don't handle well out of the box.

## Frequently Asked Questions

**Does Coolify support Next.js 15 App Router?**

Yes. Coolify uses Nixpacks for builds which supports Next.js 15 including App Router, Server Components, and Server Actions. No special configuration needed — it detects the Next.js version and builds correctly.

**How much RAM does Coolify itself use?**

Coolify + its supporting services (Traefik, PostgreSQL, Redis) use around 300-400MB RAM at idle. On a 4GB CPX21, that leaves 3.5GB+ for your apps.

**Can I use Coolify with a private GitHub repository?**

Yes. The GitHub App integration supports private repos. You grant Coolify access to specific repositories during setup, so it only sees what you authorize.

**Is Coolify really free?**

The self-hosted version is completely free and open source (Apache 2.0 license). There's a Coolify Cloud version if you don't want to manage the server yourself, but for self-hosting on Hetzner it costs nothing.

**How do I rollback a broken deploy?**

In Coolify, go to your application → **Deployments** tab. Every deploy is listed with a timestamp. Click any previous deployment → **Rollback**. It redeploys that exact version instantly.

**Can I deploy a Next.js app with a custom server (Express)?**

Yes. Set your **Start Command** to `node server.js` (or whatever your entry point is) instead of `npm run start`. Make sure your custom server listens on the port Coolify expects (default 3000).

**What happens to my app during a deploy?**

Coolify builds the new version in a separate container, runs health checks on it, and only then switches traffic. Your old version stays live until the new one is healthy. Zero downtime.

**Does Hetzner block any ports needed by Coolify?**

No. Hetzner doesn't block ports by default. Coolify needs ports 80, 443, and 8000 open. You can verify with:

```bash
ufw status
# If ufw is active, allow ports:
ufw allow 80 && ufw allow 443 && ufw allow 8000
```

**Can I use Coolify with a non-GitHub repo (GitLab, Bitbucket)?**

Yes. Coolify supports GitLab, Bitbucket, and Gitea in addition to GitHub. The setup flow is similar — go to Sources and add your provider.

**How do I check my app's logs in Coolify?**

Go to your application in the Coolify dashboard → **Logs** tab. You'll see live stdout/stderr output. For historical logs, check the **Deployments** tab — each deploy has its own build log.
