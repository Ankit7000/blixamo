If you want to self-host a Next.js app on Hetzner in 2026, the simplest reliable path is: create a CPX server, add swap, install Coolify, connect GitHub, set your env vars, point DNS, and let Coolify handle the deploy and SSL.

I have deployed a lot of Next.js projects on Hetzner, including this site. Coolify is the fastest way I know to get from a blank VPS to a live, auto-deploying Next.js app with SSL and a custom domain without hand-rolling the whole workflow yourself.

My default verdict is simple: choose this stack when you want VPS-level cost control and enough deployment convenience to avoid babysitting every release. Skip it if you want the absolute easiest hosted path or if you need very custom server behavior from day one.

## Step 1: Create the Hetzner VPS

For most small Next.js projects, start with a CPX-class server that gives you enough RAM headroom for builds and a few running services.

My default rule is simple: if you plan to run several projects on the same server, size for the build spikes, not just the idle process list.

![Hetzner server created and ready for the first Coolify install](/images/posts/deploy-nextjs-coolify-hetzner/01-hetzner-server-created.png)
*Provisioning the box is the easy part. The quality of the first deploy depends more on the steps after this screen.*

## Step 2: Add Swap First

This is the step too many tutorials skip. Next.js builds can spike memory hard on smaller machines.

```bash
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

Verify it:

```bash
free -h
```

<Callout type="tip" title="Expected result">
`free -h` should show about `2.0Gi` of swap. If swap still shows `0B`, fix that now before the first build.
</Callout>

## Step 3: Install Coolify

```bash
curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash
```

After install, open the dashboard on the provided port and finish the initial setup.

## Step 4: Connect GitHub

Use the GitHub App integration so Coolify can access the repo you actually want to deploy. This is one of the biggest reasons the workflow feels clean once it is set up.

## Step 6: Add Environment Variables

Set the env vars before the first real deploy, especially anything under `NEXT_PUBLIC_`, because those values are baked in at build time.

## Step 7: Deploy

Click deploy and watch the logs. The first deploy is where you catch missing packages, incorrect build commands, and bad environment values.

In the verified run below, Coolify finished the Nixpacks build and rolling update successfully. That proves the deploy completed inside Coolify, not that the temporary public domain or final DNS path was already working.

## Step 8: Point DNS and Wait for SSL

Point your A record at the Hetzner server IP. If you use Cloudflare, keep it on DNS-only until SSL is issued cleanly.

## What Broke on My First Deploy

The first deploy on this stack usually breaks for practical reasons, not mysterious platform reasons.

### Swap was missing

On a small VPS, the build looked fine until memory spiked. Then the process died. Adding a `2G` swap file before the first build fixed that immediately.

### `NEXT_PUBLIC_*` values were wrong at build time

Coolify will happily start a deploy with empty or outdated public env vars. The app can build, come online, and still look broken because those values were baked in incorrectly.

### SSL looked stuck because Cloudflare was proxied too early

If Cloudflare is orange-clouded before the certificate is issued cleanly, you waste time debugging the wrong layer. Keep it on DNS-only until the certificate is live.

### The deploy was marked healthy before I checked the app myself

The first checks happen inside Coolify, not in DNS. I want the deployment log to finish cleanly and the runtime log to show `next start` listening on port `3000` before I blame domains, SSL, or Cloudflare.

![Coolify runtime logs showing Next.js app started successfully on port 3000](/images/posts/deploy-nextjs-coolify-hetzner/deploy-nextjs-coolify-hetzner-04-runtime-ready.webp)
*Runtime logs confirm the Next.js app is serving correctly*

After that, I still check the service directly:

```bash
curl -I http://localhost:3000
curl -I https://yourdomain.com
```

<Callout type="tip" title="Expected result">
`localhost:3000` should return `HTTP/1.1 200 OK` or a valid redirect first. If the temporary domain or your final domain still fails after that, debug routing, reverse proxy, DNS, or SSL next instead of redeploying the app.
</Callout>

## Common Problems

### Build fails with memory errors

Add swap first. If that is still not enough, raise the Node memory limit for the build.

### SSL does not provision

DNS is usually the problem. Double-check the record and keep Cloudflare unproxied until the certificate is issued.

### The app starts but looks broken

That usually means environment variables are missing or the build-time values were wrong.

### A content route fails only after deploy

If the app uses local MDX files, check whether a page is inheriting Edge runtime while the content path still depends on Node.js APIs. The quickest fix is usually in [Fix next-mdx-remote/rsc Edge Error in Next.js 15](/blog/nextjs-mdx-remote-rsc-edge-runtime-fix).

### Port confusion

Be explicit about the app port and what the health check should hit.
